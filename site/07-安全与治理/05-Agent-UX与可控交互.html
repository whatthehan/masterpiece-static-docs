#05 · Agent UX 与可控交互

小林为订单 order_123 申请退款 100 元。系统发现两份政策版本冲突，于是先停在“需要补充信息”；确认有效政策后，它生成精确退款预览并等待审批。小林批准，界面进入“正在提交”。几秒后响应中断，她随即点击 Cancel。

此刻最危险的 UX 不是按钮不好看，而是界面抢先显示“已取消”。真实 Runtime 只知道停止请求已经收到，尚不知道退款是否已提交。本章只解决这一件事：把持久状态如实投影成用户看得懂、能采取行动的界面。 下一部分再实现这些状态背后的失败收敛协议。

#学习目标

  • 为每个用户可见状态找到唯一的 Runtime 状态或状态集合。
  • 让按钮提交类型化事件，而不是在前端伪造成功状态。
  • 展示足够的目标、证据、提案和结果，不暴露原始思维链（Chain-of-Thought）。

#1. 同一条任务怎样出现在界面上

事件时刻Runtime 状态用户看到什么当前允许的操作
两份政策冲突WAITING_INPUT“政策版本冲突，需要确认适用版本”及两份来源补充信息、选择可信来源、停止
退款提案已冻结WAITING_APPROVAL订单、金额、依据、对外动作、可逆性与过期时间批准、拒绝、返回修改
已发出提交命令EXECUTING_TOOL“正在向支付系统提交退款”请求停止后续工作；不承诺撤销提交
ACK 丢失后点击 CancelCANCEL_REQUESTED / IN_DOUBT“已收到停止请求，正在核对退款是否已发生”查看详情、等待、联系人工；不显示普通 Retry
查到支付回执COMPLETED_WITH_EFFECT_AFTER_CANCEL“退款已完成；你的停止请求发生在提交之后”及回执查看证据、反馈问题
到期仍无法确认MANUAL_INTERVENTION“结果仍未确认，已交由人工处理”及责任方/预计响应时间补充材料、联系人工

这一张表是 UI 契约。产品文案可以变化，状态含义不能变化；前端断线或页面关闭也不能改写 Runtime 事实。

#2. 用户看到的是状态投影，不是内部枚举

直接把 IN_DOUBT 显示给用户没有帮助，但把它翻译成“提交结果尚未确认”就有了明确含义。一个稳定投影至少包含：

goal          当前要完成什么
status        已确认的任务状态
current       正在进行的可理解步骤
evidence      来源、提案、回执及仍未知的部分
waiting_for   需要谁提供什么
next          系统接下来会做什么
controls      当前状态真正允许的操作
budget/time   明显的时间、费用或处理期限

模型生成的自然语言可以解释这些字段，但不能决定 status、伪造回执或开放一个非法按钮。状态和控制项由 Runtime 的类型化事件与允许转移产生。

#3. Preview 与 Approval 必须投影同一份提案

小林审批前看到的不是“允许 Agent 使用退款工具”，而是一份不可变提案：

actor: 小林
order_id: order_123
amount: CNY 100.00
policy_version: refund-policy-2026-07
proposal_hash: sha256:...
resource_version: 42
expires_at: 2026-07-12T14:30:00+08:00

界面必须展示对象、精确参数、依据、对外披露、费用、可逆性、资源版本和有效期。审批后任一关键字段变化，Runtime 都应回到重新预览或重新审批；前端不能用旧弹窗替新提案背书。

#4. 按钮是协议事件

控制发出的事件不承诺什么
Stop / Cancelcancel_requested，停止创建新工作并向在途步骤传播不承诺已提交效果被回滚
Steer为尚未执行的目标或约束创建版本化修改不改写已冻结、已审批的提案
Edit & Retry保留旧 attempt，使用新输入创建新 attempt不覆盖旧失败证据
Resume从持久 snapshot 请求恢复不绕过已过期的权限、政策、审批或 deadline
Retry failed step请求 Runtime 检查是否存在安全重试路径不允许前端直接重复 command

事件要记录 actor、目标版本、时间与相关 run_id/attempt_id。UI 只有在 Runtime 接受事件并返回新状态后才能更新；乐观动画不能伪造业务终态。

#5. 解释结果，不展示原始思维链

用户需要理解系统做了什么，不需要读取模型的隐藏推理。对于 order_123，界面可以展示：

  • 当前目标和适用政策版本。
  • 政策冲突怎样被解决，哪些来源被排除。
  • 冻结提案、审批人、审批范围与有效期。
  • 工具动作、状态变化、支付回执和真实 Outcome。
  • 仍无法确认的部分、下一责任方和人工升级入口。

引用、权威事实、模型推断和未知状态应使用不同视觉语义。不要用一个伪精确“信心分”替代来源、冲突和可验证结果。

#纸面微实验（45 分钟）

为 order_123 画六个界面：政策冲突、等待审批、提交中、停止请求已收到、核对到退款已发生、转人工。每个界面标注：

  1. 唯一对应的 Runtime 状态。
  2. 用户能看到的事实与仍未知内容。
  3. 按钮提交的事件及不保证事项。
  4. 刷新页面后从哪里恢复相同视图。

任一界面无法映射到持久状态，或在 ACK 丢失时直接显示“退款已取消”，即不通过。

#常见误区

  • 流式文字一直输出就等于进度可见。
  • 点击 Cancel 后立即把 UI 写成 CANCELLED。
  • 用户批准目标等于批准后续任意参数。
  • 前端 Retry 可以直接重复调用写工具。
  • 展示原始 Chain-of-Thought 才算可解释。

#章末检查

  1. 为什么 Cancel requested 不能立即显示为 Cancelled？
  2. Preview 和 Approval 应绑定哪些真实执行字段？
  3. 页面刷新后，哪些状态必须从服务端 snapshot 恢复？
  4. 不展示原始思维链时，哪些证据足以让用户理解结果？

#本章小结

Agent UX 是 Runtime 的诚实投影：界面展示已经确认的事实、仍未知的效果和当前合法控制，不自行发明终态。小林点击 Cancel 后，页面只能说“正在核对退款是否已发生”。下一章将沿这条未完成的事件进入失败、超时、重试与取消，给出 IN_DOUBT → RECONCILING 的权威收敛协议。
