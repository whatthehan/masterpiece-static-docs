# 04 · 纵深防御与人类控制

既然自然语言攻击无法被可靠地一次性清除，安全设计就不能押注某个分类器、护栏（Guardrail）或确认弹窗。纵深防御（Defense in Depth）要求模型、策略与环境承担不同责任；人类在环（Human-in-the-Loop，HITL）也必须对应澄清、审批、复核或升级等真实 Runtime 状态。

本章把前面识别出的边界组织成可运行的防御和响应机制。核心问题不再是“有没有安全功能”，而是危险提案在哪些执行点（enforcement point）被阻断、控制失效后的影响范围（blast radius）多大，以及系统如何停止新动作并核对已经发生的效果。

## 学习目标

- 把防御分为模型、策略和环境三层。
- 正确使用 clarification、approval、review 和 escalation。
- 为检测、响应、回滚和撤销设计运行机制。

## 1. 三层防御

```text
模型层：清晰指令、结构化输出、分类器、模型安全训练
策略层：Schema/语义校验、授权、审批、预算、审计
环境层：沙箱、凭证隔离、网络/文件/进程/资源限制
```

模型层降低错误概率；策略层强制业务不变量；环境层限制突破后的影响。关键控制至少应有两层独立保障。

## 2. 四种 Human-in-the-Loop

- Clarification：目标或参数不足。
- Approval：对具体动作表示同意。
- Review：检查产物质量或证据。
- Escalation：Agent 无法安全继续，转交责任。

把所有场景都做成“是否允许”弹窗会造成审批疲劳和责任混乱。

## 3. 有效审批界面

展示：actor、目标、精确参数、diff、外发数据、不可逆效果、费用、有效期和撤销/替代方案。用户拒绝后 Agent 可以提出低风险替代，但不能绕过同一策略反复换工具。

## 4. 检测与响应

- Trace grader 检查异常工具序列和数据流。
- 预算与 rate limit 阻止循环/成本攻击。
- 行为基线与告警发现工具调用、目标或权限漂移。
- Kill switch 停止新动作并撤销短期凭证。
- Incident record 保留模型、Prompt、工具、策略、来源和受影响资源版本。
- 修复后把攻击路径加入 regression suite。

取消不会回滚已完成动作；需要查询最终状态、去重、补偿或人工处置。

## 5. 风险递增门禁

| 能力阶段        | 最低安全证据                      |
| ----------- | --------------------------- |
| 无工具         | 敏感数据处理、输出评测、版本记录            |
| 只读工具        | ACL、来源标签、注入测试、Trace、timeout |
| 写工具         | 服务端授权、具体审批、幂等、审计、补偿         |
| 代码/浏览器      | 强沙箱、egress、秘密隔离、资源限制        |
| 长任务/多 Agent | 持久策略、委派链、恢复、撤销、级联故障测试       |

## 纸面微实验（45 分钟）

设计一个使模型生成危险 tool call 的攻击，用表格推演模型、策略、环境各层正常或失效时的真实 outcome。产出至少两个独立 enforcement point、监控信号和事故响应步骤。

## 常见误区

- 每次询问用户就是最安全方案。
- Guardrail 可以替代授权。
- 沙箱后不需要策略。
- Kill switch 会撤销已发生效果。
- 安全测试通过后无需生产监控。

## 章末检查

1. 模型、策略、环境三层分别承担什么责任？
2. Clarification 与 Approval 为什么不能共用同一语义？
3. Incident 响应为什么必须同时撤销凭证并检查最终外部状态？

## 本章小结

纵深防御的关键不是层数，而是模型降低误判概率、策略强制业务不变量、环境限制实际能力，且高影响控制至少有独立兜底。人类控制只有映射到明确状态与提案时才有效；下一章将这些状态变成用户能够理解和操作的[Agent UX 与可控交互](/masterpiece-static-docs/07-安全与治理/05-Agent-UX与可控交互.md)。
