#02 · 向量、嵌入（Embedding）与相似度

当贯穿案例开始从内部文档和公开资料中寻找证据时，模型不能靠“把所有内容都读一遍”解决问题。嵌入可以把语义相关的内容变成检索候选，但候选是否真实、最新、可引用且允许当前用户读取，仍是应用层的责任。

因此这一章不把向量当作抽象数学对象，而是沿着一条真实检索管线理解它：表示怎样产生候选、指标怎样暴露漏召回，以及权限为什么必须在内容进入 Context 之前生效。

#学习目标

  • 理解嵌入是任务相关的表示，不是事实数据库。
  • 能解释余弦相似度（cosine similarity）、top-k、召回率和重排。
  • 知道语义相近、事实正确和有权限读取是三件事。

#1. 向量表示

必须区分三类表示：

表示输入/输出用途
词元嵌入（Token embedding）Token ID → 初始向量Transformer 的输入表示
上下文化隐藏状态（contextual hidden state）某层、某位置 → 随上下文变化的向量模型内部计算
文本/文档嵌入（text/document embedding）一段文本 → 固定长度向量检索、聚类、分类

Embedding 模型通常指第三类。训练目标使某种意义上的相关文本靠近，但“相关”的定义由训练数据和目标决定；它与生成模型内部的 Token embedding 不能混用。

余弦相似度（cosine similarity）比较向量方向：

cos(a,b) = (a · b) / (||a|| ||b||)

它衡量表示空间中的接近程度，不提供逻辑蕴含、时间有效性、来源质量或访问许可。

#2. 检索是候选生成

典型检索管线：

actor/tenant/purpose → 选择授权分区或下推 ACL predicate
query → 规范化/改写 → 在授权范围内 sparse + dense retrieval
      → 防御性 ACL/metadata 复验 → fusion → rerank → context selection
  • 稀疏检索（sparse retrieval，如 BM25）依赖词项匹配，专有名词和精确字符串通常较强。
  • 稠密检索（dense retrieval）使用嵌入，能召回不同措辞的语义相关内容。
  • 混合检索（hybrid retrieval）组合两者，仍必须通过任务集验证。
  • 重排器（reranker）对较小候选集做更昂贵的相关性判断。

#3. 关键指标

  • 精确率（Precision@k）：返回的 k 个结果中有多少相关。
  • 召回率（Recall@k）：所有相关结果中有多少进入 top-k。
  • 平均倒数排名（MRR）：第一个正确结果出现得多靠前。
  • 归一化折损累计增益（nDCG）：同时考虑分级相关性与排序位置。

生成答案质量差时，先区分 retrieval failure 与 generation failure。模型无法引用从未被召回的证据。

#4. 权限必须早于上下文暴露

ACL 不能只在全局 top-k 之后做 post-filter：无权候选可能已被读取/计分、形成侧信道，还会把合法候选挤出 top-k（recall starvation）。候选生成应在租户分区或可下推 ACL predicate 内完成，进入 Context 前再防御性复验。过滤通常要绑定：

tenant + actor + resource + purpose + time + policy_version

#微实验

准备 20 个短文档，其中加入：同义表达、同名不同实体、过期版本、无权版本和恶意指令。比较关键词检索与 embedding top-5：

  • 哪些相关文档漏召回？
  • 哪些只是语义相近但不能回答问题？
  • ACL 在哪个阶段生效？
  • 恶意文本是否被标记为不可信数据？

#常见误区

  • 向量库就是长期记忆。
  • cosine 分数可以跨模型、跨索引直接比较。
  • top-1 相似文档就是正确来源。
  • 文档切得越小召回越好；过小会丢失语境，过大又稀释信号。

#章末检查

  1. 为什么 embedding 相似不能证明一个结论？
  2. Recall@k 高但最终答案差，应继续检查哪些层？
  3. 为什么先检索再做 ACL 过滤可能已经造成泄漏？

#本章小结

嵌入检索负责在授权范围内生成相关候选，相关性本身不提供事实、来源或权限保证。下一章进一步解释为什么即使离线评测表现良好，模型和整个 Agent 仍可能在真实分布、交互环境或规则变化后失效。
