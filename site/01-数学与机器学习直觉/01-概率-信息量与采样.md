# 01 · 概率、信息量与采样

研究—判断—行动 Agent 面临的第一个矛盾是：用户希望系统稳定完成任务，模型却每一步都在一个概率分布中选择输出。一次成功的演示因此不能证明系统可靠，一次流畅的回答也不能证明下一次工具参数仍然相同。

这一章的数学只服务于工程判断。你需要据此决定何时运行多次试次、怎样解释长轨迹的误差累积，以及哪些高风险结果必须交给确定性校验，而不是推导模型训练算法。

## 学习目标

- 理解模型输出是分布，不是确定答案。
- 能解释对数概率（log probability）、熵（entropy）、归一化指数函数（softmax）和温度（temperature）的工程含义。
- 知道多次试次（trial）为什么比一次演示可靠。

## 1. 从分数到概率

模型为候选 Token 产生未归一化分数（logits）。归一化指数函数（softmax）把这些分数转为总和为 1 的概率分布：

```text
P_i = exp(z_i / T) / Σ exp(z_j / T)
```

`T` 是温度（temperature）。降低它通常让分布更集中，提高它通常让更多低概率候选被采样。它改变的是选择分布，不会把错误知识变正确，也不保证完全确定。

公式只在 `T > 0` 时定义；数学上的 greedy 极限是 `T → 0⁺`。API 接受 `temperature=0` 时通常表示实现约定的 greedy/近 greedy 行为，并列最大 logit 与服务端数值细节仍由实现决定。

## 2. 序列概率与误差累积

一个回答由条件概率连乘形成。工程上使用对数概率（log probability），把乘法变成加法并避免数值下溢：

```text
log P(y|x) = Σ log P(y_t | x, y_<t)
```

较早的选择会改变后续条件分布。这就是 Agent 长轨迹中小错误可能级联的一个原因：错误工具或错误参数会改变环境，新的观察又成为下一步上下文。

## 3. 熵（Entropy）与不确定性

熵描述分布的分散程度。高熵意味着候选更分散，但低熵不代表事实一定正确：模型可能非常自信地复现错误模式。

需要区分：

- 偶然不确定性（aleatoric uncertainty）：任务或生成过程本身有多种合理答案。
- 认知不确定性（epistemic uncertainty）：模型缺知识、上下文或能力。
- 系统不确定性（system uncertainty）：工具、网络、权限和数据状态不确定。

应用不应只让模型用一句“置信度 0.9”混合三者。

## 4. 试次（Trial）与事件概率

若单次任务成功概率是 `p`，连续运行的结果会波动。一次成功不能估计稳定性。对关键行为至少记录：

- 每个任务的多次 trial。
- pass rate 与失败类型。
- pass\@k：k 次中至少一次成功，适合允许多次探索的场景。
- pass^k：k 次全部成功，适合强调一致性的场景。

不要用 pass\@k 掩盖生产系统只能执行一次的事实。

## 微实验

选一个存在多种合理措辞的分类任务和一个只有唯一合法动作的任务，各运行 10 次：

1. 比较输出文本变化。
2. 比较结构化标签或工具参数变化。
3. 记录“文本不同但语义相同”和“语义真的不同”。
4. 解释为什么唯一动作任务需要更强的程序约束。

## 常见误区

- `temperature=0` 等于形式化确定性：错误。服务实现、模型版本和并行计算仍可能造成差异。
- 概率最高等于事实正确：错误。它只表示模型分布中的相对偏好。
- 多数投票总能提高正确率：只有错误近似独立且单次正确率足够时才成立。
- 模型自报置信度可以直接做高风险门禁：未经校准不能。

## 本章小结

模型输出是分布，长轨迹还会让早期小错误改变后续环境。因此 Agent 工程必须用多次试次测量波动，并由程序约束不可逆动作。下一章把同样的边界意识带到向量表示：相似度可以帮助生成候选，却不能证明事实、权限或时效。
