# 04 · 上下文窗口、采样与能力边界

当贯穿案例积累了更多资料、工具结果和历史状态，最直觉的做法是把它们全部塞进模型窗口。但容量增加并不等于利用可靠：旧目标、冲突证据和恶意内容会与真正重要的约束争夺注意力，还会同时推高成本与攻击面。

这一章把前面三章汇合成 Context 工程判断：怎样区分“装得下”和“用得好”，怎样看待采样控制，以及压缩后哪些权威状态必须留在模型之外。

## 学习目标

- 区分标称窗口与有效上下文。
- 理解采样参数和多次试验的作用。
- 能识别截断、上下文污染和长上下文错觉。

## 1. 上下文窗口是容量约束

一次请求的总预算通常包含输入、输出以及模型内部或 API 暴露的 reasoning 相关 Token。具体计数和截断策略由模型/API 决定，实施时必须查目标模型文档。

必须区分：

- **标称上下文窗口（nominal context window）**：接口允许的最大容量。
- **有效上下文（effective context）**：在具体任务、位置和干扰下能可靠利用的范围。
- **对话状态（conversation state）**：跨调用保存或重放的 item。
- **外部记忆（external memory）**：应用持久存储、尚未进入本次采样的信息。

## 2. 能容纳不等于能利用

“Lost in the Middle”等研究表明，相关信息的位置会影响表现。更长上下文还可能带来：

- 干扰、冲突和注意力稀释。
- 更高成本、TTFT 和缓存占用。
- 更大的提示词注入（Prompt Injection）暴露面。
- 历史中过期权限或旧目标继续影响决策。

因此上下文工程（Context Engineering）的目标不是塞满窗口，而是选择最小的高信号信息集合。

## 3. 采样控制

- Greedy/argmax：每步选最高概率候选。
- Temperature：改变分布集中程度。
- Top-k：只在最高 k 个候选中采样。
- Top-p：保留累计概率达到阈值的动态候选集。

并非所有模型都开放相同参数；不要默认同时调多个旋钮。`temperature=0` 也不是托管系统端到端确定性承诺。

## 4. 截断、压缩与摘要

摘要和上下文压缩（compaction）是有损变换。最容易丢失的通常是：

- 否定条件、例外和准确数值。
- 权限来源、有效期和审批范围。
- 尚未完成的承诺与失败原因。
- 引用和证据的对应关系。

压缩结果应被当作衍生物，保留来源、版本和可回溯指针；权威状态不应只存在于自然语言摘要。

## 微实验

构造同一个事实问题：把证据分别放在上下文开头、中间和末尾，再逐步加入无关文档和冲突版本。每种条件运行多次，记录：召回正确率、引用正确率、Token、延迟和错误类型。

## 常见误区

- 一百万 Token 窗口代表任意位置都能完美回忆。
- 把所有资料放进 Prompt 总优于检索。
- 更低 temperature 必然更正确。
- 摘要可以无损替代原始状态。
- 自动截断会保留业务上最重要的信息。

## 章末检查

## 本章小结

上下文窗口只是容量上限，有效 Context 取决于位置、干扰、选择与任务；摘要又是有损派生物，不能承载唯一的权威状态。至此，模型原理已经足够支撑 L0 的 API 实验与后续 Runtime 设计。下一部分进入[评测器、试次与统计](/masterpiece-static-docs/03-评测与实验科学/01-Grader-Trial与统计.md)，先建立判断改动是否有效的共同尺度，而不是继续扩展模型理论。
