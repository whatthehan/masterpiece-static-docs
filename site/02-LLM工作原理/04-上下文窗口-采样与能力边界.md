# 04 · 上下文窗口、采样与能力边界

## 学习目标

- 区分标称窗口与有效上下文。
- 理解采样参数和多次试验的作用。
- 能识别截断、上下文污染和长上下文错觉。

## 1. 上下文窗口是容量约束

一次请求的总预算通常包含输入、输出以及模型内部或 API 暴露的 reasoning 相关 Token。具体计数和截断策略由模型/API 决定，实施时必须查目标模型文档。

必须区分：

- **Nominal context window**：接口允许的最大容量。
- **Effective context**：在具体任务、位置和干扰下能可靠利用的范围。
- **Conversation state**：跨调用保存或重放的 item。
- **External memory**：应用持久存储、尚未进入本次采样的信息。

## 2. 能容纳不等于能利用

“Lost in the Middle”等研究表明，相关信息的位置会影响表现。更长上下文还可能带来：

- 干扰、冲突和注意力稀释。
- 更高成本、TTFT 和缓存占用。
- 更大的 Prompt Injection 暴露面。
- 历史中过期权限或旧目标继续影响决策。

因此 Context Engineering 的目标不是塞满窗口，而是选择最小的高信号信息集合。

## 3. 采样控制

- Greedy/argmax：每步选最高概率候选。
- Temperature：改变分布集中程度。
- Top-k：只在最高 k 个候选中采样。
- Top-p：保留累计概率达到阈值的动态候选集。

并非所有模型都开放相同参数；不要默认同时调多个旋钮。`temperature=0` 也不是托管系统端到端确定性承诺。

## 4. 截断、压缩与摘要

摘要和 compaction 是有损变换。最容易丢失的通常是：

- 否定条件、例外和准确数值。
- 权限来源、有效期和审批范围。
- 尚未完成的承诺与失败原因。
- 引用和证据的对应关系。

压缩结果应被当作衍生物，保留来源、版本和可回溯指针；权威状态不应只存在于自然语言摘要。

## 微实验

构造同一个事实问题：把证据分别放在上下文开头、中间和末尾，再逐步加入无关文档和冲突版本。每种条件运行多次，记录：召回正确率、引用正确率、Token、延迟和错误类型。

## 常见误区

- 一百万 Token 窗口代表任意位置都能完美回忆。
- 把所有资料放进 Prompt 总优于检索。
- 更低 temperature 必然更正确。
- 摘要可以无损替代原始状态。
- 自动截断会保留业务上最重要的信息。

## 章末检查

1. 标称窗口与有效上下文有什么差异？
2. 为什么长历史会同时降低质量和扩大攻击面？
3. 哪些状态字段绝不能只保存在压缩摘要里？
