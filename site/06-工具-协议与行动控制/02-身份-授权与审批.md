# 02 · 身份、授权与审批

## 学习目标

- 区分 Authentication、Authorization、Consent、Delegation 和 Approval。
- 能防止 Confused Deputy、跨租户访问和审批 TOCTOU。
- 把策略执行放在模型之外。

## 1. 五个不同问题

| 概念             | 回答的问题                              |
| -------------- | ---------------------------------- |
| Authentication | 调用者是谁？                             |
| Authorization  | 此 actor 此刻能否对该 resource 执行 action？ |
| Consent        | 数据主体是否同意某种数据处理/共享？                 |
| Delegation     | Agent 代表谁、在什么范围和期限内行动？             |
| Approval       | 用户是否确认这一笔具体、可理解的动作？                |

用户已登录不代表有权访问所有资源；用户批准也不能赋予后端本不存在的权限。

## 2. 授权决策模型

```text
decision = policy(
  actor, tenant, resource, action,
  purpose, scope, environment,
  resource_version, time, risk
)
```

工具服务端必须在每次执行时重新判断。不要信任模型生成的 `user_id`、`role=admin` 或“用户已授权”。默认拒绝，显式允许。

## 3. Confused Deputy

高权限 Agent/代理可能替低权限请求者使用自身凭证。典型路径：

```text
attacker/untrusted content
→ asks agent/proxy
→ proxy uses its broad token
→ privileged downstream action
```

缓解包括：保留原始 actor 与委派链、token 绑定目标 audience、按资源/动作缩小 scope、不透传上游 token、下游重新授权。

## 4. 风险策略

策略结果可归一为：

```text
AUTO_ALLOW | REQUIRE_APPROVAL | DENY
```

读取、草案、可逆内部动作、高影响外部写入、权限变更和不可逆操作应有不同风险级别。模型可以提供风险相关特征，但最终决策由策略引擎或确定性代码作出。

## 5. 审批必须绑定不可变提案

```text
actor + tool + exact_arguments + target
+ expected_effect/diff + disclosure + risk
+ resource_version + expiry
+ proposal_hash + idempotency_key
```

参数、目标、资源版本或有效期发生变化，审批失效。执行前再次认证、授权和校验前置条件，防止 Time-of-Check to Time-of-Use。

## 6. 审批疲劳

每次弹窗会让用户机械确认。先用最小权限、沙箱、预授权低风险能力和批量作用域限制降低风险，只把有意义的具体决策交给人。审批 UI 应展示 diff、目标、数据披露和不可逆效果，而不是“是否允许工具 X”。

## 纸面微实验（30–45 分钟）

模拟用户批准“订单 A 退款 10 元”，随后模型改成订单 B 退款 1000 元；再模拟资源版本变化、审批过期和重复恢复。画出 actor–resource–action 决策和审批状态转移。任何 proposal hash 或前置条件不匹配都必须重新审批。

## 常见误区

- 有 OAuth scope 就完成业务授权。
- 用户点击批准后无需后端检查。
- 批准工具名等于批准任意参数。
- 内部 Agent 之间可以默认互信。
- 权限在 Run 开始时检查一次即可。
