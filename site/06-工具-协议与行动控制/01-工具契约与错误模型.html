#01 · 工具契约与错误模型

Agent 真正获得行动能力的分界，不在于模型能否生成一次工具调用（Tool Call），而在于 Runtime 能否把这次提案解释为可验证、可授权、可恢复的执行请求。以“取消订单”为贯穿案例：模型可以提出取消，但订单归属、资源版本、重试语义、审批与回执必须由确定性系统掌握。

本章先建立工具契约这条边界。后续的身份授权、协议互操作、幂等与沙箱，都会在这份契约上继续收紧；如果工具在这里仍只是一个名称和 JSON Schema，后面的安全与可靠性控制便无处落地。

#学习目标

  • 把 Tool 设计成模型与确定性执行系统之间的协议。
  • 同时描述模型 affordance、Runtime 语义和用户可见效果。
  • 能区分可重试、不可重试、等待人工和取消。

#1. 三层契约

同一个工具会被模型、Runtime 和用户从三个角度理解。三层可以共享字段，却不能被压缩成一份只描述参数形状的 Schema。

#模型可理解层

  • 稳定名称、用途、使用时机和禁止条件。
  • 输入/输出 Schema 与少量边界示例。
  • 与相似工具的区别。

#Runtime 执行层

  • 业务校验、身份、权限和策略。
  • timeout、deadline、cancellation。
  • retryability、idempotency、version/precondition。
  • concurrency/rate limit、审计和敏感字段。

#用户效果层

  • 读取、生成草案还是提交变化。
  • 影响对象、范围、第三方数据披露。
  • 可逆性、审批、回执和补偿方法。

只写一个 JSON Schema 不构成完整工具契约。

#2. 工具分类

QUERY      只读查询，但仍可能泄露数据或产生费用
COMPUTE    确定性转换，无外部持久副作用
DRAFT      生成待审阅提案或 diff
COMMAND    修改外部状态
PRIVILEGED 需要提升权限或高风险环境
IRREVERSIBLE 难以撤销或外部可见

高影响工具优先拆成 draft → review/approve → commit。避免一个 manage_customer 同时查询、修改、退款和发送消息。

#3. 输入设计

  • 让非法状态难以表示：枚举优于冲突布尔组合。
  • actor、tenant、凭证、已知 resource ID 由 Runtime 注入。
  • 模型只填写需要判断的最小字段。
  • 写入携带 idempotency_key 与 expected_version。
  • 对金额、数量、日期、URL 和自由文本定义语义上限。

#4. 输出设计

status
structured_data
source_refs / observed_at
warnings
receipt / diff
resource_version
next_cursor
error_code / retryable

大结果返回游标或工件引用。工具输出是外部观察，可能陈旧、错误或含恶意指令，进入模型前应裁剪并标记来源。

#5. 错误分类

类别示例是否重试
INVALID_ARGUMENTSchema/领域参数错误否；修正参数
NOT_FOUND/CONFLICT实体不存在、版本冲突通常否；重新观察
UNAUTHENTICATED/DENIED身份或权限不满足否；重新授权/升级
APPROVAL_REQUIRED需要具体人工决定暂停，不是失败
RATE_LIMITED/UNAVAILABLE限流、瞬时服务故障有界退避重试
DEADLINE_EXCEEDED未知是否已产生效果先查询/去重，不盲重试
CANCEL_REQUESTED/CANCELLED用户或上游停止停止新工作；核对在途效果
IN_DOUBT连接中断，无法确认 command 结果查询 receipt/业务状态，进入 reconciliation
INTERNAL/PERMANENT程序错误、不可恢复失败并告警

#6. 输入合法不等于可安全执行

Schema 只证明“形状可解析”。执行前还要在目标系统的 enforcement point 做语义、授权和 sink-specific 校验：

Sink确定性控制
SQL参数化查询、限定语句/表/租户，不拼接模型文本
Shell/进程固定 executable 与分离 argv；对 option/值/路径做 allowlist 与领域校验；目标程序支持时用 -- 终止 option 解析；禁止任意 shell 解析
文件路径限定根目录，规范化后再校验，处理 symlink/TOCTOU
URL/HTTP解析后按 scheme/host/IP/port 限制，每次 redirect 重新校验，阻断 metadata/private network
HTML/消息按最终输出上下文编码，不传递模型生成的“已清洗”标记
序列化只接受允许类型和大小，禁止危险对象恢复/动态代码路径

#纸面微实验（30–45 分钟）

把模糊的 manage_order 拆成 search_orders / get_order / draft_cancel_order / commit_cancel_order。构造不存在、跨租户、版本冲突、重复提交、提交后丢 ACK，以及“Schema 合法但 URL 指向内网”案例。产出工具契约、错误类型、enforcement point 和下一状态；任一高风险参数只交给模型判断即不通过。

#常见误区

  • HTTP 200 等于业务成功。
  • Read-only Tool 没有安全风险。
  • 模型最适合判断错误是否可重试。
  • 工具越原子、越接近底层 API 越适合模型。
  • 把错误文本返回模型就足以处理失败。

#章末检查

  1. 模型可理解层与执行层为何不能合并？
  2. timeout 后为什么不能直接断定 command 没发生？
  3. 哪些参数必须由 Runtime 注入？

#本章小结

工具契约的核心判断是：Schema 只约束模型提案的形状，Runtime 契约与用户效果契约才决定它能否执行、失败后如何收敛。下一章将 actor、resource、action 与具体提案纳入身份、授权与审批，回答“谁可以执行这一笔动作”。
