#01 · 如何使用这套教材

#1. 学习目标不是记忆术语

Agent 工程的困难来自两套系统叠加：

  • 概率系统：模型输出会随输入、上下文位置、采样和模型版本变化。
  • 确定性系统：权限、资金、消息、数据库和外部 API 要求明确、可审计的语义。

真正的能力是知道哪部分可以交给模型判断，哪部分必须由程序、策略或人控制。

模型擅长：理解、归纳、生成候选、模糊匹配、在开放空间中搜索
程序擅长：验证、授权、计数、事务、执行、记录、强制不变量
人擅长：承担责任、处理价值冲突、批准高风险或目标不清的行动

#2. 三种掌握等级

#A · 能解释

不用公式堆砌，能用自己的话说明机制、边界和失败方式。例如：为什么增加上下文长度可能让答案更差。

#B · 能证伪

能设计一个最小实验，使错误理解暴露。例如：构造一个满足 JSON Schema、但业务上越权的工具参数。

#C · 能设计门禁

能把认识转成系统约束。例如：工具执行前重新计算 actor-resource-action 权限，而不是相信模型声称“用户已授权”。

前置理论的毕业标准是核心章节达到 B，安全、权限和副作用章节达到 C。

统一验收规则：

  • B 级通过：能说明机制与边界，给出一个会失败的反例，并写出可观察的预期证据。
  • C 级通过：能指出 enforcement point、输入、允许/拒绝规则、真实效果与自动测试；只写“加强 Prompt”不通过。

#3. 推荐学习循环

每章使用同一循环：

  1. 写下当前直觉，不查答案。
  2. 阅读核心模型，画一张结构图。
  3. 做微实验或故障推演。
  4. 回答章末检查，记录仍不确定的部分。
  5. 把结论应用到 M0 的任务规格和风险清单。

不要只做摘要。摘要容易制造“读懂了”的错觉，反例和门禁才能检验理解。

练习分两类：

  • 前置微实验/桌面推演：30–45 分钟，使用纸面状态、少量 API 调用或固定 mock；属于首次动手门禁。
  • L1 后系统实验：需要持久运行时、沙箱、并发 harness 或 TS/Rust 双实现；只记录设计，完成 L1 后再执行。

#4. 贯穿全书的案例

默认案例是“研究—判断—行动 Agent”：

  • 用户给出目标和约束。
  • Agent 检索内部与公开资料，形成带来源的判断。
  • Agent 可以生成草案和预览。
  • 任何外部写操作都必须经过策略校验；高风险操作必须人工批准。

它不是某个垂直 SaaS，而是一个能承载检索、工具、状态、审批、记忆和可靠性的通用任务族。你可以替换为客服、销售、数据分析或个人效率场景，但整套门禁不变。

#5. 学习记录模板

# 章节学习记录

## 一句话模型
这个概念解决什么问题？

## 系统边界
它保证什么？不保证什么？

## 一个反例
怎样让错误理解失败？

## 对当前 Agent 的影响
需要增加什么数据、状态、策略、评测或观测？

## 尚未确认
哪些结论仍需实验或一手资料？

#6. 进入动手阶段的最低证据

  • 完成一份任务契约和 30–50 个初始案例。
  • 能画出模型、运行时、工具、策略、状态和人的信任边界。
  • 能说明 Schema 验证、语义验证、授权和审批的不同职责。
  • 能区分等待澄清、等待审批、拒绝、失败、取消请求、未知效果收敛、部分完成和预算耗尽。
  • 能为随机输出设计多 trial 评测。
  • 能列出至少五种间接 Prompt Injection 路径及系统级缓解。
  • 能解释重试为什么可能重复副作用。
  • 能说明首次实现为什么不应包含 Multi-Agent、Durable Workflow 或 Rust 迁移。

#章末检查

  1. 为什么“先学完所有理论再实践”与“完全不学理论直接写框架”都不可靠？
  2. 对一个退款 Agent，模型、程序和人分别应该承担哪些决定？
  3. 什么证据能证明你不是只会复述术语？
