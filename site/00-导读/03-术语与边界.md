# 03 · 术语与边界

这是一份贯穿全书的随查词典，**首次顺读可以跳过**。遇到陌生词或边界冲突时再回来核对：同一个词究竟指模型行为、应用状态、Harness 设施、协议对象，还是外部世界中的真实效果。

## 核心术语

| 术语                       | 本书中的精确定义                                                                    | 不等于                            |
| ------------------------ | --------------------------------------------------------------------------- | ------------------------------ |
| Model                    | 根据上下文生成响应 item 或候选动作的概率系统                                                   | Agent、数据库、权限系统                 |
| Augmented LLM            | 单次模型调用接入检索、工具或记忆能力，但流程未必由模型循环控制                                             | 必然是 Agent                      |
| Workflow                 | 由代码预先决定主要控制流的过程                                                             | 模型动态决定全部步骤                     |
| Agent                    | 模型在受约束循环中根据观察动态选择下一步动作的系统                                                   | 无边界的“自主 AI”                    |
| Agent Runtime            | 管理一次 Run 内 Loop、状态、工具、预算、取消、事件和错误的确定性程序                                     | Prompt 集合                      |
| Agent Harness            | 把模型、Context Builder、Runtime、工具、环境、权限、状态、压缩和 Trace 组合起来，使模型能够作为 Agent 行动的脚手架 | 模型本身、评测平台                      |
| Agent Application Server | 把 Thread / Run / Item、权威状态、语义事件、审批、取消与多种客户端连接起来的产品服务层                       | 模型提供方的对话状态、一条 SSE 连接           |
| Evaluation Harness       | 运行任务、准备环境、记录轨迹、执行 Grader 并聚合结果的评测设施                                         | 线上 Agent Harness；两者可以共享组件但目标不同 |
| Agent Workbench          | 全书持续演进的可运行 Agent 应用骨架及其 Task、Dataset、Trace、策略和产品界面                          | 每章一个互不相干的 Demo                 |
| Context                  | 某一次模型采样实际可见的 Token 集合                                                       | 全部历史、长期记忆                      |
| Context Engineering      | 为当前推理选择、组织、检索、压缩、外置和刷新最小充分 Context 的工程                                      | 只改 Prompt 措辞、把窗口塞满             |
| Harness Engineering      | 设计并验证模型周围的 Context、Loop、工具、权限、状态、环境与反馈设施                                    | 一个统一范围的行业标准、单纯 Prompt 调优       |
| Inner Agent Loop         | 一次 Run/Turn 内“推理—提议—执行—观察”的有界反馈循环                                           | 无限 `while`、跨天任务调度              |
| Outer Orchestration Loop | 跨 Run 触发任务、隔离工作区、调用 Harness、独立验证、保存状态并决定下一轮的工作闭环；本书教学抽象                     | 已形成统一定义的“下一代 Agent 架构”         |
| State                    | 应用判断当前运行位置所需的权威数据                                                           | 聊天文本摘要                         |
| Knowledge                | 可检索、有来源、有权限和时效属性的外部事实                                                       | 用户偏好记忆                         |
| Memory                   | 跨步骤或跨会话保留、按策略写入和读取的信息                                                       | 无限增长的对话历史                      |
| Tool                     | 具有输入/输出契约的查询或动作能力                                                           | 任意函数包装器                        |
| Resource                 | 可供应用或模型读取的上下文数据                                                             | 可执行动作                          |
| Skill                    | 按需加载的程序性知识包，可包含指令、参考、脚本和资产，并应有作用域、版本与信任来源                                   | 自动授权、任意时刻全量注入的 Prompt          |
| Hook                     | 在 Harness 生命周期事件上运行的确定性扩展点；是否可阻断取决于具体事件与返回协议                                | 所有 Hook 都是策略强制点、能撤销已发生效果       |
| Prompt                   | 指令、示例和格式约束的文本组成                                                             | 安全策略的强制执行点                     |
| Policy                   | 独立于模型、可审计的允许/拒绝规则                                                           | Prompt 中的“请勿”                  |
| Approval                 | 用户对具体、可理解、未被替换的动作内容确认                                                       | 身份认证或永久授权                      |
| Trace                    | 一次运行跨模型、工具和服务的结构化因果记录                                                       | 原始 Chain-of-Thought            |
| Eval                     | 用任务、环境、grader 和 trial 测量系统表现的过程                                             | 人工随便试几次                        |

## 路线与工程术语

| 术语                 | 含义                                                                                      |                                            |
| ------------------ | --------------------------------------------------------------------------------------- | ------------------------------------------ |
| M0 准备（非里程碑）        | 先写 3 个锚点、非 Agent baseline 和 Outcome Grader，再用一次无工具的 L0 前置接口探针暴露任务歧义；它不是公共阶段或职业等级        |                                            |
| M0                 | 正式任务与评测基线：Task Contract、非 Agent baseline、平衡 seed cases 和 30–50 个版本化案例已冻结，退出后才进入 L0 里程碑  |                                            |
| L0                 | 使用官方模型 SDK 看清 request/response/item/stream、Schema、usage 与错误语义的接口里程碑；尚未等同于 Agent Runtime |                                            |
| L1                 | 不依赖 Agent 框架，手写有状态、预算、取消、Trace 和工具校验的单 Agent Runtime                                    |                                            |
| Permission         | Harness/产品对某类工具、文件、网络或执行能力的技术允许/询问规则                                                    | 领域 Authorization；能运行工具不代表当前 actor 有权操作目标资源 |
| ACL                | Access Control List，资源访问控制条件；检索必须在授权范围内生成候选                                             |                                            |
| SSE                | Server-Sent Events，服务器通过有序事件流向客户端推送数据；TCP chunk 不等于完整 SSE event                         |                                            |
| Canonical RunEvent | 应用内部稳定、带版本的运行事件；供应商流事件和 UI 协议均通过 adapter 与它转换                                           | 直接复用某个 SDK 的 event type、TCP chunk          |
| Affordance         | 工具向模型表达“能做什么、何时使用、效果是什么”的可用性契约                                                          |                                            |
| Tombstone          | 表示对象已删除/失效的持久标记，用于阻止缓存或索引再次暴露旧数据                                                        |                                            |
| Canary             | 只把少量受控流量路由到新版本，以测量风险后再扩大                                                                |                                            |
| Reducer            | 根据当前 state 与 event 纯函数式地产生下一 state 的逻辑                                                  |                                            |
| Sidecar            | 与主服务分进程部署、通过稳定协议提供特定能力的邻接服务                                                             |                                            |
| N-API              | Node 原生扩展 ABI；用于进程内 Rust/C/C++ 扩展，不等于服务隔离                                               |                                            |
| Enforcement point  | 真正强制规则的位置，如工具服务端授权或网络 egress policy，而非文字建议                                              |                                            |

## 三组必须保持的区分

### 1. 输出、建议与效果

```text
模型文本："退款已完成"
工具建议：refund(order=123, amount=100)
真实效果：支付系统中是否只产生了一次合法退款
```

评测应优先检查真实效果，而不是模型声明。

### 2. 身份、权限与批准

- Authentication：调用者是谁。
- Authorization：该 actor 此刻能否对该 resource 执行该 action。
- Approval：用户是否理解并同意这一笔具体动作。

三者不能互相替代。

### 3. 可靠性、安全与能力

- 能力：在理想条件下能不能完成任务。
- 可靠性：面对随机性和部分失败能否稳定完成或安全停止。
- 安全：面对错误、滥用或攻击时是否保持不变量。

提高模型能力可能同时扩大可执行动作的攻击面，因此三者必须分别评测。

## 关于“推理”的边界

应用可以记录计划、工具选择、证据、状态变化和简洁决策摘要，但不应依赖、展示或持久化模型隐藏的原始 Chain-of-Thought。理由不是“模型没有推理”，而是：

- 不同模型暴露的推理 item 语义不同，接口会演进。
- 自然语言推理痕迹不是完整、忠实的因果证明。
- 它可能包含敏感上下文。
- 可审计系统需要结构化事件和证据，而不是不可验证的内心独白。

## 本章小结

可靠的 Agent 工程首先依赖边界清晰：Context 不等于 Harness，Loop 不等于无限自治，输出不是真实效果，批准也不等于授权。现在进入 M0，用少量熟悉任务建立第一个可比较基线；需要时再回到本页查词。
