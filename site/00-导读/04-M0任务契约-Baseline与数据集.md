# 04 · M0：任务规格、基线（Baseline）与数据集

转型的真正起点不是第一次调用模型，而是先回答“系统怎样才算成功，以及怎样证明”。如果这一步缺失，后续增加工具、记忆或多 Agent 只会放大一个尚未定义清楚的问题。

对贯穿全书的研究—判断—行动案例，M0 会先固定允许读取的资料、必须审批的动作、真实结果的检查位置和更简单的基线。此后每一阶段的技术变化，都必须回到同一批任务和风险上比较。

M0 分两道门禁收敛：任何模型探针前先用 12–20 个平衡种子案例通过 G0a（M0 草案）；扩展并冻结 30–50 个版本化案例后，才算正式通过 M0，并可进入 L0 SDK 实践。理论阅读和纸面微实验可以在两道门禁之间穿插。

## 学习目标

- 在 Prompt 和 Agent 之前定义成功。
- 建立可复现的非 Agent baseline。
- 设计覆盖正常、边界、失败和攻击的初始数据集。

## 1. 评测对象

```text
Task：一个有输入、环境与成功标准的问题
Trial：对 Task 的一次独立尝试
Outcome：尝试结束后的真实环境状态
Trace/Trajectory：过程中发生的结构化记录
Grader：检查某一成功维度的逻辑
Harness：提供环境、运行、记录和评分的设施
Suite：围绕能力或风险组织的一组 Tasks
```

评测的是 `model + prompt + context + tools + runtime + policy + environment` 的组合，不要把系统得分错误归因于模型单体。

## 2. Task Contract

每个任务族至少定义：

- 输入、可用信息、初始环境和用户身份。
- 期望结果与允许存在的多个正确答案。
- 允许动作、禁止动作和人工升级条件。
- 业务、安全、成本和延迟约束。
- 真实 outcome 的检查方法。
- 任务是否可解以及一个参考解决路径。

成功标准不清时，Agent 只是在优化模糊印象。

### 填写示例：研究—判断—行动

```text
用户价值：基于获准资料形成带引用判断，并生成待批准邮件草案。
允许：检索当前租户文档、公开网页、生成草案和引用表。
必须审批：向任何外部收件人发送邮件。
禁止：读取其他租户；把内部原文发送给公共工具；自行扩大目标。
Clarification：目标、收件人或时间范围缺失。
Outcome：报告 claims 均有来源；邮件仅停留在 draft；无跨租户读取。
Baseline：一次模型调用，只使用用户粘贴的资料，不调用工具。
预算：记录上限，不在 M0 阶段凭感觉设“无限”。
```

### G0a / M0 验收 Rubric

每项 0/1/2 分，至少 16/20，且“安全不变量”“Outcome 检查”不得为 0：

| 维度       | 0     | 1        | 2                        |
| -------- | ----- | -------- | ------------------------ |
| 用户价值     | 只有功能名 | 有用户与目标   | 有可验证真实结果                 |
| 输入/环境    | 未定义   | 定义主要输入   | 含身份、来源、缺失/冲突             |
| 允许/禁止    | 无边界   | 只有允许项    | 自动/审批/拒绝清晰               |
| Outcome  | 看模型文本 | 部分规则     | 查询权威环境和业务状态              |
| 安全不变量    | 无     | 泛化“要安全”  | 可测试的数据/动作不变量             |
| 人工控制     | 无     | 只有审批     | 澄清/审批/升级分开               |
| Baseline | 无     | 随意人工对比   | 可重复的更简单方案                |
| 案例平衡     | 只有正例  | 有少量边界    | 正反、失败、攻击均覆盖              |
| 版本       | 无     | 只记模型     | 关键组件/数据/环境齐全             |
| 预算       | 无上限   | 只有 Token | step/time/token/money 范围 |

## 3. Baseline 阶梯

```text
确定性代码/人工流程
→ 单次模型调用
→ 单次调用 + 检索
→ 固定 Workflow
→ Agent
```

每次增加自主性，都要相对前一层证明任务收益足以覆盖成本、延迟和风险。

## 4. 初始数据集

30–50 个高质量任务足以开始，建议分层：

- 正常：代表主要真实流量。
- 模糊：缺参数、冲突目标、多种解释。
- 边界：金额、长度、时间、权限边缘。
- 失败：超时、限流、空结果、部分结果。
- 拒绝/升级：必须停止、澄清或转人工。
- 对抗：直接/间接注入、越权、数据泄漏。
- 反向案例：某行为不应发生，例如不该搜索、不该调用工具。

只测试“应该调用”会优化出过度调用；数据集必须平衡正反方向。

## 5. 数据集分区

- Development：日常诊断和迭代。
- Regression：每个已知生产/测试故障沉淀成案例。
- Holdout：阶段门禁，减少过拟合。
- Shadow/Production samples：验证真实分布，需脱敏、授权和保留策略。

## 前置微实验（45 分钟）

先为贯穿案例写 12–20 个平衡种子任务，不调用模型，完成 Task Contract、风险和非 Agent baseline 方案，并按 Rubric 通过 G0a。之后才运行受控的单次模型、无工具探针，记录准确、拒绝、延迟、Token、成本和失败分类；再补齐并冻结 30–50 个版本化案例，正式退出 M0。此时仍不急于写 Agent。

通过证据：G0a 的 12–20 个任务和正式 M0 的 30–50 个任务都有明确输入、预期 outcome、至少一个 grader 和版本字段；不能用“看起来合理”评分。

## 常见误区

- 先实现，再决定如何判断成功。
- 测试数量越多越科学；含糊任务只会增加噪声。
- 所有任务只需一个标准答案。
- Agent 自称完成即可作为 outcome。
- 只测试应该做的行为，不测试应该保持克制的行为。

## 章末检查

1. Task、Trial 与 Outcome 有什么差异？
2. 为什么 baseline 必须包含更简单的非 Agent 方案？
3. 参考解为什么能帮助发现坏任务或坏 grader？
