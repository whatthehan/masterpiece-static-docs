#04 · 八周理论学习计划

默认每周 8–12 小时：4–6 小时阅读，2–3 小时闭卷复述/画图，2–3 小时纸面微实验与产出。时间不足就延长周数，不压缩门禁。本计划的目标是获得“开始第一个 Agent 的理论资格”，不在理论期搭建生产系统。

#第 1 周 · 先定义问题和证据

#学习（4–5 小时）

  • 00 导读全部，尤其是 M0 Task Contract、baseline、case、trial、grader、slice、gate。
  • 03 评测模块先读概念和流程；统计细节留到第 3 周。
  • 选定贯穿任务：建议“研究—判断—生成动作草案”，不接真实写工具。

#产出（3–4 小时）

  • 一份填完的 M0：用户、任务族、输入/输出、成功/失败、禁止动作、预算、数据切片和发布门禁。
  • 一个非 Agent baseline 方案：搜索+模板、单次 LLM 或确定性 workflow。
  • 12–20 个 seed cases：正常、模糊、边界、拒绝、失败和攻击至少各 2 个。
  • 一张 case → trial → grader → metric → gate 图。

#门禁（1 小时）

不借助资料说明“什么结果算成功”和“为什么需要 Agent”。M0 初版按导读评分至少 16/20，且结果与安全项不得为 0。未通过时不进行模型探针/选型；完整 30–50 案例在第 3 周收敛，并于首个 Agent 实现前冻结版本。

#第 2 周 · 模型究竟产生什么

#学习（5–6 小时）

  • 01 概率、信息量、向量/嵌入、训练/泛化/分布偏移。
  • 02 Tokenization、Transformer/causal attention、logits/sampling、KV Cache、后训练、上下文限制。

#产出（3–4 小时）

  • 一张“文本 → Token → embedding/hidden state → logits → sampling”图。
  • 一张 KV Cache / Prompt Cache / Conversation State / Memory 对照表。
  • 三份 30–45 分钟微实验记录：Tokenizer 边界、多 trial 采样、信息位置变化。
  • 将“模型概率高”“模型说得像”等错误 grader 改写为真实 outcome 证据。

#门禁（1 小时）

闭卷解释：Token probability 为什么不是真值；causal mask 保证什么；标称窗口为什么不是有效上下文；低温度为什么不是系统确定性。四题任一不能给出反例则重学。

#第 3 周 · 评测统计、TypeScript/Node 和模型接口

#学习（6–7 小时）

  • 03 完整：Outcome/Trajectory、trial 层级、人工金标/Judge 校准、Wilson/exact、rule of three、MDE/power、paired/cluster analysis、多重比较与顺序停止。
  • 04/01–05 按顺序学习：TypeScript + Node 运行时、指令/Prompt/Context、模型 API/stream event、JSON Schema、Structured Outputs 和工具调用。

#产出（3–4 小时）

  • 将 seed set 扩到 30–50 个 case，标记 task/trial ID、slice、grader 和版本字段。
  • 一张样本量/MDE/区间/顺序停止策略卡。
  • 一个完整 JSON Schema，含 missing/null、enum、additionalProperties、正反样例和版本策略。
  • 一张“TCP chunk → SDK event → reducer → complete Item”图。
  • 一张 Node event loop、stream/backpressure、AbortSignal、discriminated union 责任表。

#门禁（1 小时）

能解释为什么不用 Wald 近似处理小样本/极端率；为什么 trial 不是独立 task；为什么 Schema 合法不等于语义安全；为什么断流的半个 tool arguments 不能执行。

#第 4 周 · 受控 Agent Loop 与 Context Engineering

#学习（5–6 小时）

  • 04/06 Agent Loop 与状态机；04/07 架构模式及多 Agent 边界；04/08 框架/SDK 学习优先级。
  • 05/01 Context Engineering：选择、组织、压缩、传播与信任标记。
  • 只学 multi-agent 风险和委派 envelope，不实现多 Agent。

#产出（3–4 小时）

  • 一张 Thread/Run/Item/Event + snapshot/event 图。
  • 一张 state × event → guard/effect/next 表，包含 WAITING_INPUT、WAITING_APPROVAL、CANCEL_REQUESTED、IN_DOUBT、RECONCILING 与表意明确的终态。
  • 八条 Runtime 不变量与五条非法转移。
  • 一份 Context Manifest：来源、信任、新鲜度、预算、裁剪理由和稳定/动态区分。

#门禁（1 小时）

手工推演：正常完成、需补充信息、审批拒绝、工具提交后丢 ACK、用户取消、预算耗尽。任一路径允许模型直接行动或终态继续产生工作，则不通过。

#第 5 周 · 来源、检索、RAG 与 Memory

#学习（5–6 小时）

  • 05/02–04：Provenance、ACL/freshness、sparse/dense/hybrid retrieval、rerank、Context packing、State/Memory/Compression。
  • 重点理解 ACL-scoped candidate generation、后置过滤的 recall starvation/旁信道、Memory 写入策略与删除传播。

#产出（3–4 小时）

  • 一张“query → ACL-scoped candidate generation → rank → defensive recheck → pack → cite”图。
  • 一个 provenance record schema：source ID、observed_at、trust、tenant/ACL、version/hash、citation span。
  • Memory write/read/delete policy，覆盖 consent、namespace、conflict、staleness、cache/index/summary/backup。
  • 检索与 Memory 评测表：authorized recall/precision、stale/conflict、leakage、downstream utility、deletion effectiveness。

#门禁（1 小时）

能画出并解释两个失败：无权高相似文档占满 top-k；用户删除后 vector cache 仍命中。若解法只是“最后再过滤”或“删掉对话”，则不通过。

#第 6 周 · 工具、身份、协议与副作用控制

#学习（6–7 小时）

  • 06 全部：三层工具契约、错误模型、身份/授权/委派/审批、MCP、幂等/补偿/沙箱。
  • 对 MCP 只学稳定 2025-11-25 的角色、能力和安全责任；将 2026-07-28 RC 当作迁移预告，不当实施基线。
  • Sink-specific safety：SQL、shell/argv、path/symlink、URL/redirect/SSRF、HTML 编码、危险反序列化。

#产出（3–4 小时）

  • 一个 query 和一个 command 的完整工具契约：模型层、Runtime 层、用户效果层。
  • actor–tenant–resource–action–purpose 授权矩阵和委派链。
  • 一个绑定精确提案的 approval record 与 TOCTOU 反例。
  • 一张 MCP Host/Client/Server + Tool/Resource/Prompt 图。
  • 一张“commit → ACK loss → IN_DOUBT → reconcile”时序和 sink 校验表。

#门禁（1 小时）

能证明：Schema 合法仍可能越权/SSRF/注入；OAuth scope 不等于业务授权；Approval 不等于 Authorization；timeout 后盲试会重复副作用；Rust/容器不等于沙箱。

#第 7 周 · 安全、可控 UX 和可靠性

#学习（6–7 小时）

  • 07 全部：威胁建模、Prompt Injection、最小权限/隐私、纵深防御/人类控制、Agent UX。
  • 08/01–02：失败分类、deadline/timeout/retry/cancel/reconciliation、并发/背压/预算。
  • 08/03 只掌握 durable 心智模型与“为什么暂不实现”；lease/fencing/outbox/DLQ 属 L1 后实施。

#产出（3–4 小时）

  • 贯穿案例的数据流/信任边界图、OWASP Agentic Top 10 2026 覆盖表和八条可测安全不变量。
  • 三层注入攻击链与 outcome/trajectory Eval。
  • 运行、等待补充、等待审批、取消核对中、部分完成五个 UX 状态草图。
  • max_attempts/max_retries 放大计算，以及 100×10 fan-out 的有界队列/超载方案。
  • 两 Worker+lease 过期的纸面推演，用于理解而非实现 durable engine。

#门禁（1 小时）

随机抽取三条攻击/故障链，每条都要说出至少两个独立 enforcement point、真实 outcome 和 UI 文案。将“模型拒绝了”当作安全结论，或将“点击 Cancel”当作已回滚，均不通过。

#第 8 周 · 观测、系统整合与毕业

#学习（4–5 小时）

  • 08/04 Trace、SLI/SLO、Audit、成本与隐私。
  • 10/01 综合系统心智模型。
  • 使用 10/06 章节验收矩阵补缺，不快速重读全部。

#产出（3–4 小时）

  • 一张端到端分层图和 Run 路径。
  • 一张 Trace tree，区分 metric/log/trace/audit/eval，并定义成功任务成本、p95/p99 与安全指标。
  • 整理 M0、30–50 cases、baseline、graders、统计策略、威胁模型、状态机、工具契约和 UX 草图为一个证据包。
  • 列出第一个 TypeScript + Node 实现的 in/out scope：3–5 个 mock/只读工具；无真实不可逆动作、无 Agent 框架、无 durable/multi-agent/Rust runtime。

#门禁（2–3 小时）

  1. 先用 20 分钟闭卷重画 10/01。
  2. 完成 10/02 闭卷检查，按 10/03 原子化评分。
  3. 总分与 A/B/C 域内门禁全过，且无硬失败，才进入首个实现。

#毕业后的首个实现（L0 → L1）

  1. 使用官方 TypeScript SDK 完成原始 request/response/stream adapter。
  2. 手写有限状态 Agent Loop，接 3–5 个 mock/只读工具。
  3. 实现类型化 event reducer、budget、cancel、bounded concurrency、trace 和 M0 自动回归。
  4. 达到自定义 L1 证据：任务质量超 baseline，关键安全不变量为零违规，失败注入可收敛，成本/延迟可接受。
  5. 之后再各自决定是否引入 LangGraph/OpenAI Agents SDK、durable engine、multi-agent 或 Rust。框架用来对照它封装了哪些已知原理，不用来替代原理。
