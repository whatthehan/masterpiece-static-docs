# 06 · 章节验收矩阵

读完一章只能证明内容曾经进入视野，不能证明它已经变成工程能力。本矩阵把全书压缩为可重建的图、表、反例和门禁，既可用于逐章验收，也可在闭卷失分后定位最小补课范围。

## 使用方法

每章学完后，不用“读过了”打勾，而是留下可复查产出。通用评分是：

- 0：无产出，或只复述名词。
- 1：能解释概念，但无边界/反例/工程决策。
- 2：能从记忆重建图/表/计算，给出反例并映射到贯穿案例。
- 3：能处理未见变体，写出可测不变量、失败证据和 enforcement point。

每行至少达到 2；标记“安全关键”的行必须达到 3。若产出依赖外部实系统，前置阶段用 mock 或纸面时序代替；不因实现重而跳过理论验收。

## 八周启动证据包

1. 3 个锚点形成最小任务草案；经过 L0 前置接口探针修正后，扩展并冻结 30–50 个版本化 cases，正式退出 M0。草案不是公共阶段，探针也不等于 L0 已通过。
2. 统计/评测计划：graders、trials、区间、MDE/power、停止与发布门禁。
3. 可运行的 TypeScript/Node L0 adapter、L1 runtime、流 event reducer、Agent 状态机和不变量。
4. Context Manifest、ACL-scoped retrieval、provenance 和 Memory 删除策略。
5. Query/Command 工具契约、授权矩阵、approval proposal、sink 安全和 ACK-loss 时序。
6. 威胁模型、攻击链、纵深防御矩阵、UX 状态映射和 incident playbook。
7. 背压/预算上界、Trace/SLO/成本设计。
8. 闭卷答卷、评分表和未清零错误模型列表。

这组证据证明读者已经完成 `M0 → L0 → L1` 并能继续构建，不证明生产系统或场景专项已经毕业。

## 全书毕业追加证据

1. 至少 100 个版本化案例，覆盖生产分布、Provider/Model 故障、安全 Protected Slice 与恢复路径。
2. 跨小时等待和 Worker 重启实证；同一业务意图没有重复效果，旧 Epoch 写入被拒绝。
3. 真实 UI 如实展示等待、审批、未知效果、核对、部分完成与人工接管。
4. Provider 故障策略经过故障注入：采用 Fallback 时通过 Eval/Contract 验证；不采用时，有界等待、拒绝、有限结果或人工接管可复查。Quota、Circuit Breaker、Stream Drain 和旧/新 Run 切流可演练。
5. Shadow、Canary、Rollback、Kill Switch、灾备恢复和配置发布状态机均留下结果与责任人。
6. 选择 1–2 个 L6 专项，按输入、工具、状态、风险和 Eval 相对更简单 Baseline 证明准入收益。
7. 全链路作品证据能由外部评审复现，而不是只展示 Demo、截图或闭卷分数。

## 本章小结

矩阵的价值不在于把所有单元格变成绿色，而在于区分“已经能开始构建”和“已经能对生产结果负责”。先用[八周综合闭卷检查](/masterpiece-static-docs/10-毕业门禁/02-动手前闭卷检查.md)定位基础缺口，再用[场景专项与全书毕业](/masterpiece-static-docs/10-毕业门禁/08-场景专项与全书毕业.md)核验生产与专项证据；闭卷高分不能替代真实恢复、发布和用户体验。
