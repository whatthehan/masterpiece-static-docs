# 03 · 参考答案与评分

本章只能在闭卷答题后使用。这里提供的是最小评分点，不是要求背诵的标准措辞；答案可以采用不同设计，只要明确覆盖同一不变量、执行位置与可观察证据。

## A · 核心概念评分点

每题两个原子要点，每点 1 分。只重复题干或举例不得分。

1. 模型给出的是给定上下文下下一 Token 的条件分布（1）；没有内置事实查询/真值校验（1）。
2. Token 是离散单位，token embedding 是初始向量（1）；contextual hidden state 随上下文改变，text embedding 是用于检索等任务的序列表示（1）。
3. Causal mask 阻止当前位置读取未来 Token，保持自回归因果结构（1）；不保证事实、安全、可解释或长文利用效果（1）。
4. KV Cache 复用本次 decode 注意力中间量，Prompt Cache 复用相同前缀计算（1）；Conversation State 保存 API 交互 item/reference，Memory 是应用治理的跨 Run 信息（1）。
5. 窗口是容量上限而非任务能力保证（1）；位置、干扰、任务复杂度、上下文组织和模型训练会影响有效利用（1）。
6. 后训练/低温度可改善行为倾向、指令遵循或降低采样变异（1）；不保证事实、授权、业务不变量或端到端确定性（1）。
7. 原始 CoT 可不忠实、含敏感数据且接口不稳定（1）；应记录目标/计划摘要、来源、动作、状态、策略决策和 outcome（1）。
8. Workflow 的控制流主要由代码预定，Agent 让模型根据 observation 选择下一步（1）；Augmented LLM 有检索/工具不代表存在自主 Loop 或模型控制流（1）。
9. M0 定义任务、可接受结果、禁止行为、数据切片和评测方法（1）；baseline 给出“不用 Agent 的更简单系统”可对比效果/成本（1）。
10. Structured Outputs 在提供方支持子集内约束结构，不保证事实、语义、授权或 sink 安全（1）；missing/显式 `null`/默认/删除是不同业务语义（1）。
11. 模型只提议；Runtime 需等完整 item 后做协议/Schema/领域语义校验（1）；随后还有服务端授权、风险/审批、sink/沙箱执行和 receipt/outcome 核对（1）。
12. TCP chunk 只是传输分段，需解析为带 ID/类型的语义 event 并用 reducer 重建 item（1）；未完整 arguments 不能验证，断流/重放也可导致重复或半执行（1）。
13. Thread 是长期交互容器，Run 是一次目标执行，Item 是语义产物（1）；Event 是增量变化，Runtime State 是从权威事件/存储得到的当前控制状态（1）。
14. Host 管应用/用户/策略，Client 连接一个 Server，Server 暴露 tool/resource/prompt 等（1）；MCP 是互操作协议，不自带 planner、业务授权、沙箱或 Agent Loop（1）。
15. Authentication=是谁，Authorization=能否对资源行动，Consent=数据主体是否同意处理（1）；Delegation=代表谁在什么范围/期限行动，Approval=是否确认这一个精确提案（1）。
16. Context=当次模型输入，State=权威运行数据，Memory=跨 Run 信息，Corpus=外部知识源（1）；ACL 必须在候选生成前缩小搜索空间，否则会泄漏/旁信道并导致 recall starvation（1）。
17. Deadline 是总时限，timeout 是一步等待上限（1）；cancellation 是停止新工作的协作信号，rollback 是撤销效果，前者不保证后者（1）。
18. 同一业务 intent 所有 retry 使用相同 key，相同 key+不同 payload 必须拒绝（1）；`IN_DOUBT` 表示 execution/effect 证据不足，必须查 receipt/权威状态而不是盲试（1）。
19. Outcome 检查真实结果，Trajectory 检查路径/工具/策略不变量（1）；trial 估计随机变异，区间表达有限样本下的不确定性（1）。
20. Discriminated union 让状态/event 可穷尽处理，event loop 要避免阻塞（1）；stream/backpressure 管流量与有界缓冲，`AbortSignal` 负责取消传播但不承诺回滚（1）。

## B · 故障场景评分点

每题 5 个独立要点，每点 1 分。

### 21. 丢失 ACK + 取消竞态

1. 持久化 `CANCEL_REQUESTED`，停止新工作，向在途调用传播（1）。
2. command 进入 `IN_DOUBT → RECONCILING`，不直接标 `CANCELLED`（1）。
3. 用原 idempotency key/call ID 查 receipt 和权威退款状态（1）。
4. 已生效则返回原回执，进入 `COMPLETED_WITH_EFFECT_AFTER_CANCEL`或补偿/人工；未生效且证据充分才 `CANCELLED`（1）。
5. 禁止新 key 盲目再退款；UI 明示“核对中/已生效”而非伪称已回滚（1）。

### 22. 间接 Prompt Injection + SSRF Sink

1. 外部网页标记不可信，不提升为高优先级指令，不自动写 Memory（1）。
2. 策略禁止 CRM 敏感数据→公开 egress 的组合，并在资源服务重新授权（1）。
3. URL executor 对 scheme/host/DNS/IP/port/每次 redirect 重新校验，阻断 private/metadata endpoint（1）。
4. 环境使用最小凭证和 egress allowlist，不让同一执行面同时拥有广泛 CRM 读取与公网外发（1）。
5. Outcome Eval 检查真实未外泄/未访问内网，Trajectory Eval 检查禁止数据流和阻断点，不只看拒绝文本（1）。

### 23. 审批 TOCTOU

1. 审批绑定 actor/tenant/tool/action 和精确 arguments/target（1）。
2. 包含 amount/currency、expected diff/effect 和外发数据（1）。
3. 绑定 resource version、proposal hash、expiry 和 idempotency key（1）。
4. 任一关键字段/版本/期限变化，`WAITING_APPROVAL` 中的许可失效并需重新审批（1）。
5. 执行时重新认证、服务端授权和前置条件检查，不把审批当授权（1）。

### 24. 级联重试

1. `max_attempts=4` 三层是 `4³=64`（1）。
2. 各“重试 4 次”表示在首次之外再试 4 次，因此每层共 5 attempts，即 `5³=125`（1）。
3. 统一参数语义，集中重试责任，避免各层独立放大（1）。
4. 有限指数退避+jitter，传播剩余 deadline 和全局 retry budget（1）。
5. 每次 attempt 用独立 attempt ID，但 command 保持同一业务幂等键，unknown effect 先查询（1）。

### 25. ACL 检索与 Memory 删除

1. 在候选生成/vector search 前就用 tenant/ACL/purpose 限定 corpus/index partition（1）。
2. Context packing/tool response 前再做防御性权限/来源/新鲜度检查（1）。
3. 分别评测 authorized recall、precision、跨租户泄漏与“高相似无权文档占满 top-k”的 recall starvation（1）。
4. 删除传播至 Thread/Memory/source chunk/vector index/cache/Trace/Eval/backup；不能立即物理删除的层使用 tombstone+访问阻断+保留期（1）。
5. 用 canary ID 做正/反查询、cache/index 失效检查和删除有效性指标，直到所有可服务副本不可见（1）。

### 26. 流式断连与背压

1. 按 event/item/call ID 和 sequence/cursor 解析，reducer 对重复 event 幂等，对 gap/不可恢复乱序失败（1）。
2. 部分 arguments 标记 incomplete，只有显式 completed item 才进入 Schema/语义校验和执行（1）。
3. 重连使用 cursor/replay 协议或从权威 snapshot 恢复，不把 TCP chunk 直接追加成业务记录（1）。
4. 队列满时通过 bounded queue/semaphore 等待至 deadline、拒绝或降级，不无限缓冲（1）。
5. `AbortSignal` 向流/队列/工具传播，UI 显示“重连/等待容量/取消中”；若已有在途 command 则 reconcile（1）。

## C · 系统设计评分点

每题 10 个独立要点，每点 1 分。

### 27. 有界 Agent Loop

1. 模型只产生 candidate/proposal，Runtime/Executor 执行。
2. 状态覆盖 `CREATED/MODEL_RUNNING/VALIDATING/EXECUTING/OBSERVING`。
3. 覆盖 `WAITING_INPUT/WAITING_APPROVAL`，区分 clarification 与 approval。
4. 覆盖 `CANCEL_REQUESTED/CANCELLING`并传播。
5. 覆盖 `IN_DOUBT/RECONCILING`，未知效果不盲试。
6. 终态区分 completed/refused/denied/incomplete/failed/cancelled/budget/partial/manual intervention。
7. 有至少 10 行 `state×event→guard/effect/next` 表，含非法转移拒绝。
8. 稳定 call/idempotency/attempt ID，审批绑定不变提案。
9. step/Token/time/money/fan-out/in-flight 预算在创建新工作前强制；耗尽时先停新工作，有 in-flight/unknown effect 则保持收敛状态，不直接进预算终态。
10. 列出至少八条可检测不变量，含终态不行动、授权、幂等、ACL 和 UI↔Runtime 一致。

### 28. Eval 与实验方案

1. M0 写清用户/任务/输入/期望 outcome/禁止动作/预算。
2. 有非 Agent baseline 与当前 Agent baseline。
3. 30–50 case 覆盖正常、模糊、边界、失败、拒绝、攻击和 protected slices。
4. 确定性真实 outcome grader 优先，不用最终文本代替业务状态。
5. Trajectory grader 检查工具、权限、数据流、预算和禁止序列。
6. 主观质量先建人工金标、评审手册与一致性，再校准 LLM Judge。
7. 按 task 做多 trial，保留配对/群集结构和独立环境。
8. 数据分 development/regression/holdout，production/shadow samples 另行治理；固定 model/prompt/tool/policy/runtime/dataset 版本。
9. 报告 Wilson/exact 区间或 task-level paired bootstrap，事先定义 MDE/power，控制多重比较和顺序停止。
10. 发布门禁同时覆盖总体/protected slice、安全零容忍项、p95/p99、cost per successful task 和 rollback。

### 29. TypeScript + Node Runtime

1. 为 State/Event/Item/ToolResult/Error 定义 discriminated unions 并穷尽 `never` 检查。
2. Thread/Run/Item/Event 与 snapshot 分离，有稳定 ID、版本和原子转移。
3. 将 SDK 流适配为内部语义 event，reducer 支持重复/断流/不完整 item。
4. 手写 Agent Loop 与状态表，模型不直接执行。
5. 用一条 `AbortSignal` 树传播至模型、队列、工具和子任务，并处理在途副作用。
6. 用 semaphore/有界队列/per-tenant quota 做背压，明确满队列策略。
7. 传播剩余 deadline，集中 retry 责任与全局 retry budget。
8. 工具 adapter 实现 Schema/语义/授权/sink/幂等/receipt 边界。
9. 有 trace/audit/eval hooks，且敏感正文不默认进日志。
10. CPU/阻塞工作用 Worker Threads/专用进程或外部服务隔离；有 reducer、状态机、故障注入和有界并发测试。

## 评分后的动作

把失分项写成个人“错误模型清单”，记录错误结论、反例、正确不变量和对应章节。即使总分达标，域内或硬性门禁未过也必须重考。

不要从第一页开始泛读。使用 [章节验收矩阵](/masterpiece-static-docs/10-毕业门禁/06-章节验收矩阵.md) 把每个失分点映射到最小复习范围和必留产出；需要重新安排节奏时，再回到 [八周启动计划](/masterpiece-static-docs/10-毕业门禁/04-八周理论学习计划.md)。
