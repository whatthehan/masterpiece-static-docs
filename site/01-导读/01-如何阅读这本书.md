# 01 · 如何阅读这本书

本书面向具有前端或 TypeScript 工程经验、能够熟练使用 Claude Code、Codex 等 Coding Agent，但尚未系统构建过 Agent 应用的工程师。

熟练使用 Coding Agent 与独立开发 Agent 应用之间仍有一段明显距离。前者主要是在既有 Harness 中描述任务、审查修改并处理异常；后者还需要设计模型接口、Context、工具契约、运行状态、权限、评测、故障恢复和产品交互。模型只是其中一个组件。

全书围绕同一个 Agent Workbench 逐步展开。每引入一个概念，都会回答三个问题：它解决什么工程问题，应该落在哪一层，以及怎样证明实现确实有效。

## 1. 从一个明确的工程任务开始

考虑下面这项常见任务：

> 代码库中有一个测试用例当前未通过。定位导致断言失败的实现缺陷，在不改变公开 API 的前提下修复它，并重新运行相关测试确认结果。

Claude Code 或 Codex 通常不会一次性生成最终补丁，而是经历一系列步骤：读取仓库规则、搜索相关代码、查看测试输出、修改文件、再次执行测试，必要时继续定位。表面上是一段连贯对话，实际涉及多个工程层。

| 可观察行为                            | 背后的工程问题                                    |
| -------------------------------- | ------------------------------------------ |
| 读取 `AGENTS.md`、`CLAUDE.md` 或项目文档 | 哪些指令应该进入本次 Context，优先级如何确定                 |
| 只打开少量相关文件                        | 如何检索并控制 Context，而不是把整个仓库塞进窗口               |
| 调用搜索、编辑和 Shell                   | 工具怎样描述、校验、执行并返回结构化结果                       |
| 根据测试输出继续修改                       | Agent Loop 如何保存状态、消费观察并决定下一步               |
| 危险操作前请求确认                        | Permission、Authorization 与 Approval 分别负责什么 |
| 上下文过长后进行压缩                       | 哪些信息可以摘要，哪些状态必须保存在模型之外                     |
| 展示 Diff 与测试结果                    | 如何用外部证据确认任务完成，而不是相信模型声明                    |

这组映射是全书的阅读入口。后续章节会把这些熟悉的产品行为还原为可实现、可测试的系统对象。

## 2. Coding Agent 只是入口，不是全书边界

Coding Agent 的优势在于工程反馈清晰：文件修改可以查看 Diff，代码可以编译，测试可以重复运行，Git 还能保留历史。因此它适合用来解释 Agent 的基本结构。

通用 Agent 应用往往没有这么理想的反馈条件。发送邮件、修改工单、发起退款或更新客户资料都可能立即影响真实用户；一次错误动作也未必能简单撤销。为了覆盖这些问题，本书采用一个“退款资格与执行助手”作为贯穿案例：

```text
读取当前操作人有权访问的订单和退款政策
→ 判断是否满足退款条件并列出证据
→ 生成退款金额与操作预览
→ 经过服务端授权和具体审批
→ 使用幂等键提交退款
→ 从支付系统核对最终状态
```

这个案例同时包含检索、判断、工具调用、权限、审批、持久状态和故障恢复。读者也可以将它替换为熟悉的领域任务，但应保留同样清晰的输入、边界与完成证据。

## 3. 全书的叙事顺序

正文按照系统依赖展开，而不是按照框架名称排列：

```mermaid
flowchart LR
    P["定义任务与成功标准"] --> M["理解模型与 API"]
    M --> R["实现有界 Agent Runtime"]
    R --> C["管理 Context、状态与事件"]
    C --> K["接入知识与受控动作"]
    K --> D["实现持久执行与故障恢复"]
    D --> O["建立评测、安全与生产运营"]
    O --> S["选择场景专项"]
```

这个顺序体现了真实依赖：任务尚未定义时，模型输出无法比较；模型接口尚未看清时，Agent Loop 只能依赖框架猜测；单次运行的状态和错误语义尚不稳定时，持久工作流只会保存更多混乱。后文会在需要时为这些能力引入阶段名称，但理解正文不依赖预先记住任何编号。

## 4. 每章采用同一种学习结构

正文尽量遵循以下结构：

1. **问题场景**：先说明缺少这一层时会出现什么具体故障。
2. **核心机制**：解释必要原理，不用术语替代推理过程。
3. **工程实现**：把机制映射到 TypeScript、服务边界或数据模型。
4. **边界与误区**：说明这一层不能保证什么，以及常见的错误归因。
5. **实验或验收**：用可重复证据检验理解和实现。

阅读时不必背诵公式或框架 API。更有效的方法是保留一个持续演进的项目，并在每章结束后回答：本章让系统多了哪项能力，又引入了哪种故障模式？

## 5. 一套贯穿全书的项目目录

Agent Workbench 可以从很小的代码库开始：

```text
agent-workbench/
├─ apps/
│  ├─ web/                 # 任务、审批、时间线与结果界面
│  └─ api/                 # Agent Application Server
├─ packages/
│  ├─ model/               # Provider adapter 与流式事件
│  ├─ runtime/             # Agent Loop、预算、取消和状态机
│  ├─ tools/               # 工具契约与执行器
│  ├─ evals/               # Task、Dataset、Grader 与报告
│  └─ contracts/           # 共享 Schema 与事件类型
└─ fixtures/               # 可重复的任务环境和故障样本
```

目录同样按能力逐步形成：先加入 Task Contract、固定案例和 Grader；直接调用模型 API 时再加入 Provider Adapter；开始实现反馈循环时再加入 Runtime 与 Tools。没有实际职责的目录不需要提前创建。

## 6. 三类掌握证据

每个重要概念至少应达到以下一个层级：

| 层级  | 判断标准              | 示例                             |
| --- | ----------------- | ------------------------------ |
| 能解释 | 能说明机制、边界与典型故障     | 解释 Context Window 与应用状态为何不是一回事 |
| 能验证 | 能设计实验区分两种可能原因     | 比较检索缺失与模型生成错误                  |
| 能约束 | 能把结论落实为代码、策略或发布门禁 | 对高风险工具增加服务端授权与幂等校验             |

模型原理至少达到“能解释”，Runtime 和评测应达到“能验证”，权限、安全与副作用必须逐步达到“能约束”。

## 7. 学习记录模板

```md
# 章节学习记录

## 问题
缺少这一机制时，系统会出现什么故障？

## 机制
这一层如何解决问题？关键数据和控制流是什么？

## 边界
它明确不保证什么？

## 实验
怎样用一个最小案例证实或证伪当前理解？

## Workbench 变更
新增了哪些类型、状态、策略、Trace 字段或 Eval case？
```

## 章末练习

选取一条近期 Claude Code 或 Codex 的完整任务记录，整理出以下六类信息：

1. 初始目标和约束。
2. 每次模型调用前可见的 Context。
3. 工具调用及其返回结果。
4. 运行状态和停止条件。
5. Permission、Approval 或 Sandbox 的介入位置。
6. 最终完成证据。

若某一步只能从模型的自然语言声明中判断，而没有外部证据，应将其标记为后续需要补齐的验证点。

## 本章小结

从前端工程师转向 Agent 应用工程师，核心变化不是多学几个模型 API，而是开始对模型周围的运行环境和真实结果负责。下一章会把一条 Coding Agent 任务拆成 Model、Context、Runtime、Harness 与 Application 五个层次，建立后续章节共用的系统地图。
