# 03 · 术语与边界

Agent 领域的许多争论并不是实现差异，而是同一个词指向了不同层。例如，“状态”可能指 API 保存的消息，也可能指业务数据库中的订单；“工具调用成功”可能只表示 HTTP 返回 200，并不表示真实业务目标已经达成。

本章是一份随查词典。首次阅读只需理解核心对象和几组关键边界，遇到具体概念时再返回查阅。

## 1. 核心对象

| 术语                       | 本书中的含义                                                                 | 明确不等于                  |
| ------------------------ | ---------------------------------------------------------------------- | ---------------------- |
| Model                    | 根据当前 Context 生成文本、结构化 Item 或候选动作的概率模型                                  | Agent Runtime、数据库、权限系统 |
| Agent                    | Model 在受约束的反馈循环中，根据 Observation 动态选择下一步动作的系统                           | 不受控制的“自主 AI”           |
| Workflow                 | 主要步骤、分支和等待条件由代码预先规定的流程                                                 | 所有步骤都由 Model 决定        |
| Agent Runtime            | 管理一次 Run 内模型调用、工具执行、状态、预算、取消和错误的程序                                     | 一组 Prompt              |
| Agent Harness            | 将 Model、Context Builder、Runtime、Tools、Sandbox、Policy 和 Trace 组合起来的运行环境 | 具体业务应用本身               |
| Agent Application Server | 连接客户端、Harness 与领域系统，管理 Thread、Run、Item、审批和事件的服务层                       | 一条模型 API 请求            |
| Context                  | 某一次模型调用实际可见的 Token 集合                                                  | 系统保存的全部数据              |
| State                    | 应用判断当前任务位置和下一步所需的权威数据                                                  | 一段聊天摘要                 |
| Tool                     | 具有输入、输出、权限、错误和副作用语义的可调用能力                                              | 随意暴露给模型的函数             |
| Observation              | Tool 或环境返回给 Runtime 的结果                                                | 自动可信的事实                |
| Outcome                  | 任务结束后，权威外部系统中的实际结果                                                     | Agent 的完成声明            |
| Trace                    | 串联模型、工具、策略、状态变化和 Outcome 的结构化记录                                        | 原始 Chain-of-Thought    |
| Eval                     | 使用 Task、环境、Trial 和 Grader 测量系统表现的过程                                    | 人工随意体验几次               |

## 2. Context、Knowledge、Memory 与 State

这四个概念最容易混在同一段消息历史里。

| 概念        | 典型内容                  | 推荐存储                                       | 何时进入模型                  |
| --------- | --------------------- | ------------------------------------------ | ----------------------- |
| Context   | 当前指令、相关证据、工具说明、状态摘要   | 请求快照或可重放记录                                 | 本次调用已经可见                |
| Knowledge | 政策、文档、产品资料等外部事实       | 带来源、权限和版本的知识系统                             | 经检索与权限过滤后               |
| Memory    | 用户偏好、历史经验、可复用信息       | 有写入门禁、Time to Live（TTL）和删除机制的 Memory Store | 仅在当前任务相关时               |
| State     | 当前步骤、审批状态、重试次数、领域对象版本 | Workflow Store 或领域数据库                      | 由 Context Builder 选择性投影 |

一个订单的实际退款状态属于领域 State；退款政策属于 Knowledge；用户偏好“优先邮件联系”可以是 Memory；本次调用中选入的订单摘要、政策片段和偏好才构成 Context。

Context 是临时视图，不是权威数据源。Compaction 生成的摘要同样只是派生数据，不能覆盖原始事件或领域事实。

## 3. Tool、Resource、Skill、Hook 与 MCP

这些对象解决不同问题，不构成从低级到高级的升级阶梯。

| 概念                          | 解决的问题                                              | 示例                               |
| --------------------------- | -------------------------------------------------- | -------------------------------- |
| Tool / Function             | 执行一个原子查询或动作                                        | `getOrder`、`createRefundPreview` |
| Resource                    | 向调用方提供可读取数据                                        | 一份政策文档、一个 Schema                 |
| Agent Skill                 | 按开放目录格式、按需加载一组程序性知识、脚本与资源                          | “如何审查退款争议”的操作包                   |
| Dynamic Tool Discovery      | 从能力目录筛选本轮候选，再按权限加载完整 Tool Schema                   | 从大 Catalog 中加载当前工单需要的只读查询        |
| Hook / Middleware           | 在生命周期节点运行确定性逻辑                                     | Tool 执行前做参数审计                    |
| Model Context Protocol（MCP） | 在 Client 与 Server 之间发现和调用 Tool、Resource、Prompt 的协议 | 接入独立的知识或业务服务                     |
| Subagent                    | 用独立 Context、权限或模型完成一个子任务                           | 独立审查证据完整性                        |

Agent Skill 不提供执行授权，Dynamic Tool Discovery 的候选结果也不代表允许调用；Hook 是否能阻断动作取决于它运行的位置，MCP Server 也不会因为遵循标准协议就自动可信。

### 协议分层速查

协议名称相似，不代表处于同一层：

| 名称                                     | 所在边界                                    | 主要交换对象                                    | 明确不提供                              |
| -------------------------------------- | --------------------------------------- | ----------------------------------------- | ---------------------------------- |
| Server-Sent Events（SSE）                | HTTP Server → Client Transport          | 按帧发送的文本 Event                             | Agent、Tool、领域状态或自动重放语义             |
| OpenAI Responses Streaming             | Model Provider → Provider Adapter       | 基于 SSE 的类型化 Response / Item / Delta Event | 应用 Run 终态与 UI 持久恢复                 |
| Agent–User Interaction Protocol（AG-UI） | Agent Backend ↔ User-facing Application | Run、Message、Tool、State Event              | Agent Runtime、领域 Event Store 与业务授权 |
| Agent-to-UI Protocol（A2UI）             | Agent / Server → Trusted Renderer       | Surface、Component、Data Model 与 Action 描述  | 任意 JSX 执行、Transport 与业务授权          |
| Model Context Protocol（MCP）            | Host / Client ↔ Capability Server       | Tool、Resource、Prompt                      | Multi-Agent 协作与业务 Policy           |
| MCP Apps Extension                     | MCP Server → Host 内嵌 View               | `ui://` Resource、沙箱界面与 Host Bridge        | 可信 Approval 与完整应用 UI               |
| MCP Tasks Extension                    | MCP Client ↔ Server 长时操作                | Durable Handle、状态、轮询、输入与取消                | Application Workflow、补偿与业务 Outcome |
| Agent2Agent Protocol（A2A）              | 独立 Agent System ↔ Agent System          | Agent Card、Message、Task 与 Artifact        | 对方内部 Tool、Prompt、Memory 或天然信任      |

一条生产链路可以同时使用多层协议：OpenAI Streaming 进入 Provider Adapter，应用产生 Canonical RunEvent，再由 AG-UI 投影给前端；A2UI Payload 可以经 AG-UI 或 A2A 传输；Remote Agent 内部又可以使用 MCP 调用 Tool。每一层都需要独立的版本、身份、错误和重放策略。

对应章节：[模型 API 与流式事件](/masterpiece-static-docs/05-模型接口与Agent内核/03-模型API-状态与流式事件.md)、[AG-UI 前端事件适配](/masterpiece-static-docs/05-模型接口与Agent内核/10-AG-UI与前端事件适配.md)、[MCP](/masterpiece-static-docs/07-工具-协议与行动控制/03-MCP与互操作协议.md)、[Skills 与 MCP 扩展](/masterpiece-static-docs/07-工具-协议与行动控制/06-Agent-Skills与MCP扩展.md)、[A2A](/masterpiece-static-docs/07-工具-协议与行动控制/05-A2A与跨Agent协作协议.md)、[A2UI](/masterpiece-static-docs/08-安全与治理/06-A2UI与声明式生成界面.md)。

## 4. Task、Trial、Grader 与 Dataset

| 术语                     | 定义                                               |
| ---------------------- | ------------------------------------------------ |
| Task                   | 一项带输入、初始环境和成功标准的可重复任务                            |
| Trial                  | 系统对某个 Task 的一次独立尝试                               |
| Grader                 | 检查某个质量维度的规则、程序、模型或人工流程                           |
| Dataset                | 版本化的一组 Tasks 及其环境、标签和元数据                         |
| Suite                  | 围绕某项能力或风险组织的一组 Tasks                             |
| Evaluation Harness     | 准备环境、运行 Trial、收集 Trace、执行 Grader 并聚合报告的设施        |
| Evaluation Environment | 接收 Action、推进状态并提供 Observation 与权威 Outcome 的可交互世界 |
| Simulator              | 按版本化契约、Seed 与 Fault Schedule 可重复实现环境状态转移的测试设施    |
| Synthetic Case         | 由约束、变异或模型生成并经验证的补充任务；不自动代表真实分布                   |
| Human Review           | 按 Rubric 进行人工标注、保留分歧并执行裁决的评测过程                   |

同一个 Task 可以运行多次 Trial，用于观察模型输出的波动。一个 Task 也可以有多个 Grader，分别检查任务完成、安全、证据忠实度和成本，而不必压成一个含义模糊的总分。

## 5. Thread、Run、Item 与 Event

- **Thread**：承载一项长期任务或会话的容器。
- **Run**：一次可开始、暂停、恢复、取消并最终进入终态的执行。
- **Item**：消息、Tool Call、Tool Result、审批、证据、计划或错误等持久对象。
- **Event**：对象状态发生变化的事实，例如 `run.started`、`tool.requested`、`approval.required`。

Item 表示“存在一个什么对象”，Event 表示“这个对象发生了什么变化”。客户端可以从 Snapshot 恢复当前状态，再应用后续 Event；审计系统则可能保留完整事件序列。

在 Multi-Agent 系统中还要区分：

- **Parent Run**：持有总目标、总预算、取消与最终结果责任；
- **Child Run**：持有独立 Context、子预算、Attempt 和生命周期；
- **Artifact**：Child 发布的不可变、带来源和版本的结果；
- **Handoff**：active owner 转移；它不同于 Parent 保留责任的 Delegation / Agent-as-Tool。

远端 A2A Task 完成只代表协议任务终止，本地 Parent 仍需验收 Artifact 并决定业务状态。

## 6. Authentication、Authorization、Permission 与 Approval

```text
Authentication：调用者是谁？
Permission：当前运行环境允许使用哪些能力？
Authorization：该 actor 能否对该 resource 执行该 action？
Approval：用户是否确认这一笔具体动作？
```

以退款为例：

- 已登录客服账号解决 Authentication。
- Harness 允许调用支付 Tool 解决 Permission。
- 服务端判断该客服是否能操作订单 123 解决 Authorization。
- 用户确认“向原支付方式退 100 元”解决 Approval。

即使四项都满足，执行时仍要检查订单版本、幂等键和审批有效期，避免审批后参数被替换或资源状态已经变化。

## 7. 输出、动作提议与真实效果

```text
模型文本：退款已经完成。
动作提议：refund({ orderId: "123", amount: 100 })
Tool Receipt：支付接口已接收请求，requestId = "r_456"
真实 Outcome：支付系统最终只产生一笔合法、金额匹配的退款。
```

四者分别代表自然语言声明、候选调用、执行回执和业务结果。网络中断时，Tool 可能已经提交成功却没有返回 acknowledgement（ACK，确认响应）；此时简单重试可能造成重复退款。评测和恢复逻辑必须查询权威 Outcome，而不是只看文本或单次 HTTP 响应。

## 8. 能力、可靠性与安全

- **能力（Capability）**：理想条件下是否能完成任务。
- **可靠性（Reliability）**：面对随机性、部分故障和重试时，能否稳定完成或安全停止。
- **安全（Safety / Security）**：面对误用、攻击或恶意输入时，是否仍保持关键不变量。

能力更强并不必然更可靠或更安全。能够操作更多工具的模型，也拥有更大的潜在影响范围。因此三类指标必须分别评测。

## 9. 关于推理记录

产品可以记录和展示：

- 结构化计划与当前步骤。
- 检索到的证据及其来源。
- Tool Call、参数摘要与执行结果。
- Policy 决策、Approval 和状态变化。
- 简洁的 reasoning summary。

不应把隐藏的原始 Chain-of-Thought 作为审计依据。自然语言推理痕迹未必是完整忠实的因果解释，还可能包含敏感数据。可审计系统需要结构化事件、版本和外部证据。

## 10. 阅读时的概念依赖

后续章节会按下列依赖逐步展开，无需额外记忆一套路线代号：

1. 先定义 Task 和 Outcome，再判断是否需要 Agent。
2. 先观察原始 Model API 和 Stream，再学习 Framework 封装。
3. 先实现有界 Agent Runtime，再扩展 Context、Knowledge 和 Memory。
4. 先建立 Authorization、Approval 和 Idempotency，再接入真实写操作。
5. 先证明单 Agent 方案的边界，再考虑 Multi-Agent、Computer Use 或 Rust 迁移。

## 章末检查

1. 为什么订单退款状态属于 State，而不是 Memory？
2. Tool 返回成功与业务 Outcome 成功之间可能出现哪些差异？
3. Permission、Authorization 与 Approval 分别应在哪一层实施？

## 本章小结

Agent 应用的首要工程工作是建立边界：Context 不等于全部状态，Tool Call 不等于真实效果，Approval 不等于 Authorization，Trace 也不等于模型的思维过程。下一章开始为贯穿项目定义第一份 Task Contract 和可重复的评测基线。
