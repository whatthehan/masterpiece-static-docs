#03 · 术语与边界

Agent 领域的许多争论并非源于实现差异，而是同一个词被用于指代不同层次。例如，“状态”可能指 Provider 保存的消息，也可能指业务数据库中的订单；“工具调用成功”可能只表示 HTTP 返回 200，并不表示真实业务目标已经达成。

本章是一份随查词典。首次阅读先掌握 LLM、Agent System、Agentic UI 和工程证据四组核心对象，遇到具体概念时再返回查阅。

#1. 三层核心对象

#1.1 LLM 层

术语本书中的含义明确不等于
Model根据当前 Context 生成文本、结构化 Item 或候选动作的概率模型Agent Runtime、数据库、权限系统
Model Request发送给 Provider 的指令、消息、工具说明、输出约束与推理配置完整业务 Task
Model ItemMessage、Reasoning Summary、Tool Call 等模型接口中的类型化对象应用持久化的领域 Item
DeltaStreaming 中尚未闭合的增量片段可直接执行的完整 Tool Call
Structured Output受 Schema 约束的模型输出通过业务语义与权限校验的数据

#1.2 Agent 系统层

术语本书中的含义明确不等于
AgentModel 在受约束的反馈循环中，根据 Observation 动态选择下一步动作的系统不受控制的“自主 AI”
Workflow主要步骤、分支和等待条件由代码预先规定的流程所有步骤都由 Model 决定
Agent Runtime管理一次 Run 内模型调用、工具执行、状态、预算、取消和错误的程序一组 Prompt
Agent Harness将 Model、Context Builder、Runtime、Tools、Sandbox、Policy 和 Trace 组合起来的运行环境具体业务应用本身
Agent Application Server连接客户端、Harness 与领域系统，管理身份、Thread、Run、Item、审批和事件的服务层一条模型 API 请求
Context某一次模型调用实际可见的 Token 集合系统保存的全部数据
State应用判断当前任务位置和下一步所需的权威数据一段聊天摘要
Tool具有输入、输出、权限、错误和副作用语义的可调用能力随意暴露给模型的函数
ObservationTool 或环境返回给 Runtime 的结果自动可信的事实
Outcome任务结束后，权威外部系统中的实际结果Agent 的完成声明

#1.3 Agentic UI 层

术语本书中的含义明确不等于
Agentic UI展示 Agent 状态、证据与不确定性，并支持 Interrupt、Approval、Recovery 和 Handoff 的人机协作界面只有消息气泡的聊天窗口
Canonical Event应用内部已持久化、语义稳定的状态变化事实Provider 的原始 Token Delta
Public Snapshot面向客户端的权威公开状态投影，用于首次加载和恢复服务端完整 State 或私有 Context
UI Reducer由 Snapshot 和后续 Event 确定性计算 UI State 的函数业务领域状态机
Intent / Action用户在界面中表达的意图，由服务端重新校验和授权客户端直接执行的业务 Command
Surface一块可独立更新的声明式界面区域任意远程 HTML 或 JavaScript
Trusted Renderer只使用本地受信组件 Catalog 渲染声明式 UI 的宿主执行模型生成前端代码的沙箱替代品
Action Gateway接收 UI Action 并执行认证、Schema、Policy 与 Authorization 校验的服务端入口对 Tool 的无条件转发

#1.4 横切工程证据

术语本书中的含义明确不等于
Trace串联模型、工具、策略、状态变化、UI 投影和 Outcome 的结构化记录原始 Chain-of-Thought
Eval使用 Task、环境、Trial 和 Grader 测量系统表现的过程人工随意体验几次
Audit针对身份、权限、审批、动作和结果保留的可审查事实调试日志的别名
SLO用于约束用户可感知质量与可靠性的服务目标单次 Benchmark 分数
Behavior Bundle决定一版线上行为的 Model Route、Prompt、Context Builder、Knowledge、Tool、Policy、Workflow、Runtime 与公开协议版本集合只有模型名称或容器镜像
Controlled Agent Improvement（受控 Agent 改进）将生产信号整理为改进候选，并经独立 Eval、审批和发布门禁更新后续版本在线 Agent 自行修改当前 Prompt、Tool、Policy 或代码

#2. Context、Knowledge、Memory 与 State

这四个概念最容易混在同一段消息历史里。

概念典型内容推荐存储何时进入模型
Context当前指令、相关证据、工具说明、状态摘要请求快照或可重放记录本次调用已经可见
Knowledge政策、文档、产品资料等外部事实带来源、权限和版本的知识系统经检索与权限过滤后
Memory用户偏好、历史经验、可复用信息有写入约束、存活时间（Time to Live，TTL）和删除机制的 Memory Store仅在与当前任务相关时
State当前步骤、审批状态、重试次数、领域对象版本Workflow Store 或领域数据库由 Context Builder 选择性投影

订单的实际退款状态属于领域 State；退款政策属于 Knowledge；用户偏好“优先邮件联系”可以是 Memory；本次调用中选入的订单摘要、政策片段和偏好才构成 Context。

Context 是临时视图，不是权威数据源。Compaction 生成的摘要同样只是派生数据，不能覆盖原始事件或领域事实。Public Snapshot 则是面向 UI 的另一种投影，也不能反向覆盖服务端 State。

#2.1 RAG、Agentic RAG 与 GraphRAG

概念本书中的含义采用边界
Retrieval-Augmented Generation（RAG）从版本化 Source 经 Ingestion、检索、重排、Context Packing 到 Claim–Evidence 校验的完整管线先建立固定 Hybrid + Rerank Baseline
Agentic RAG模型在有状态、有预算、有停止条件的 Retrieval Loop 中提出查询、补证和完成候选只解决固定 Pipeline 已被 Eval 证明无法覆盖的多步检索
GraphRAG从语料派生 Entity、Relationship、Community 与 Report，再用图结构辅助局部或全局检索派生图不是权威事实；ACL、时间、更新、删除、引用与成本必须重新验证
Index Build Manifest固定 Source Snapshot、Parser、Chunker、Embedding、Schema、输入输出摘要与发布状态的索引构建清单不能用“向量库当前内容”代替可回滚版本
Memory Consolidation从多条有来源的 Episode 与 Outcome 提炼 Semantic / Procedural Memory Candidate只能生成待验证候选，不等于 Context Compaction，也不能创造事实

#3. Tool、Resource、Skill、Hook 与 MCP

这些对象解决不同问题，不构成从低级到高级的升级阶梯。

概念解决的问题示例
Tool / Function执行一个原子查询或动作getOrder、createRefundPreview
Resource向调用方提供可读取数据一份政策文档、一个 Schema
Agent Skill按开放目录格式、按需加载一组程序性知识、脚本与资源“如何审查退款争议”的操作包
Dynamic Tool Discovery从能力目录筛选本轮候选，再按权限加载完整 Tool Schema从大 Catalog 中加载当前工单需要的只读查询
Hook / Middleware在生命周期节点运行确定性逻辑Tool 执行前做参数审计
Model Context Protocol（MCP）在 Client 与 Server 之间发现和调用 Tool、Resource、Prompt 的协议接入独立的知识或业务服务
Subagent用独立 Context、权限或模型完成一个子任务独立审查证据完整性

Agent Skill 不提供执行授权，Dynamic Tool Discovery 的候选结果也不代表允许调用；Hook 是否能阻断动作取决于它运行的位置，MCP Server 也不会因为遵循标准协议就自动可信。

#4. AG-UI、A2UI 与相关协议

协议名称相似，不代表处于同一层。

名称所在边界主要交换对象明确不提供
Server-Sent Events（SSE）HTTP Server → Client Transport按帧发送的文本 EventAgent、Tool、领域状态或自动重放语义
OpenAI Responses StreamingModel Provider → Provider Adapter基于 SSE 的类型化 Response、Item 与 Delta Event应用 Run 终态与 UI 持久恢复
Agent–User Interaction Protocol（AG-UI）Agent Backend ↔ User-facing ApplicationRun、Message、Tool、State EventAgent Runtime、领域 Event Store 与业务授权
Agent to UI Protocol（A2UI）Agent / Server → Trusted RendererSurface、Component、Data Model 与 Action 描述任意 JSX 执行、Transport 与业务授权
Model Context Protocol（MCP）Host / Client ↔ Capability ServerTool、Resource、PromptMulti-Agent 协作与业务 Policy
MCP Apps ExtensionMCP Server → Host 内嵌 Viewui:// Resource、沙箱界面与 Host Bridge可信 Approval 与完整应用 UI
MCP Tasks ExtensionMCP Client ↔ Server 长时操作Durable Handle、状态、轮询、输入与取消Application Workflow、补偿与业务 Outcome
Agent2Agent Protocol（A2A）独立 Agent System ↔ Agent SystemAgent Card、Message、Task 与 Artifact对方内部 Tool、Prompt、Memory 或天然信任

一条生产链路可以同时使用多层协议：

Provider Streaming Event
→ Model Adapter
→ Canonical Event / Public Snapshot
→ AG-UI Projection
→ UI Reducer

A2UI Payload 可以经 AG-UI、A2A 或其他 Transport 传输；Remote Agent 内部也可以使用 MCP 调用 Tool。每一层都需要独立处理版本、身份、错误和重放策略。

AG-UI 与 A2UI 是 Agentic UI 主线的核心协议，但仍位于应用边界：前者不替代领域 Event Store，后者不替代固定的高风险 Approval UI。对应章节：Application Server 与 UI 事件协议、AG-UI、Agent UX与A2UI。

其他协议见：模型 API 与流式事件、MCP、Skills 与 MCP 扩展与A2A。

#5. Task、Trial、Grader 与 Dataset

术语定义
Task一项带输入、初始环境和成功标准的可重复任务
Trial系统对某个 Task 的一次独立尝试
Grader检查某个质量维度的规则、程序、模型或人工流程
Dataset版本化的一组 Tasks 及其环境、标签和元数据
Suite围绕某项能力或风险组织的一组 Tasks
Evaluation Harness准备环境、运行 Trial、收集 Trace、执行 Grader 并聚合报告的设施
Evaluation Environment接收 Action、推进状态并提供 Observation 与权威 Outcome 的可交互世界
Simulator按版本化契约、Seed 与 Fault Schedule 可重复实现环境状态转移的测试设施
Synthetic Case由约束、变异或模型生成并经验证的补充任务；不自动代表真实分布
Human Review按 Rubric 进行人工标注、保留分歧并执行裁决的评测过程

同一个 Task 可以运行多次 Trial，用于观察模型输出的波动。一个 Task 也可以有多个 Grader，分别检查任务完成、安全、证据忠实度、UI 状态和成本，而不必压成一个含义模糊的总分。

#6. Thread、Run、Item 与 Event

  • Thread：承载一项长期任务或会话的容器；
  • Run：一次可开始、暂停、恢复、取消并最终进入终态的执行；
  • Item：消息、Tool Call、Tool Result、Approval、Evidence、Plan 或 Error 等持久对象；
  • Event：对象状态发生变化的事实，例如 run.started、tool.requested、approval.required。

Item 表示“存在一个什么对象”，Event 表示“这个对象发生了什么变化”。客户端可以从 Public Snapshot 恢复公开状态，再应用后续 Event；审计系统则保留需要追责的完整事件与策略证据。

Provider Event、Canonical Event 与 UI Event 也必须区分：

Provider Event：模型接口中的原始流事件
Canonical Event：应用已接受并持久化的语义事实
UI Event：针对某类客户端协议投影出的公开事件

应用边界上的对象还要按生命周期和用途区分：

  • Durable / Replayable Canonical Event：已经提交、需要重放的语义事实，例如 Tool Terminal Result、Approval Decision 和 Run Terminal State；
  • Transient / Live-only Signal：只改善当前交互体验的高频 Delta 或 Progress，丢失后可以由 Snapshot 或 Canonical State 恢复；
  • Internal / Persistence-only Record：服务于索引、游标、预算或恢复的不公开记录。

只有第一类称为 Canonical Event；Live-only Signal 和 Internal Record 不能因为也采用事件形态传输或存储，就混入公开事实序列。对象是否持久化由语义决定，不由“它是否经过 SSE”决定。高频 Delta 可以合并，Canonical Terminal Event 必须恰好一次地进入权威序列；二者不能共享一个含义模糊的 message 类型。

#7. Multi-Agent、Delegation、Handoff 与 A2A

  • Parent Run：持有总目标、总预算、取消与最终结果责任；
  • Child Run：持有独立 Context、子预算、Attempt 和生命周期；
  • Artifact：Child 发布的不可变、带来源和版本的结果；
  • Delegation：Parent 保留最终责任，将子任务交给 Worker；
  • Handoff：当前责任方（Active Owner）发生转移；
  • Agent-as-Tool：将另一个 Agent 包装成当前 Runtime 可调用的能力；
  • A2A Task：跨独立 Agent 系统交换的协议任务。

远端 A2A Task 完成只代表协议任务终止，本地 Parent 仍需验收 Artifact 并决定业务状态。Multi-Agent 与 A2A 是 Agent 系统层的必懂能力，但不是默认生产架构；采用前要在相同 Dataset、近似总预算和故障模型下与单 Agent Baseline 比较。

#8. Authentication、Authorization、Permission 与 Approval

Authentication：调用者是谁？
Permission：当前运行环境允许使用哪些能力？
Authorization：该 Actor 能否对该 Resource 执行该 Action？
Approval：用户是否确认这一笔具体动作？

以退款为例：

  • 已登录客服账号解决 Authentication；
  • Harness 允许调用支付 Tool 解决 Permission；
  • 服务端判断该客服是否能操作订单 123 解决 Authorization；
  • 用户确认“向原支付方式退 100 元”解决 Approval。

即使四项都满足，执行时仍要检查订单版本、幂等键和审批有效期，避免审批后参数被替换或资源状态已经变化。A2UI Action 只能表达用户意图，不能绕过上述检查。

Permission 最好表达为带约束的 Capability Set。父 Agent 委派子任务时必须满足 child capability ⊆ parent capability；这里的子集同时覆盖 Action、Tool、Canonical Resource、Path、Network Origin、Executable、额度与期限，不是只比较 read / write / execute 动词。read + write 与 read + execute 可能是不可比较的两组能力，不能用“权限等级 2/3”替代约束证明。未知 Tool、缺失 Effect 声明或无法确定目标 Resource 时，应拒绝或进入人工审查，不能默认当作只读。

#9. 输出、动作提议、回执与真实效果

模型文本：退款已经完成。
动作提议：refund({ orderId: "123", amount: 100 })
Tool Receipt：支付接口已接收请求，requestId = "r_456"
真实 Outcome：支付系统最终只产生一笔合法、金额匹配的退款。

四者分别代表自然语言声明、候选调用、执行回执和业务结果。网络中断时，Tool 可能已经提交成功，却没有返回确认响应（Acknowledgement，ACK）；此时直接重试可能造成重复退款。评测、恢复逻辑和 UI 状态都必须查询权威 Outcome，不能只看文本或单次 HTTP 响应。

#10. 能力、可靠性与安全

  • 能力（Capability）：理想条件下是否能完成任务；
  • 可靠性（Reliability）：面对随机性、部分故障和重试时，能否稳定完成或安全停止；
  • 安全（Safety / Security）：面对误用、攻击或恶意输入时，是否仍保持关键不变量。

能力更强并不必然更可靠或更安全。能够操作更多工具、委派更多 Agent 或生成更多 UI Surface，也意味着更大的协调成本和潜在影响范围。因此三类指标必须分别评测。

#11. 关于推理记录

产品可以记录和展示：

  • 结构化计划与当前步骤；
  • 检索到的证据及其来源；
  • Tool Call、参数摘要与执行结果；
  • Policy 决策、Approval 和状态变化；
  • 简洁的推理摘要（Reasoning Summary）。

不应把隐藏的原始 Chain-of-Thought 作为审计依据。自然语言推理痕迹未必是完整忠实的因果解释，还可能包含敏感数据。可审计系统需要结构化事件、版本和外部证据。

#12. 阅读时的概念依赖

  1. 先定义 Task 和 Outcome，再判断是否需要 Agent；
  2. 先理解 LLM 的生成与 Streaming 语义，再处理 Runtime 封装；
  3. 先实现有界单 Agent Runtime，再扩展 Context、Knowledge 和 Memory；
  4. 先建立 Authorization、Approval 和 Idempotency，再开放写操作；
  5. 先证明单 Agent 方案的边界，再用 Eval 判断是否采用 Multi-Agent 与 A2A；
  6. 先建立 Canonical Event 与 Public Snapshot，再接入 AG-UI；
  7. 先设计可信 Agent UX 与 Action Gateway，再让 A2UI 生成低风险 Surface。
  8. 先具备 Trace、Holdout 和可回滚发布，再让 Agent 或 Optimizer 生成改进候选。

#本章小结

Agent 应用的首要工程工作是建立边界：Context 不等于全部状态，Tool Call 不等于真实效果，Provider Event 不等于 UI 权威状态，A2UI Action 不等于已授权 Command，Trace 也不等于模型的思维过程。下一章开始为贯穿项目定义第一份 Task Contract 和可重复的评测基线。
