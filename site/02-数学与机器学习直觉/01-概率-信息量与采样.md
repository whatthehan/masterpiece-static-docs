# 01 · 概率、信息量与采样

即使订单、退款政策和用户请求完全相同，Resolution Desk 连续运行两次，也可能给出不同的解释顺序、证据组合或候选动作。Claude Code、Codex 等 Coding Agent 处理同一个代码任务时，也可能搜索不同文件、尝试不同修改路径。这并非普通函数中的偶发抖动，而是生成模型工作方式的直接结果：模型每一步都根据条件概率分布选择下一个 Token，完整的文本或结构化动作由这些局部选择逐步形成。

Agent 工程不需要先掌握完整概率论，但必须理解这种随机性如何影响测试、风险和系统设计。

## 贯穿项目：Resolution Desk

本章暂不实现模型接口或 Agent Runtime，而是为 Resolution Desk 建立第一组可重复观察材料：固定订单、政策版本、用户请求和预期边界，再收集多次候选输出。产物是一组录制的试次样例（Recorded Trial Fixture），每条记录带有 `taskId`、`trialId` 和通过标记。后续章节会直接把这组样例纳入数据集（Dataset）与评测（Eval），因此此时不需要已经存在可运行的 Agent。

## 1. 模型输出的是分布

模型在每个生成位置为候选 Token 产生一组分数，通常称为 **logits**。Softmax 将 logits 归一化为总和为 1 的概率分布：

```text
P_i = exp(z_i / T) / Σ exp(z_j / T)
```

- `z_i` 是候选 Token 的 logit。
- `T` 是温度（Temperature）。
- `P_i` 是采样到该 Token 的概率。

为了说明分布形状，可以把若干完整续写简化成三个候选动作。真实模型直接为下一个 Token 评分，一个动作通常由多个 Token 组成，并不存在下表这样的单一“动作 logit”。

| 候选       | Logit | 直观含义          |
| -------- | ----: | ------------- |
| 继续搜索调用方  |   4.2 | 当前最符合模型模式     |
| 直接修改实现   |   3.9 | 也相当合理         |
| 修改公开 API |   0.4 | 概率较低，但不等于绝不发生 |

降低 Temperature 通常会使分布更加集中；提高 Temperature，则会增加低概率候选被选中的机会。Temperature 只改变采样分布，不能修正错误事实，也不能把安全约束变成确定性保证。

API 中的 `temperature: 0` 通常表示贪心解码（Greedy Decoding），或接近贪心解码的服务端策略。这不等于整个托管系统具有数学意义上的完全确定性；模型版本、服务端实现、并行计算和工具环境仍可能造成差异。

## 2. 序列生成具有路径依赖

语言模型按条件概率逐步生成序列：

```text
P(y | x) = Π P(y_t | x, y_<t)
```

工程上通常使用 log probability，把连乘转换为求和：

```text
log P(y | x) = Σ log P(y_t | x, y_<t)
```

关键不在公式本身，而在条件项 `y_<t`：前面已经选择的内容会改变后续分布。

在普通文本生成中，一处措辞变化可能只影响风格；在 Agent Loop 中，一次早期选择会改变外部环境：

```text
选择了错误 Tool
→ 得到不相关 Observation
→ Observation 进入下一轮 Context
→ 后续计划建立在错误前提上
→ 产生更多无效或危险动作
```

因此，长轨迹的风险并不是“每一步错误率简单相加”。动作会反馈到环境，错误可能级联，也可能被后续验证及时纠正。Runtime 需要预算、状态、验证和停止条件来控制这条反馈链。

## 3. 概率最高不等于事实正确

Softmax 只表达模型对候选序列的相对偏好。一个错误陈述如果在训练数据中模式强、在当前 Context 中缺少反证，也可能获得很高概率。

这解释了为什么下面几项不能直接画等号：

```text
低 Entropy ≠ 事实正确
高 log probability ≠ 业务合法
模型自报 0.95 置信度 ≠ 经过校准的成功概率
多数采样结果一致 ≠ 外部系统状态正确
```

信息熵（Entropy）描述概率分布的分散程度：

```text
H(P) = -Σ P_i log P_i
```

高 Entropy 表示概率分散在更多候选上；低 Entropy 表示概率集中在少数候选上。这个量可以帮助观察生成分布，却不能替代检索、程序校验或权威系统查询。

## 4. 工程上需要分开处理的三类不确定性

前两类沿用机器学中常见的划分；“系统不确定性”则是本书为 Agent 应用的外部状态单独引入的工程类别，并非并列的标准机器学分类。

### 偶然不确定性（Aleatoric Uncertainty）

任务本身允许多个合理答案。例如，将一段说明改写得更清晰，不存在唯一措辞。此类不确定性可以通过 Rubric 判断语义质量，不必强迫输出逐字一致。

### 认知不确定性（Epistemic Uncertainty）

模型缺少完成判断所需的知识、Context 或能力。例如，当前生效的退款政策没有进入 Context。正确处理通常是检索、调用 Tool、请求补充信息，或明确表示无法判断（Abstention），而不是要求模型生成一个置信度数字。

### 系统不确定性（System Uncertainty）

模型之外的环境状态无法确认。例如，支付请求超时后，调用方不知道退款是否已经提交。此时需要幂等键、状态查询、对账或人工异常处理；增加模型的推理投入（Reasoning Effort）无法消除未知的外部效果。

三类不确定性对应不同的工程动作。如果全部压缩成一个 `confidence: 0.8` 字段，应用将无法选择正确处理方式。

## 5. Trial：从单次结果转向结果分布

系统针对一项任务进行的一次独立尝试称为一个试次（Trial）。同一个 Task 需要运行多个 Trial，才能观察成功率、稳定性和失败模式。

假设 20 个 Task 各运行 5 次，可以报告：

- 每个 Task 的 pass rate。
- 按正常、权限、故障等 Slice 聚合的成功率。
- 失败类型及其频率。
- 延迟、Token 用量和成本分布。
- 同一 Task 在不同 Trial 间是否表现稳定。

两个常见指标需要区分：

- **pass\@k**：k 次尝试中至少一次成功。适合允许生成多个候选再选择的任务。
- **pass^k**：k 次尝试全部成功。适合关注一致性的任务。

若生产环境只允许执行一次退款，较高的 pass\@5 无法弥补较低的 pass\@1；系统并没有五次试错机会。评估高风险动作时，更应直接衡量违规率（Violation Rate）和真实业务结果（Outcome）。

## 6. 随机性应该留在哪里

一个可靠的 Agent 应用不会试图消灭所有随机性，而是把随机性限制在合适位置：

| 系统部分    | 可以接受的变化          | 必须确定的约束                            |
| ------- | ---------------- | ---------------------------------- |
| 解释与摘要   | 措辞、段落结构          | 证据不得被歪曲                            |
| 任务规划    | 合法步骤的顺序          | 必须遵守预算和禁止动作                        |
| Tool 选择 | 多个只读 Tool 中的合理选择 | 参数必须通过 Schema 与语义校验                |
| 高风险动作   | 可以生成不同候选 Preview | Authorization、Approval、金额上限、幂等必须确定 |
| 完成判断    | 可以给出简洁说明         | 必须由权威 Outcome 或确定性 Grader 确认       |

模型负责在解空间中提出候选，代码和策略负责保持不变量。

## 7. 最小实验

为 Resolution Desk 准备两项固定任务，各收集 10 个 Trial。此处可以使用模型控制台完成独立实验，也可以手工准备一组 Recorded Output；不需要接入 SDK、Tool 或 Agent Loop。

1. 开放任务：订单号缺失时，生成一段简洁的澄清问题。措辞可以不同，但不得猜测订单或承诺已经退款。
2. 封闭任务：给定订单 `ord_1001`、可退余额 `6800` 分和当前政策上限 `5000` 分，从 `eligible / needs_review / ineligible` 中选择唯一结论，并给出候选退款金额。

记录以下字段：

```ts
type TrialRecord = {
  taskId: string;
  trialId: string;
  model: string;
  config: Record<string, unknown>;
  output: unknown;
  semanticResult: string;
  passed: boolean;
  failureType?: string;
  latencyMs: number;
  usage?: { inputTokens: number; outputTokens: number };
};
```

实验需要区分“文本不同但语义等价”和“语义或动作确实不同”。开放任务中的措辞变化通常可以接受；封闭任务中的参数变化则说明，硬约束不能由采样过程保证。将 20 条结果保存为 Resolution Desk 的首批 Trial Fixture；[Grader、Trial 与统计](/masterpiece-static-docs/04-评测与实验科学/01-Grader-Trial与统计.md)会在同一批记录上增加评分器（Grader）、统计区间和数据切片（Slice）分析。

## 常见误区

- `temperature: 0` 可以替代多 Trial 评测。
- 概率最高的答案就是事实最准确的答案。
- 多数投票一定能提高正确率。若多个 Trial 共享同一系统性偏差，投票只会重复偏差。
- 模型自报置信度可以直接决定是否执行高风险 Tool。
- 单次 Demo 成功足以证明 Agent 已经稳定。

## 本章小结

生成模型输出的是条件概率分布，Agent Loop 又会把每次选择反馈到外部环境。工程目标不是强行让模型变成确定性函数，而是用多 Trial 测量波动，并由 Runtime、Policy 与权威 Outcome 将随机性限制在安全范围内。下一章将同样的边界应用到 Embedding 与检索：相似度能够生成候选，但不能证明事实、权限或时效。
