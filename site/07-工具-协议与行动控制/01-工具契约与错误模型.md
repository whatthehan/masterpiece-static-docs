# 01 · Tool Contract 与错误模型

模型能生成一个 Tool Call，只说明它选择了某项能力并给出了候选参数。真正把 Agent 与外部系统连接起来的，是 Tool Contract：Runtime 如何验证输入、以谁的身份执行、超时后是否能重试、结果如何返回，以及用户最终会看到什么变化。

如果工具只有名称和 JSON Schema，读取与写入、可逆与不可逆、业务冲突与网络失败就会混在同一条调用路径中。本章把 Tool 设计为模型与确定性系统之间的完整协议。

## 本章目标

- 从模型、Runtime 和用户效果三个视角定义 Tool。
- 区分 Query、Draft、Command 和高风险能力。
- 设计输入、输出、timeout、retry 和 cancellation 语义。
- 把错误映射为 Runtime 可以处理的稳定类型。
- 在最终 sink 前执行确定性安全检查。

## 1. Tool Contract 有三层

### 模型可理解层

模型需要知道：

- Tool 的稳定名称和用途；
- 什么时候使用、什么时候不要使用；
- 输入与输出 Schema；
- 与相似 Tool 的边界；
- 少量典型例子和失败提示。

这一层影响 Tool selection 和参数生成质量。

### Runtime 执行层

Runtime 需要知道：

- 领域语义与前置条件；
- actor、tenant、resource、action 与 policy；
- timeout、deadline 和 `AbortSignal`；
- retryability、idempotency 和 expected version；
- concurrency、rate limit、audit 和敏感字段；
- result size、receipt 与 reconciliation 方法。

这一层决定 proposal 是否能成为 command。

### 用户效果层

产品需要解释：

- 这是读取、生成草案还是提交变化；
- 影响哪个对象、范围和第三方；
- 是否披露数据；
- 是否可逆，如何撤销或补偿；
- 是否需要 approval；
- 成功后提供什么 receipt。

一个 JSON Schema 只能覆盖三层中的一小部分。

## 2. 先分类基本行为，再标记风险属性

| 类型      | 含义                             | 示例                           |
| ------- | ------------------------------ | ---------------------------- |
| QUERY   | 不修改业务状态，但可能泄露数据或产生费用           | `get_order`、`search_policy`  |
| COMPUTE | 确定性转换，无外部持久副作用                 | `calculate_refund_amount`    |
| DRAFT   | 生成待审阅 proposal、diff 或 artifact | `draft_refund`、`draft_email` |
| COMMAND | 修改外部状态                         | `commit_refund`、`send_email` |

`PRIVILEGED` 和 `IRREVERSIBLE` 不是第五、第六种基本行为，而是与 `ToolKind` 正交的影响属性：`deploy_prod` 可以同时是 `command + privileged`，`delete_account` 则可以是 `command + privileged + irreversible`。基本行为决定执行协议，影响属性决定额外的 policy、approval、credential 和 sandbox 要求。高影响能力通常拆成：

```text
read current state
→ draft / preview
→ validate
→ approve
→ commit
→ verify receipt
```

不要设计一个同时查询、修改、退款和通知的 `manage_customer`。

## 3. 一份可执行的 TypeScript Tool 定义

```ts
type ToolKind = "query" | "compute" | "draft" | "command";
type ToolImpact = "privileged" | "irreversible";

type ToolDefinition<TInput, TOutput> = {
  name: string;
  description: string;
  kind: ToolKind;
  impacts: ToolImpact[];
  inputSchemaVersion: string;
  inputSchema: unknown;
  risk: "low" | "medium" | "high";
  requiresApproval: boolean;
  timeoutMs: number;
  maxAttempts: number;
  execute(
    input: TInput,
    context: ToolExecutionContext,
  ): Promise<ToolResult<TOutput>>;
};

type ToolExecutionContext = {
  runId: string;
  attemptId: string;
  actorId: string;
  tenantId: string;
  scopes: string[];
  idempotencyKey?: string;
  expectedResourceVersion?: string;
  signal: AbortSignal;
};
```

`actorId`、`tenantId`、scope 和 credential 来自 Runtime，不出现在模型填写的 input 中。已知字段由代码注入，可以同时减少错误与越权风险。

## 4. 输入设计：让非法状态难以表示

- 用 enum 或 tagged union 代替互相冲突的 boolean。
- 金额使用整数最小货币单位，并设置业务上限。
- ID 作为 opaque string，不让模型构造内部身份字段。
- 写操作携带 `expected_version` 与稳定 idempotency key。
- 日期、URL、路径和自由文本都设置长度与格式边界。
- 默认值由明确的应用层产生，不让模型猜测。

例如，`commit_refund` 不需要模型提交 `userId`、`tenantId` 或 `isAdmin`。它只需要提供订单、金额和理由；资源归属和权限从请求上下文读取。

## 5. 输出要帮助 Runtime 继续决策

Tool Result 不应只返回一段自然语言。一个通用结构是：

```ts
type ToolResult<T> =
  | {
      ok: true;
      data: T;
      observedAt: string;
      sourceRefs?: string[];
      resourceVersion?: string;
      receiptRef?: string;
      warnings?: string[];
      nextCursor?: string;
    }
  | {
      ok: false;
      error: ToolError;
    };
```

大型结果返回 artifact ref 或 cursor，避免把全部对象塞回 Context。输出仍属于 observation：它可能过期、包含外部恶意文本或敏感字段，进入模型前需要裁剪、脱敏和信任标记。

## 6. 稳定错误模型

```ts
type ToolError = {
  code:
    | "INVALID_ARGUMENT"
    | "NOT_FOUND"
    | "CONFLICT"
    | "UNAUTHENTICATED"
    | "DENIED"
    | "APPROVAL_REQUIRED"
    | "RATE_LIMITED"
    | "UNAVAILABLE"
    | "DEADLINE_EXCEEDED"
    | "CANCEL_REQUESTED"
    | "IN_DOUBT"
    | "PROTOCOL_ERROR"
    | "INTERNAL";
  retryable: boolean;
  publicMessage: string;
  detailsRef?: string;
};
```

| 错误                             | Runtime 行为                        |
| ------------------------------ | --------------------------------- |
| `INVALID_ARGUMENT`             | 不重试；修正 proposal 或请求输入             |
| `NOT_FOUND` / `CONFLICT`       | 重新读取权威状态，再决定下一步                   |
| `UNAUTHENTICATED` / `DENIED`   | 不重试；重新授权或明确拒绝                     |
| `APPROVAL_REQUIRED`            | 暂停并创建 approval，不属于普通失败            |
| `RATE_LIMITED` / `UNAVAILABLE` | 在预算内退避重试                          |
| `DEADLINE_EXCEEDED`            | 对 command 先确认真实效果，不能盲重试           |
| `CANCEL_REQUESTED`             | 停止新工作，检查在途副作用                     |
| `IN_DOUBT`                     | 查询 receipt/业务状态，进入 reconciliation |
| `PROTOCOL_ERROR`               | 停止解析当前响应；Command 已发出时先按效果未知处理     |
| `INTERNAL`                     | 失败、告警并保留诊断信息                      |

错误是否可重试由 Tool Contract 和具体错误共同决定，不由模型从错误文本临时猜测。

## 7. Timeout、Retry 与 Cancellation 必须定义到副作用层

对 query，timeout 通常意味着可以在预算内重试；对 command，timeout 只说明本地没有拿到结果，服务端可能已经提交副作用。

```text
query timeout
  → bounded retry may be safe

command timeout
  → effect unknown
  → reuse same idempotency key
  → query receipt / source of truth
  → settle as succeeded / failed / manual review
```

收到 cancel 后停止创建新动作，并向可取消的下游传播信号。已经提交的 command 不能用本地 cancel 伪装成未发生。

## 8. Schema 合法不等于可以安全送入 Sink

最终解释器不同，安全控制也不同：

| Sink            | 确定性控制                                                             |
| --------------- | ----------------------------------------------------------------- |
| SQL             | 参数化查询、限定表/语句/tenant，不拼接模型文本                                       |
| Shell / process | 固定 executable、分离 argv、allowlist option/value/path；禁止任意 shell 解析   |
| File path       | 限定根目录、规范化后校验、处理 symlink 与 TOCTOU                                  |
| URL / HTTP      | 校验 scheme/host/IP/port，每次 redirect 重验，阻断 metadata/private network |
| HTML / message  | 按最终输出上下文编码，不信任模型声称“已清洗”                                           |
| Deserialization | 只接受允许类型与大小，禁止动态代码或危险对象恢复                                          |

这些检查必须位于 sink adapter。上游通用 sanitizer 不知道最终输入会被 SQL、shell 还是 HTML 解释。

## 9. Tool 描述也需要评测

Tool selection 错误不一定是模型能力不足，也可能来自接口设计：

- 两个 Tool 名称和描述几乎相同；
- Tool 过于宽泛；
- 参数包含模型无法知道的字段；
- 同时暴露大量无关 Tool；
- 错误信息只有自由文本，没有稳定 code；
- Tool Result 过大，淹没后续 Context。

评测应覆盖选择准确率、参数语义、无效调用率、平均步骤、错误恢复和单位成功任务成本。

## 实践：拆分 Resolution Desk 的售后能力

### 进入本章时已有能力

Resolution Desk 已有带来源的政策 RAG、受治理 Memory 和只读 Agent Loop，但工具仍需要统一描述执行语义、用户效果与错误恢复方式。

### 本章增加的能力

将一个模糊的 `manage_order` 拆为：

```text
search_orders
get_order
get_shipment
get_refund_policy
draft_refund
commit_refund
get_refund_outcome
```

这份清单用于定义契约与 Fixture；普通 Runtime 此时只注册 Query 与 Draft。`commit_refund` 不进入模型可用工具集，直到隔离幂等实验和第 08 模块的安全、可信 UX 门禁全部通过。

然后准备 Fixtures：资源不存在、跨 Tenant、金额越界、资源版本冲突、重复提交、提交后响应丢失，以及 Schema 合法但 URL 指向内网。为每个 Tool 写出：

- model-facing definition；
- Runtime contract；
- user-visible effect；
- error code 与 retry rule；
- enforcement point；
- Trace 和 receipt 字段。

### 验收证据

同一组 Fixture 覆盖资源不存在、跨 Tenant、金额越界、资源版本冲突、重复提交、提交后响应丢失，以及 Schema 合法但 URL 指向内网。Runtime 对每种失败给出稳定错误码、合法下一状态和 Retry 规则；Query、Draft、Command 与 Outcome Query 不会共享含糊的成功语义。

## 常见误区

- HTTP 200 等于业务动作成功。
- Read-only Tool 没有安全或成本风险。
- 模型最适合决定一个错误是否可重试。
- Tool 越接近底层 API，模型越容易正确使用。
- 返回自然语言错误就足以完成恢复。

## 本章小结

Tool Contract 连接模型提案、Runtime 执行和用户可见效果。Schema 只定义输入形状；身份、权限、版本、retry、receipt 和 sink safety 决定动作能否可靠执行。下一章将把 actor、resource、action 和具体 proposal 纳入[身份、授权与审批](/masterpiece-static-docs/07-工具-协议与行动控制/02-身份-授权与审批.md)。
