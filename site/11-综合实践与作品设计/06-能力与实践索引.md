# 06 · 能力与实践索引：把阅读转化为工程证据

读完相关章节，只代表接触过这些概念；工程理解还需要通过图、代码、Fixture、反例或故障结果来验证。本章把全书知识映射到可选择的实践，不要求每个项目同时实现全部条目。

## 1. 如何使用这份索引

先根据当前项目和职业目标选择一个薄切片，再用三列信息检查理解是否落地：

- **建议实践**给出最小工件或实验；
- **可观察结果**说明机制成立时应该看到什么；
- **常见失败信号**帮助识别只会复述术语、却没有守住边界的实现。

标记“安全关键”的主题涉及权限、数据隔离或真实副作用，不能只用书面说明代替故障实验。暂时不适合连接真实系统时，可以使用 Mock、Recorded Fixture 或纸面时序；与当前场景无关的主题可以明确跳过并记录理由。

## 2. 任务、模型与实验基础

| 主题                        | 建议实践                                                | 可观察结果                                 | 常见失败信号                        |
| ------------------------- | --------------------------------------------------- | ------------------------------------- | ----------------------------- |
| 任务契约与 Baseline            | 用户、输入、Outcome、禁止动作、预算；30–50 个 Case；非 Agent Baseline | Case 可重复，Outcome 不依赖模型自评，Dataset 有版本  | 只有 Demo Prompt；成功标准是“看起来不错”   |
| 概率与 Sampling              | 同一 Case 的多 Trial 和分布                                | 能区分高 Token Probability、低随机性与事实正确      | 只跑一次就比较模型                     |
| Token 与 Context           | Tokenization 记录、截断与位置干扰实验                           | 能解释容量、有效利用和成本差异                       | 把 Context Window 当无限可靠 Memory |
| Transformer / KV Cache    | Prefill / Decode / Cache 图                          | 不把 Causal Mask 当正确性，不把 KV Cache 当长期状态 | 用模型原理替代应用验证                   |
| Outcome / Trajectory Eval | 分层 Grader 与 Trace 示例                                | 权威业务结果和执行轨迹分别评分                       | 只判断最终文本；LLM Judge 未校准         |
| 实验设计                      | Dataset 分区、多 Trial、区间、版本与停止规则                       | 不把相关 Trial 当独立样本，不污染 Holdout          | 选择性报告最好结果                     |

## 3. Model API 与 Agent Runtime

| 主题                         | 建议实践                                                                 | 可观察结果                                                     | 常见失败信号                                              |
| -------------------------- | -------------------------------------------------------------------- | --------------------------------------------------------- | --------------------------------------------------- |
| TypeScript 状态建模            | State/Event/Item/Error Discriminated Union                           | 未处理分支能在编译期暴露                                              | 大量可选字段和布尔值组合                                        |
| Provider Streaming Adapter | OpenAI SSE Frame → Responses Event → Item Reducer 与 Recorded Fixture | 多 Item、重复、Gap、断流和不完整 Function Call 均有确定行为                 | TCP Chunk 或参数 Delta 直接驱动 Tool / UI                  |
| Structured Output          | Schema、正反 Fixture、语义校验表                                              | Missing/Null/Default 区分；Schema 合法仍需授权/Sink 校验             | 把 JSON 合法当业务正确                                      |
| Agent Loop                 | 可运行状态机、10 行以上转移表、8 条不变量                                              | Model 只提议；预算、Cancel、终态和 Unknown Effect 完整                 | 开放 `while`；终态仍能调用 Tool                              |
| Tool Contract              | Query/Command、Schema、语义、权限、Timeout、Idempotency、Receipt               | 同一错误能归因到明确层，Command Timeout 可收敛                           | Tool 只是一个函数名和参数 Schema                              |
| Canonical Run Event        | RunEvent、Snapshot、Event Store、UI Reducer                             | 重放、断线、Snapshot 和旧 Client 测试通过                             | Provider Event 被当作领域事实                              |
| AG-UI Edge Adapter         | Canonical Fixture → AG-UI Event、Shared State 与 Control Command       | Native UI 与 AG-UI Client 最终状态一致，外层 Run 才决定 `RUN_FINISHED` | 把 Provider `response.completed` 或 AG-UI Event 当领域终态 |
| Framework 对照               | 同一 Task/Eval 下的手写 Runtime 与 Framework 版本                             | 能说明 Framework 管了什么、没管什么、如何移除                              | 先学 API，再猜底层语义                                       |

## 4. Context、Knowledge 与 Memory

| 主题                             | 建议实践                                             | 可观察结果                                   | 常见失败信号         |
| ------------------------------ | ------------------------------------------------ | --------------------------------------- | -------------- |
| Context Builder                | 可执行 Builder 与 Context Manifest                   | 每项有 Source、Trust、Freshness、Budget 和纳入理由 | 把全部历史拼进 Prompt |
| Compaction / Reset / Isolation | 前后对拍 Fixture 与丢失检查                               | 关键约束、未决事项和证据引用不丢失                       | 摘要覆盖原始事实       |
| Provenance 与 Freshness         | Source Schema、冲突与过期处理表                           | 结果能追到版本化来源，冲突不会由模型静默决定                  | 只有相似度分数        |
| ACL-scoped Retrieval（安全关键）     | 检索前 Tenant/ACL 图、Recall/Precision Eval           | 无权内容从候选生成前排除，后置检查只做兜底                   | Top-k 后才过滤 ACL |
| Memory Policy（安全关键）            | Candidate/Write/Read/Update/Delete 规则            | 来源、置信、TTL、冲突与删除传播完整                     | 模型反思直接写长期事实    |
| 删除验证（安全关键）                     | Thread、Chunk、Vector、Cache、Trace、Eval 与 Backup 映射 | Canary ID 在所有可服务副本中不可见                  | 只删除聊天记录        |

## 5. Identity、Action 与 Security

| 主题                          | 建议实践                                                        | 可观察结果                                       | 常见失败信号                                |
| --------------------------- | ----------------------------------------------------------- | ------------------------------------------- | ------------------------------------- |
| 身份与委派（安全关键）                 | actor–tenant–resource–action–purpose 矩阵、Delegation Chain    | 身份由可信代码注入，Resource Service 再授权              | 共享高权限 API Key；信任 `authorized=true`    |
| Approval（安全关键）              | 不可变 Proposal 与 Approval Record                              | 参数、目标、版本或期限变化后旧 Approval 失效                 | “允许 Agent 使用工具”式泛化审批                  |
| Idempotency / Receipt（安全关键） | Intent、Attempt、幂等记录与 ACK-loss 时序                            | 同一 Intent 多次投递只产生一次可观察效果                    | Timeout 后换 Key Retry                  |
| MCP Boundary                | Host/Client/Server 与信任边界图                                   | 能说明 MCP 解决互操作，不替代 Runtime、Policy 或 Sandbox  | 接入 MCP 就认为工具可信                        |
| A2A Boundary（安全关键）          | AgentCard、Task Reducer、A2A v1 Wire Fixture、断线后 `GetTask` 对账 | Remote Agent 的身份、Scope、预算、终态和 Artifact 都可追踪 | 把 Remote Agent 当普通函数；SSE 断开后直接重建 Task |
| Threat Model（安全关键）          | 数据流、资产、攻击者、滥用故事和不变量                                         | 每条不变量有 enforcement point 与真实 Outcome Eval   | 只列 OWASP 名称                           |
| Prompt Injection（安全关键）      | Direct/Indirect/Persistent/Cross-agent 攻击链                  | 每条至少两个独立阻断，模型失效后仍无越权效果                      | 只过滤固定字符串                              |
| Sandbox / Sink（安全关键）        | File/Network/Process/Secret/Resource 控制矩阵                   | SSRF、Path、Shell、egress 和资源耗尽测试通过            | 把 Container 或 Rust 当安全边界              |
| Incident Response           | Detect–Contain–Establish Facts–Recover–Learn Playbook       | Kill Switch 后在途效果仍被核对，真实 Case 回流 Eval       | 所有 Run 被批量写成 Failed                   |

## 6. Reliability、UX 与 Operations

| 主题                             | 建议实践                                                               | 可观察结果                                    | 常见失败信号                                       |
| ------------------------------ | ------------------------------------------------------------------ | ---------------------------------------- | -------------------------------------------- |
| Timeout / Retry / Cancel（安全关键） | Execution/Effect 双轴、Retry 放大计算、Reconciliation 时序                   | Cancel 不等于 Rollback；Command Timeout 不盲试  | Timeout 直接写失败未执行                             |
| Backpressure                   | Fan-out 上界、Bounded Queue、Bulkhead 与满队列语义                           | 任一输入都有并发、排队、Token 和费用上界                  | `Promise.all` 当并发控制                          |
| Durable Execution              | Checkpoint、Event History、Activity、双 Worker 时序                      | 新 Epoch 接管，旧写入被 Fencing/CAS 拒绝           | Checkpoint 只是最后一条消息                          |
| Exactly-Once 边界                | Delivery/Handler/Commit/Effect 四层说明                                | 多次 Attempt 收敛为一个业务效果                     | 宣称 Queue 的 Exactly-Once 等于第三方效果一次            |
| Agent UX                       | Public Run State、六个恢复界面、Control Event                              | UI 不发明终态；Stop/Retry/Resume 与 Runtime 一致  | Loading/Error 两个状态承载所有任务                     |
| A2UI Renderer（安全关键）            | 固定 Catalog、Envelope/Graph Validator、Surface Reducer、Action Gateway | 未知组件、危险 URL、伪造 Action 与超量 Payload 均被拒绝   | 执行模型生成的 JSX；动态加载 Catalog；信任客户端 Authorization |
| Trace / Audit / Eval           | 同一 `run_id` 的因果链和责任链                                               | 能从 Outcome 追到 Context、Approval、Tool 与版本  | 日志很多但无法复盘                                    |
| SLO / Cost                     | Task Success、Time to Truth、P95/P99、Cost per Successful Task        | 安全零容忍不被普通错误预算稀释                          | 只看 Model Uptime 和 Token 单价                   |
| Release                        | Behavior Bundle、Offline/Shadow/Canary/Rollback 记录                  | Shadow 无副作用；旧 Run 可恢复；Rollback 恢复完整行为    | 只回滚容器镜像                                      |
| Provider Failure               | Fallback 或非 Fallback 故障策略                                          | 采用 Fallback 时通过同一 Eval；不采用时有界等待/拒绝/人工可演练 | 故障时临时换模型                                     |
| Drain / Disaster Recovery      | Worker Drain、RTO/RPO 与恢复演练                                         | 未知效果不丢失，Queue Replay 不重复副作用              | 关闭连接就算安全停机                                   |

| 主题                            | 建议实践                                            | 可观察结果                                           | 常见失败信号                                    |

## 8. 两种使用方式

八周学习阶段可以从每个模块各选一个最小实践，重点形成 Task Contract、可运行 Runtime、Context/Tool 边界和一条故障实验。完整节奏见[八周学习与实践路径](/masterpiece-static-docs/11-综合实践与作品设计/04-八周学习与实践路径.md)。

准备生产作品或 Production Readiness Review 时，再根据实际风险补齐持久恢复、安全关键项、真实 UI、SLO 与发布实验。场景专项需要哪些证据，由[场景专项与作品设计](/masterpiece-static-docs/11-综合实践与作品设计/08-场景专项与作品设计.md)中的任务约束决定，而不是由一张统一清单强制决定。

## 本章小结

这份索引的价值是把“理解过”转换成“能够重建、实现和证伪”。可以先使用[综合能力自测](/masterpiece-static-docs/11-综合实践与作品设计/02-综合能力自测.md)定位知识缺口，再按项目风险选择实验；截图和 Demo 不能替代真实 Outcome 与故障恢复。
