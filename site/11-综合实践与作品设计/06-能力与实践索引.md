# 06 · 能力与实践索引：把阅读转化为工程证据

读完相关章节，只代表接触过这些概念；工程理解还需要通过图、代码、Fixture、反例或故障结果来验证。本章把全书的三层主线映射到 Resolution Desk：LLM 底层解释模型能力与限制，Agent 层负责状态、工具、知识和行动，Agentic UI 层把权威状态变成人可以理解和控制的界面；Eval、安全、可靠性与可观测证据横切三层。迁移到其他产品时，再按真实能力和风险裁剪条目。

## 1. 如何使用这份索引

Resolution Desk 主线先完成正常、拒绝和恢复三类纵向切片，再用三列信息检查理解是否落地：

- **建议实践**给出最小工件或实验；
- **可观察结果**说明机制成立时应该看到什么；
- **常见失败信号**帮助识别只会复述术语、却没有守住边界的实现。

标记“安全关键”的主题涉及权限、数据隔离或真实副作用，不能只用书面说明代替故障实验。Resolution Desk 始终使用 Mock、Recorded Fixture 或纸面时序，不连接真实支付与生产订单；与迁移后场景无关的主题可以删除，不必为了覆盖索引制造功能。

| 主题                             | 建议实践                                                                       | 可观察结果                                           | 常见失败信号                                   |
| ------------------------------ | -------------------------------------------------------------------------- | ----------------------------------------------- | ---------------------------------------- |
| Context Builder                | 可执行 Builder 与 Context Manifest                                             | 每项有 Source、Trust、Freshness、Budget 和纳入理由         | 把全部历史拼进 Prompt                           |
| Compaction / Reset / Isolation | 前后对拍 Fixture 与丢失检查                                                         | 关键约束、未决事项和证据引用不丢失                               | 摘要覆盖原始事实                                 |

| 删除验证（安全关键）                     | Thread、Chunk、Vector、Cache、Trace、Eval 与 Backup 映射                           | Canary ID 在所有可服务副本中不可见                          | 只删除聊天记录                                  |

## 5. 主线二与安全边界：Identity、Action 与 Protocol

| 主题                           | 建议实践                                                                                              | 可观察结果                                                                                    | 常见失败信号                                           |
| ---------------------------- | ------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- | ------------------------------------------------ |
| 身份与委派（安全关键）                  | actor–tenant–resource–action–purpose 矩阵、Delegation Chain                                          | 身份由可信代码注入，Resource Service 再授权                                                           | 共享高权限 API Key；信任 `authorized=true`               |
| Approval（安全关键）               | 不可变 Proposal 与 Approval Record                                                                    | 参数、目标、版本或期限变化后旧 Approval 失效                                                              | “允许 Agent 使用工具”式泛化审批                             |
| Idempotency / Receipt（安全关键）  | Intent、Attempt、幂等记录与 ACK-loss 时序                                                                  | 同一 Intent 多次投递只产生一次可观察效果                                                                 | Timeout 后换 Key Retry                             |
| MCP Boundary                 | Host/Client/Server 与信任边界图                                                                         | 能说明 MCP 解决互操作，不替代 Runtime、Policy 或 Sandbox                                               | 接入 MCP 就认为工具可信                                   |
| Agent Skill（安全关键）            | approved publisher/version/digest、progressive disclosure、trigger/outcome eval                     | 加载内容与脚本可追踪，Skill 声明不能扩大 Tool 权限                                                          | 把 Markdown 指令或 `allowed-tools` 当作授权              |
| Dynamic Tool Discovery（安全关键） | discover→shortlist→authorize→load→execute→observe 与 Recall\@k/Exposure Eval                       | Context 缩小且 Unauthorized Exposure 为 0，无结果时安全停止                                           | 搜到 Tool 就执行；把远端 Description 当可信指令                |
| MCP Extensions               | Apps sandbox/CSP/Host Proxy、Tasks Handle Fixture、Auth Extension Claims                            | 可展示不等于可信确认；Task 完成不等于业务 Outcome；Token 不替代资源授权                                            | iframe 当受信 UI；MCP Task 替代 Workflow；机器 Token 冒充用户 |
| A2A Boundary（安全关键）           | AgentCard、Task Reducer、A2A v1 Wire Fixture、断线后 `GetTask` 对账，以及相对单 Agent Baseline 的 Admission Plan | Remote Agent 的身份、Scope、预算、终态和 Artifact 都可追踪；不采用远程分支也能说明删除理由                              | 把 Remote Agent 当普通函数；SSE 断开后直接重建 Task；协议可用就默认上线  |
| Threat Model（安全关键）           | 数据流、资产、攻击者、滥用故事和不变量                                                                               | 每条不变量都有确定的执行位置与真实 Outcome Eval                                                           | 只列 OWASP 名称                                      |
| Prompt Injection（安全关键）       | Direct/Indirect/Persistent/Cross-agent 攻击链                                                        | 每条至少两个独立阻断，模型失效后仍无越权效果                                                                   | 只过滤固定字符串                                         |
| Agent Red Team（安全关键）         | Threat Hypothesis、攻击 Harness、确定性 Outcome、最小复现与 Regression 入库                                      | 核心 Tool/MCP/Memory/AG-UI/A2UI/Native UX 攻击可测；生产启用 Multi-Agent/A2A 时追加远程攻击链；模型识别与系统阻断分别评分 | 只做聊天越狱；攻击成功后没有回归样本和 owner                        |
| Sandbox / Sink（安全关键）         | File/Network/Process/Secret/Resource 控制矩阵                                                         | SSRF、Path、Shell、egress 和资源耗尽测试通过                                                         | 把 Container 或 Rust 当安全边界                         |
| Incident Response            | Detect–Contain–Establish Facts–Recover–Learn Playbook                                             | Kill Switch 后在途效果仍被核对，已确认的事件模式回流 Eval                                                    | 所有 Run 被批量写成 Failed                              |

## 6. 主线三：Agentic UI

| 主题                                | 建议实践                                                                                     | 可观察结果                                                                         | 常见失败信号                                                                       |
| --------------------------------- | ---------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| Application Server / Public State | Canonical RunEvent、Snapshot、Sequence、Command 与公开状态投影                                     | UI 只显示已提交事实；断线后通过 Replay 或 Snapshot 恢复                                        | 浏览器或 Provider Event 自行定义业务终态                                                 |
| AG-UI Edge Adapter                | Canonical Fixture → AG-UI Event、Shared State 与 Control Command                           | Native UI 与 AG-UI Client 最终状态一致；固定版本明确 success 或 interrupt `RUN_FINISHED` 的边界 | 把 Provider `response.completed` 或 AG-UI Event 当领域终态；混用稳定 SDK 与 Interrupts 扩展 |
| Agent UX / Native Trust（安全关键）     | 运行时间线、Evidence、Waiting、Unknown Effect、Cancel/Resume 与原生 Approval UI                      | 人能区分运行状态、外部效果和责任；高风险确认显示精确 Proposal                                           | 只有 Loading/Error；模型文案或生成界面冒充已审批                                              |
| A2UI Renderer（安全关键，必做）            | 至少一个低风险 Surface；`@a2ui/web_core/v0_9`、固定 Catalog、Envelope/Graph Validator、Action Gateway | 未知组件、危险 URL、伪造 Action 与超量 Payload 均被拒绝；Action 回到服务端重新授权                       | 把 A2UI 当可选 Demo；执行模型生成的 JSX；动态加载 Catalog；信任客户端 Authorization                 |
| UI 组合与安全矩阵                        | Canonical Event → AG-UI Runtime → 受控 A2UI Payload；高风险 Approval 保留在原生界面                   | 协议可以组合但权威边界不移动；UI Contract 与 Red Team Fixture 进入核心发布门禁                        | 让 A2UI 承载最终退款 Approval；DOM 或 Shared State 成为权威事实                             |

## 7. 横切证据：Reliability、Observability 与 Operations

| 主题                             | 建议实践                                                                       | 可观察结果                                                     | 常见失败信号                                           |
| ------------------------------ | -------------------------------------------------------------------------- | --------------------------------------------------------- | ------------------------------------------------ |
| Timeout / Retry / Cancel（安全关键） | Execution/Effect 双轴、Retry 放大计算、Reconciliation 时序                           | Cancel 不等于 Rollback；Command Timeout 不盲试                   | Timeout 直接写失败未执行                                 |
| Backpressure                   | Fan-out 上界、Bounded Queue、Bulkhead 与满队列语义                                   | 任一输入都有并发、排队、Token 和费用上界                                   | `Promise.all` 当并发控制                              |
| Durable Execution              | Checkpoint、Event History、Activity、双 Worker 时序                              | 新 Epoch 接管，旧写入被 Fencing/CAS 拒绝                            | Checkpoint 只是最后一条消息                              |
| Exactly-Once 边界                | Delivery/Handler/Commit/Effect 四层说明                                        | 多次 Attempt 只对应一个可观察的业务效果                                  | 宣称 Queue 的 Exactly-Once 等于第三方效果一次                |
| Trace / Audit / Eval           | 同一 `run_id` 的因果链和责任链                                                       | 能从 Outcome 追到 Context、Approval、Tool 与版本                   | 日志很多但无法复盘                                        |
| OpenTelemetry Propagation      | HTTP→Queue→Worker→Model/MCP 的 Trace Context、低基数 Span 名称与脱敏 Attribute       | Critical Path 与累计 fan-out 成本可见，Trace/Audit/Eval 各守边界      | 只在 Web Handler 建 span；Prompt 和 Token 泄露到普通 Trace |
| SLO / Cost                     | Task Success、Time to Truth、P95/P99、Cost per Successful Task                | 安全零容忍不被普通错误预算稀释                                           | 只看 Model Uptime 和 Token 单价                       |
| Model Routing / Cascade        | Route Policy、Risk/Capability/Region 约束、固定模型对照与 Router Eval                 | Protected Slice 无错误下放，升级率与单位成功任务成本可解释                     | 模型自报难度决定权限；只比较请求均价                               |
| Release                        | Behavior Bundle、Offline/Shadow/Canary/Rollback 记录                          | Shadow 无副作用；旧 Run 可恢复；Rollback 恢复完整行为                     | 只回滚容器镜像                                          |
| Controlled Improvement（安全关键）   | Signal/Finding/Eval Target/Change Proposal/Candidate Bundle 与权限矩阵          | Optimizer 不见 Holdout、不能改 Grader/Policy/发布；失败候选不上线，退化版本可回滚 | 把生产反馈、Reflection 或自动 Prompt 结果直接写入在线配置           |
| Provider Failure               | Fallback 或非 Fallback 故障策略                                                  | 采用 Fallback 时通过同一 Eval；不采用时有界等待/拒绝/人工可演练                  | 故障时临时换模型                                         |
| Drain / Disaster Recovery      | Worker Drain、RTO/RPO 与恢复演练                                                 | 未知效果不丢失，Queue Replay 不重复副作用                               | 关闭连接就算安全停机                                       |
| Production Topology            | Application Server/Worker/Queue/Store/Tool Gateway 责任图、Probe、Fencing 与迁移矩阵 | SSE 可重连、Worker 可接管、旧 Event/Checkpoint 可恢复或转人工             | Pod Running 被当成业务健康；Sticky Session 替代持久状态        |

## 9. 两种使用方式

按主线构建 Resolution Desk 时，这份索引用于检查每个里程碑是否同时留下概念、运行与故障证据。完整节奏见[八周构建路径](/masterpiece-static-docs/11-综合实践与作品设计/04-八周学习与实践路径.md)，最终验收条件见[总装与验收](/masterpiece-static-docs/11-综合实践与作品设计/09-Resolution-Desk总装与验收.md)。

把能力迁移到其他场景时，根据真实输入、Action 和 Outcome 裁剪索引；安全关键项不能删除，场景无关的协议和高级能力不必保留。具体方法见[从 Resolution Desk 迁移到其他 Agent 场景](/masterpiece-static-docs/11-综合实践与作品设计/08-场景专项与作品设计.md)。

## 本章小结

这份索引的价值是把“理解过”转换成“能够重建、实现和证伪”。可以先使用[综合能力自测](/masterpiece-static-docs/11-综合实践与作品设计/02-综合能力自测.md)定位三层知识缺口，再按项目风险选择实验；AG-UI Adapter、受控 A2UI Surface 和原生高风险 Approval 边界属于核心成品证据，截图和 Demo 不能替代真实 Outcome 与故障恢复。
