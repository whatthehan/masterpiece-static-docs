# 附录 A · 一手资料索引

> 资料状态核验日期：2026-07-19。模型 API、SDK、协议和安全清单会持续变化，实施前必须重新查看对应官方页面。

这份索引用于查证，而不是要求从头读完的附加书单。正文先建立问题和心智模型；遇到机制、版本或边界结论时，再回到论文、规范和官方文档确认。

表中的阅读级别：

- **精读**：理解正文直接依赖的定义、图、实验和限制；
- **查阅**：实现具体能力时按需核对接口与边界；
- **延伸**：主线完成后，根据场景继续深入。

资料也需要区分性质：标准与官方文档用于接口事实，论文用于机制与实验边界，厂商工程文章用于观察真实实现。任何单一产品的架构都不能自动外推成行业标准。

索引按全书的三层主线组织：第 1–3 节建立 LLM 底层，第 4–6 节进入 Agent 系统，第 7 节专门处理 Agentic UI；安全、可靠性、Eval 与语言运行时是横切证据，不能被误解为第四套互不相关的知识。

本书的 UI 主线不是在协议名称中二选一：AG-UI 负责 Agent Runtime 与用户应用之间的双向事件、状态和控制；A2UI 负责受信 Renderer 可以解释的声明式 Surface。二者都以 Application Server 的 Canonical Event、Public State、服务端 Authorization 和原生高风险 Approval 为边界。

MCP Apps 是相邻的重要分支：它让 MCP Server 通过预声明 `ui://` 资源向 Host 提供沙箱 iframe，并以 JSON-RPC `ui/*` 消息双向通信，适合第三方工具拥有交互式 View 的场景；它不是 A2UI Renderer，也不替代 AG-UI 的应用运行时通道。OpenAI Apps SDK 与 Vercel AI SDK UI 用于观察产品或框架如何实现交互式 / 生成式 UI，不作为本书的跨厂商领域契约。

截至核验日期，A2UI v0.9.1 是 Current Production，v1.0 是 Candidate；生产实现应固定 v0.9.1，并复用 `@a2ui/web_core/v0_9`。MCP Apps 的 SEP-1865 状态为 Final，但它仍是可选 Extension，必须经过 Capability Negotiation，且 Host 支持范围需要逐一核对。AG-UI 文档与 SDK 可能在 Event 数量和 Interrupt 语义上演进，因此正文只依赖事件类别与领域映射，并用固定版本 Contract Fixture 锁定 Snapshot、Resume、Terminal 与 A2UI Payload 序列。

OpenTelemetry 的 GenAI Semantic Conventions 仍在演进；应用应固定版本并通过 Adapter 映射，不把实验字段直接写入领域模型。自 2026-03 起，官方建议新事件优先通过 Logs API 发出，并与当前 Span 关联；Span Event API 将渐进弃用，现有数据与视图仍受支持。该结论已于 2026-07-19 重新核验。

Self-Refine 与 Reflexion 说明模型可以在推理或跨 Trial 的文本反馈中修订候选，但它们不提供独立真值，也不自动更新模型权重。DSPy / GEPA 属于 Python 生态的优化案例，适合观察 Dataset、Metric、Trajectory Reflection 与候选搜索，不成为本书 TypeScript 主线依赖。生产系统的正式变更仍须经过独立 Holdout、安全门禁、审批和可回滚发布。

## 11. 查阅后的最低产出

每次查阅一手资料后，记录：

1. 该资料支持了哪条具体技术判断；
2. 判断成立的版本、实验条件或适用范围；

## 本章小结

一手资料的价值在于校准事实边界，而不是增加收藏数量。标准定义协议，论文说明机制和限制，厂商文档说明具体实现；三者需要在工程决策中明确区分。随后可使用[能力与实践索引](/masterpiece-static-docs/11-综合实践与作品设计/06-能力与实践索引.md)检查这些判断是否已经转化为可验证产出。
