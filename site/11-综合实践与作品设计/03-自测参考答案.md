# 03 · 自测参考答案与判断要点

本章给出关键判断点，而不是唯一标准答案。设计可以不同，但需要覆盖相同的不变量、执行位置与可观察证据。只重复题干、堆叠术语或给出一段 Prompt，不能证明已经理解相应机制。

## A · 核心概念

每题给出两个必须说明的判断要点。

1. 模型给出给定 Context 下下一 Token 的条件分布；它没有内置事实查询或真值校验，因此概率不能直接映射为正确率。
2. Token 是离散单位，Token Embedding 是其初始向量表示；Contextual Hidden State 随上下文改变，Text Embedding 是面向检索等任务的序列表示。
3. Causal Mask 阻止当前位置读取未来 Token，维持自回归结构；它不保证事实、安全、可解释性或长上下文利用效果。
4. KV Cache 复用本次 Decode 的注意力中间量，Prompt Cache 复用相同前缀计算；Conversation State 保存 API 交互 Item/引用，Memory 是应用治理的跨 Run 信息。
5. Context Window 是容量上限，不是任务能力保证；位置、干扰、任务复杂度、组织方式和训练分布共同决定有效利用率。
6. 后训练或低 Temperature 可能改善指令遵循、行为倾向或降低采样变异；不能保证事实、授权、业务不变量或端到端确定性。
7. 原始 Chain-of-Thought 可能不忠实、含敏感信息且接口不稳定；应记录目标、计划摘要、来源、动作、状态、策略决策、Receipt 与 Outcome。
8. Workflow 的控制流主要由代码预先决定，Agent Loop 让模型根据 Observation 选择下一步；给 LLM 增加检索或工具不一定形成自主 Loop。
9. 任务契约和案例集定义输入、Outcome、禁止行为、预算与评测方法；非 Agent Baseline 提供更简单系统的质量、延迟和成本参照。
10. Structured Outputs 只能在 Provider 支持的 Schema 子集内约束结构，不能保证事实、语义、授权和 Sink 安全；Missing、显式 `null`、Default 与 Delete 是不同业务语义。
11. 模型只生成 Candidate；Runtime 等完整 Item 后做协议、Schema 和领域语义校验；随后还要经过授权、风险/Approval、Sink/Sandbox 执行与 Receipt/Outcome 核对。
12. TCP Chunk 是传输分段，应用需解析带 ID/类型的语义 Event，再由 Reducer 重建 Item；不完整 Arguments 无法验证，断流/重放还会引入半执行或重复执行风险。
13. Thread 是长期交互容器，Run 是一次目标执行，Item 是语义产物；Event 表示增量变化，Runtime State 是权威 Event/Store 导出的当前控制状态。
14. Host 管理应用、用户和策略，Client 连接 Server，Server 暴露 Tool/Resource/Prompt 等能力；MCP 是互操作协议，不自带 Planner、业务授权、Sandbox 或 Agent Loop。
15. Authentication 回答“是谁”，Authorization 回答“能否对资源行动”，Consent 回答“数据主体是否同意处理”；Delegation 表示代表谁在何范围/期限行动，Approval 表示是否确认这一份具体提案。
16. Context 是本次模型输入，Runtime State 是权威运行数据，Memory 是跨 Run 信息，Corpus 是外部知识源；ACL 必须在候选生成前缩小检索空间，否则会泄露、产生旁信道并造成 Recall Starvation。
17. Deadline 是总时限，Step Timeout 是一次等待上限，Cancellation 是停止新工作的协作信号；Compensation 是新的抵消动作，Rollback 是版本或状态恢复，二者都不能由 Cancel 自动保证。
18. 同一业务 Intent 的所有安全 Retry 复用相同幂等键，相同 Key + 不同 Payload 必须拒绝；`IN_DOUBT` 表示 Effect 证据不足，应查询 Receipt/权威状态而不是盲试。
19. Outcome Eval 检查真实结果，Trajectory Eval 检查路径、Tool 与 Policy 不变量；Trial 估计随机变异，区间表达有限样本下的不确定性。
20. Discriminated Union 让 State/Event 可穷尽处理，Event Loop 要避免被 CPU/阻塞工作占用；Stream/Backpressure 管理流量与有界缓冲，`AbortSignal` 传播取消意图但不承诺 Rollback。

## B · 故障场景

每个场景给出五个相互独立的判断要点。

### 21. ACK 丢失与 Cancel 竞态

1. 持久化 `CANCEL_REQUESTED`，停止创建新工作，并向在途调用传播。
2. Command 进入 `IN_DOUBT → RECONCILING`，不能直接写成 `CANCELLED`。
3. 使用原 Idempotency Key / Call ID 查询 Receipt 和权威退款状态。
4. 效果已发生则记录原 Receipt 与 `EFFECT_AFTER_CANCEL`；确认未发生才允许 `CANCELLED`。
5. 禁止用新 Key 盲目退款；UI 显示“效果核对中”或已确认事实，不伪称已撤销。

### 22. Indirect Prompt Injection 与 SSRF

1. 外部网页标记为不可信数据，不提升为高优先级指令，也不自动写入 Memory。
2. Policy 阻断 CRM 敏感数据流向公网，并在资源服务重新授权。
3. URL Executor 对 Scheme、Host、DNS/IP、Port 和每次 Redirect 重新校验，阻断 Private / Metadata Endpoint。
4. 执行环境使用最小凭证与 egress allowlist，不让同一执行面同时拥有广泛 CRM 读取与任意公网发送。
5. Outcome Eval 检查真实未外泄/未访问内网，Trajectory Eval 检查禁止数据流和阻断点，而不是只看拒绝文本。

### 23. Approval TOCTOU

1. Approval 绑定 actor、tenant、tool/action 与精确 Target。
2. 绑定 Amount/Currency、预期 Diff/Effect 与外发数据。
3. 绑定 Resource Version、Proposal Hash、Expiry 与 Idempotency Intent。
4. 任一关键字段、版本或期限变化都使旧 Approval 失效并返回重新预览。
5. 执行时重新认证、服务端授权和检查前置条件，不把 Approval 当 Authorization。

### 24. 级联 Retry

1. 三层 `max_attempts = 4` 的最底层上界是 `4³ = 64`。
2. 每层“首次之外 Retry 4 次”表示每层共 5 个 Attempt，上界是 `5³ = 125`。
3. 统一配置语义，将 Retry 责任集中在最了解操作语义的一层。
4. 使用有限指数退避 + Jitter，并传播剩余 Deadline 和全局 Retry Budget。
5. Attempt ID 每次不同；Command 保持同一业务幂等键，未知效果先查询。

### 25. ACL Retrieval 与删除

1. 在候选生成 / Vector Search 前，先按 Tenant、ACL、Purpose 限定 Corpus 或 Index Partition。
2. Context Packing / Tool Response 前再次检查权限、来源与 Freshness。
3. 分别评测 Authorized Recall、Precision、跨租户泄漏和无权内容占满 Top-k 的 Recall Starvation。
4. 删除传播至 Thread、Memory、Source Chunk、Vector、Cache、Trace、Eval 与 Backup；不能立即删除的层使用 Tombstone + 访问阻断 + 保留期。
5. 使用 Canary ID 做正反查询和 Cache/Index 失效检查，直到所有可服务副本不可见。

### 26. Stream 断连与 Backpressure

1. 按 Event/Item/Call ID 与 Sequence/Cursor 解析；Reducer 对重复 Event 幂等，对 Gap 或不可恢复乱序失败。
2. 部分 Arguments 保持 `incomplete`，只有显式完成的 Item 才进入校验和执行。
3. 重连从 Cursor Replay 或权威 Snapshot 恢复，不直接拼接 TCP Chunk。
4. 队列满时使用 Bounded Queue / Semaphore，等待到 Deadline、拒绝或降级，不无限缓冲。
5. `AbortSignal` 传播到 Stream、Queue、Tool；UI 显示重连、等待容量或取消中，已有 Command 则进入 Reconciliation。

## C · 系统设计

每个系统设计题给出十个可独立核对的判断要点。

### 27. 有界 Agent Loop

1. Model 只生成 Candidate / Proposal，Runtime 与 Executor 决定和执行。
2. 状态覆盖创建、模型运行、校验、工具执行和 Observation。
3. 区分 `WAITING_INPUT` 与 `WAITING_APPROVAL`。
4. 覆盖 Cancel Request、传播与收敛。
5. 覆盖 `IN_DOUBT / RECONCILING`，未知效果不盲试。
6. 终态区分 Completed、Refused、Denied、Failed、Cancelled、Partial、Budget Exhausted 和 Manual Intervention。
7. 至少 10 行状态转移表，非法转移明确拒绝。
8. 稳定 Call / Intent / Idempotency / Attempt ID，Approval 绑定不可变 Proposal。
9. Step、Token、Time、Money、Fan-out 和 In-flight Budget 在创建工作前强制；有未知效果时不直接进入预算终态。
10. 至少八条可检测不变量，包括终态不行动、授权、幂等、ACL 和 UI/Runtime 一致。

### 28. Eval 与实验方案

1. 任务契约写清用户、输入、期望 Outcome、禁止动作和预算。
2. 同时保留非 Agent Baseline 与当前 Agent Baseline。
3. 30–50 个 Case 覆盖正常、模糊、边界、失败、拒绝、攻击和 Protected Slice。
4. 优先使用权威业务状态做确定性 Outcome Grader。
5. Trajectory Grader 检查 Tool、权限、数据流、预算和禁止序列。
6. 主观质量先建立人工金标、评审手册和一致性，再校准 LLM Judge。
7. 随机任务按 Task 做多 Trial，保留配对结构和独立环境。
8. Dataset 分 Development / Regression / Holdout，固定 Model、Prompt、Tool、Policy、Runtime 与 Dataset 版本。
9. 报告适合数据结构的区间或 Bootstrap；预先定义改善阈值和停止规则。
10. 发布门禁同时覆盖总体、Protected Slice、安全零容忍项、P95/P99 与 Cost per Successful Task。

### 29. TypeScript + Node Runtime

1. 为 State、Event、Item、ToolResult 与 Error 定义 Discriminated Union，并使用 `never` 做穷尽检查。
2. Thread、Run、Item、Event 与 Snapshot 分离，具有稳定 ID、版本和原子状态转移。
3. SDK Stream 先适配为内部语义 Event，Reducer 处理重复、断流与不完整 Item。
4. Agent Loop 与状态表由应用持有，Model 不直接执行。
5. 一棵 `AbortSignal` 树传播到 Model、Queue、Tool 和子任务，并处理在途副作用。
6. Semaphore、Bounded Queue 与 Per-tenant Quota 提供 Backpressure，并定义满队列语义。
7. 下游继承剩余 Deadline，Retry 责任集中且有全局 Budget。
8. Tool Adapter 负责 Schema、语义、Authorization、Sink、Idempotency 与 Receipt 边界。
9. Trace、Audit、Eval Hook 完整，敏感正文不默认进入日志。
10. CPU/阻塞任务通过 Worker Thread、独立进程或外部服务隔离；Reducer、状态机、故障和并发都有测试。

## D · 协议专项

### 30. SSE、OpenAI Streaming 与 AG-UI

HTTP Chunk 只是网络分片，SSE Frame 提供事件 Framing，OpenAI Responses Event 表达 Provider 的 Response / Item 生命周期，Canonical RunEvent 表达应用已提交事实，AG-UI Event 是前端互操作投影。`response.output_item.done` 可闭合 Provider Item，`response.completed` 只闭合本次模型 Response；Tool 执行和下一次模型调用尚未结束时，稳定 AG-UI 基线不能发出成功语义的 `RUN_FINISHED`。若采用 Interrupts 扩展，则只能在 Snapshot 与 Resume Contract 完整时以 `outcome.type = "interrupt"` 闭合当前交互，不能据此推断领域 Workflow 已终止。应用 Replay 使用自己的 Snapshot、Cursor 与 Event Store，不能从 SSE 名称推导。

### 31. A2UI Renderer

Surface 管理声明式界面生命周期，Catalog 是本地组件与函数 Allowlist，邻接表 Component Graph 只表达结构，Data Model 只是界面状态，Action 只是待验证的用户意图。Renderer 至少验证 Envelope Schema / Version、Catalog、唯一 Root 与图结构、资源上限、URL / 富文本 Sink、Data 最小化和 a11y；Action Gateway 重新认证、读取领域状态并执行 Authorization / Approval / Idempotency。Fixture 应覆盖未知 Catalog、双 Envelope、乱序、图环、危险 URL、超量 Payload、敏感 Data Model、伪造 Action 和重复点击。

### 32. MCP 与 A2A

MCP Tool 适合发现和调用聚焦能力；独立 Remote Agent 拥有自己的 Runtime、长 Task、输入等待和 Artifact 生命周期时，A2A 的语义更贴合。Client 先验证 AgentCard、Interface、版本与 Security Scheme，再决定返回无状态 Message 还是跟踪 Task。`INPUT_REQUIRED` / `AUTH_REQUIRED` 可继续，`COMPLETED` / `FAILED` / `CANCELED` / `REJECTED` 是不可复活终态；断线后用 `GetTask` 对账。Discovery 与签名不等于信任，每次读取、取消、订阅和提交仍需租户级 Authorization、预算与审计。

### 33. 开放协议的版本漂移

A2A v1 以固定 Release 的 Normative `a2a.proto` 为数据定义基线；示例文档和 SDK 类型都属于派生实现。应用固定 A2A / A2UI 版本组合，把外部类型隔离在 Adapter，显式协商 Major.Minor 与 Extension URI，并保存 Header、Part、State、Artifact 的 Golden Wire Fixture。SDK 只有在这些 Fixture、失败路径与旧 Client 兼容测试通过后才能升级；预发布支持不应静默进入生产依赖。

## 如何利用参考答案修正理解

将每个遗漏映射到[能力与实践索引](/masterpiece-static-docs/11-综合实践与作品设计/06-能力与实践索引.md)，只重读最小必要范围，然后重做对应反例或故障实验。概念答案完整但系统设计无法落地时，以可运行状态机和 Fixture 为准；书面回答不能替代真实生产证据。
