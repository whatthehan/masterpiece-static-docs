# 08 · 从 Resolution Desk 迁移到其他 Agent 场景

Resolution Desk 提供的是一套完整参考实现，不是所有 Agent 产品的固定模板。迁移到 Research、Coding、Computer Use、Voice 或其他领域时，应保留 Model、State、Action 与 Outcome 的工程边界，再根据真实任务裁剪知识、审批、持久化和协议能力。

本章给出迁移时的采用条件与验收门槛，用于判断 Resolution Desk 中哪些机制能够直接复用，哪些只在新场景确有对应风险时才引入。

## 1. 先区分核心能力与场景扩展

| 范围                   | 已证明                                                                           | 仍需按新场景验证                                  |
| -------------------- | ----------------------------------------------------------------------------- | ----------------------------------------- |
| Resolution Desk 核心成品 | Task Contract、Web UI、Model Adapter、有界 Runtime、可信 Knowledge、受控 Action、恢复与 Eval | 新领域的用户价值、数据分布、权限和权威 Outcome               |
| 场景扩展                 | 在同一工程边界上增加一项经过验证的能力                                                           | 不代表需要同时实现所有 Framework、Protocol 或 Agent 类型 |

纯研究或内容系统不需要伪造交易流程；Transactional 系统也不必为了覆盖目录强行加入 Multi-Agent。迁移时首先重写 Task Contract 和 Dataset，而不是直接复制 Refund、Approval 或 Payment 领域对象。

## 2. 十类场景专项

| 专项                     | 新增输入与工具                                                            | 必须持有的状态                                                              | 主要风险                                         | 采用前的对照 Eval                                                   |
| ---------------------- | ------------------------------------------------------------------ | -------------------------------------------------------------------- | -------------------------------------------- | ------------------------------------------------------------- |
| Research               | Web、Document、Dataset；Search、Browser、Retrieval、Citation             | Source Version、Evidence Graph、Query Trace、Open Questions             | Injection、过期来源、虚假引用、结论越界                     | Evidence Recall、Faithfulness、结论正确率；优于固定检索或人工 Baseline         |
| Transactional          | 用户意图、领域事实、Policy；Query / Command API                               | Proposal、Approval、Resource Version、Receipt、Idempotency Record        | 越权、TOCTOU、重复效果、Unknown Effect                | 权威 Outcome、违规率、重复效果率、Time to Truth                            |
| Coding / DevTool       | Repository、Rule、Issue；Search、Edit、Shell、Test、Git                   | Workspace、Diff、Test Result、Branch / Attempt                          | 数据破坏、Secret 泄漏、供应链、错误修改                      | Test / Regression、Diff Quality、Review 接受率、Task Cost           |
| Computer Use           | Screenshot、DOM / Accessibility Tree；Mouse、Keyboard、Browser/Desktop | Screenshot、Action History、Window/App State                           | 错目标、敏感界面、视觉漂移、不可逆点击                          | Task Success、Action Accuracy、Recovery 与 Human Takeover        |
| Realtime / Voice       | Audio、Video、Realtime Event；Streaming、RTC、VAD、Interrupt             | Session Buffer、Turn State、Latency Budget                             | 延迟、抢话、误打断、隐私、网络抖动                            | First Response / Interrupt Latency、Recovery Rate、Task Quality |
| Ambient / Proactive    | Email、Calendar、Background Event；Connector、Scheduler、Notification   | Subscription、Cursor、Preference、Cooldown                              | 意外行动、过度打扰、权限蔓延、误触发                           | Intervention Precision、Accept/Undo Rate、False Positive        |
| Multi-Agent            | Task Decomposition、Delegation Envelope；Agent-as-Tool，跨系统时可选 A2A    | Parent/Child Run 或 A2A Task、Ownership、Budget、Cancel、Delegation Chain | 循环委派、权限放大、协调失败、Cost Explosion                | 相对单 Agent 的 Quality/Latency/Cost 与独立验证收益                      |
| Generative UI          | UI Intent、Component Catalog、A2UI Envelope；Trusted Renderer         | Surface、Component Graph、Data Model、Action、Catalog Version            | 误导界面、危险 URL、数据泄漏、伪造 Action、资源耗尽              | 相对固定 UI / Schema Form 的任务效率、错误率、a11y 与安全收益                    |
| Advanced Memory        | Interaction、Artifact、Feedback；Extract、Store、Retrieve、Update、Delete | Candidate/Validated Memory、Provenance、Confidence、TTL                 | Poisoning、Privacy、Staleness、Deletion Residue | Usefulness、Correctness、错误写入率、删除传播                             |
| Controlled Improvement | Trace、Eval、Feedback；Change Proposal、Experiment、Release Pipeline    | Change Proposal、Experiment、Approval、Release、Rollback                 | Eval Gaming、自强化漂移、未经批准上线                     | Holdout、安全、因果收益与 Rollback Drill                               |

专项名称只描述问题空间，不决定实现。Research Agent 可能只需要单 Agent + Workflow；Transactional Agent 也可能不需要 RAG。先从任务约束推导架构，而不是从标签反推技术栈。

## 3. 扩展能力的采用条件

任何扩展进入系统前，都必须能回答下面这些当前问题：

- 当前用户任务与简单 Baseline 是什么；
- Baseline 在哪项可测指标上不足；
- 新能力引入哪些输入、工具、持久状态和信任边界；
- 哪些不变量由 Model 之外的确定性系统实施；
- Dataset 需要增加哪些正常、拒绝、故障和 Protected Slice；
- 质量、延迟、成本与安全的通过阈值是什么；
- 未达到阈值时，能否移除该能力而不破坏领域契约。

没有更简单 Baseline，就无法证明复杂能力的必要性：

- Multi-Agent 必须与单 Agent + Workflow 对比；
- Generative UI 必须与固定界面或 Schema-driven Form 对比；
- Advanced Memory 必须与普通 Profile / Domain Store 对比；
- Agentic RAG 必须与固定 Retrieval Pipeline 对比；
- Computer Use 必须说明可靠 API 为何不可用或不足；
- Controlled Improvement 必须在成熟 Eval 与发布治理之后启用。

“流行”“像某个产品”或“模型更强”都不是采用证据。

## 4. 任务与 Eval

本节至第 7 节是可按场景裁剪的证据目录，不是所有作品都必须通过的统一清单。实现某项能力，才需要提供对应证据；不实现时，应说明不适用的原因。

- Dataset 规模由任务差异、随机性与风险决定。早期作品可以从 30–50 个高信息量测试样本开始；业务切片较多或风险较高时，再扩展到 100 个以上。不以数量本身作为质量标准；
- Development、Regression 与 Holdout 分离，生产失败回流有治理记录；
- Outcome 与 Trajectory 分开评分，关键结果以权威系统状态为准；
- 随机任务运行多 Trial，报告分布与适当的不确定性区间；
- 主观 Judge 有人工金标、评审 Rubric 与校准结果；
- 准备面向真实用户放量时，必须为 Protected Slice、安全不变量、Tail Latency 与 Cost per Successful Task 明确发布阈值；
- 所有结果记录 Model、Prompt、Context、Tool、Policy、Runtime 与 Dataset Version。

## 5. Runtime 与用户控制

- **所有 Agent 应用**：TypeScript + Node 主线可运行，具备有界 State Machine、Budget、Cancel 和稳定事件；
- **使用外部知识时**：Context Manifest 保存 Source、Trust、Freshness、ACL 与 Token Budget；
- **包含 Command 时**：Query / Command 分离，Tool Contract 包含语义校验、Authorization、Approval、Idempotency、Receipt 与 Outcome Query；
- **需要长时间等待时**：Run 才需要跨小时 Event、Durable Checkpoint 和 Worker 重启恢复；
- **使用多 Worker 或可重试写操作时**：Fencing/CAS、原 Idempotency Key 与权威查询证明同一 Intent 没有重复效果；
- **UI** 只展示当前场景真实存在的状态和合法控件。没有 Approval 的研究工具不需要伪造 Waiting Approval，但 Stop、Retry 和 Resume 仍不能由前端伪造权威状态。

## 6. Security

- 所有项目都说明 Asset、Actor、Data Flow、Trust Boundary 和与能力相符的 Abuse Story；
- 处理 Web、Document、Email 或 Tool Result 时，外部内容默认不可信，Indirect Prompt Injection 至少有两个独立阻断层；
- 多租户 Retrieval 在候选生成前执行 ACL，跨租户泄漏为零容忍项；
- Secret 不进入 Model Context、Tool Result、普通 Trace 与 Eval Dataset；
- 使用 URL、File、Shell 或 SQL 等 Sink 时，提供语义校验与相应的 Network/File/Process/Resource 限制；
- 实现 Memory Write 时，定义 Source、Confidence、TTL、Conflict、删除传播和必要的 Approval Policy；
- 开始服务真实用户前，演练 Kill Switch、Credential Revoke 和 Incident Response，并将已经确认的攻击模式加入回归数据集。

## 7. Reliability 与 Operations

- 包含 Command 时，Execution Status 与 Effect Status 分离，Timeout 后在效果未知时进入 Reconciliation；
- Retry 责任集中，Deadline、Backoff、Jitter 与全局 Budget 有上界；
- 有并发或外部配额风险时，Bounded Queue、Bulkhead、Quota 与 Circuit Breaker 经过过载测试；
- 使用 Durable Workflow 时，Checkpoint、Event History、Lease、Heartbeat、Fencing/CAS 和 DLQ 才需要故障实证；
- Span、Log、Audit 与 Eval 通过同一 `run_id` 对齐；Metric 只保留低基数维度，需要下钻时通过 Exemplar 定位代表性 Trace；
- 准备生产发布时，SLO 包含 Task Success、Time to Truth、Tail Latency 与 Cost，而不只是 Model Uptime；
- 依赖外部 Provider 时，注入不可用故障：若采用 Fallback，主备通过同一 Contract/Eval；若不采用，有界等待、拒绝、Partial 或人工接管与声明一致；
- 只有准备面向真实用户放量的 Behavior Bundle，才需要完成 Offline Eval、Shadow、Canary、Ramp、Rollback 与 Kill Switch 演练；
- 采用多 Worker 或声明 Disaster Recovery 目标时，验证 Worker Drain、旧/新 Run Routing 和对应 RTO/RPO。

## 8. 独立练习项目如何组织

本书仓库不承载应用源码。读者自己的独立练习项目应使外部评审者能够在不依赖口头解释的情况下找到并运行：

- Quick Start、架构图与 Trust Boundary；
- Versioned Dataset、Grader 与 Experiment Report；
- State Machine、Tool Contract、Policy 和 Event Schema；
- 一条正常路径、一条安全拒绝路径和一条故障恢复路径；
- Trace、Audit 或可离线复现的等价证据；
- 关键 Trade-off、已知限制和明确不做项。

准备服务真实用户的作品，还要补充 SLO Dashboard、Release Record、Rollback Drill 与 Incident Playbook。作品不能只提供 Demo Video、截图或一段理想路径日志。

## 9. 选择与场景相符的端到端验证

一个作品至少跑通一条正常因果链和一条关键故障链。验证路径应从场景风险推导，不必强制包含 Approval 或外部写操作。

**Research / Knowledge 作品**

```text
two conflicting sources
→ ACL / freshness check
→ evidence selection and explicit uncertainty
→ retrieval or provider failure
→ bounded retry / partial result / cancel
→ citation and outcome grading
→ Trace / Eval / Cost review
```

**Coding / DevTool 作品**

```text
仓库规则与包含失败用例的测试报告
→ scoped search
→ patch proposal and diff review
→ sandboxed test execution
→ forbidden command rejected
→ regression result confirms outcome
→ Trace / Eval / Cost review
```

**Transactional 作品**

只有选择了受控写操作，才在 Mock、Sandbox 或获准的 Staging 环境演练完整效果链：

```text
two conflicting sources
→ ACL / freshness check
→ clarification
→ immutable proposal
→ precise approval
→ command committed, ACK lost
→ user cancel requested
→ provider limited, worker restarted
→ new ownership epoch restores checkpoint
→ reconcile with original idempotency key
→ UI presents authoritative outcome
→ Trace / Audit / Eval / Cost review
→ fix released through Shadow / Canary / Rollback drill
```

三类路径都把 Context、Loop、Harness、UX、Reliability 和可验证 Outcome 放在同一条因果链中；Authorization 和副作用恢复只在存在 Command 时加入。任何一步如果只能用“模型应该能够处理”解释，就仍缺少工程控制或证据。

## 10. 从一个完整系统迁移能力

熟练使用 Claude Code、Codex 等工具形成的直觉，在这里获得了更完整的解释：项目规则进入 Context，搜索与测试组成反馈 Loop，Permission 和 Sandbox 限制行动，Diff 与 Test Result 比完成声明更可信，Compaction、Reviewer 和 Subagent 都是可评测的 Harness 组件。

同样的原则可以迁移到研究、交易、客服、数据和个人效率场景。核心问题始终是：模型看见了什么，谁允许动作发生，状态由谁持有，失败后如何得到明确结论，Outcome 由什么证据确认，以及新版本凭什么开始服务真实用户。

完成本书不是背完术语，也不是在每个项目中复刻 Resolution Desk；真正形成的能力，是把一个自然流畅的 Agent 行为拆成可实现、可证伪、可恢复和可运营的工程系统，并根据场景删除不必要的复杂度。
