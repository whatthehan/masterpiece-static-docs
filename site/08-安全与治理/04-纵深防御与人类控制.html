#04 · 纵深防御与 Human-in-the-Loop

Resolution Desk 已完成威胁模型、Injection 防线和受限委派，但退款写操作仍不应依赖任一单层控制。模型层可能没有识别危险请求；策略层应因订单版本或金额异常拒绝；即使策略配置也出错，支付执行器仍受金额上限、独立凭证和环境隔离限制。这样的系统不是“永不犯错”，而是不会让一个错误直接穿透所有边界。

这就是纵深防御（Defense in Depth）的工程含义：不同责任层使用相互独立的机制降低攻击成功率、强制业务不变量并限制真实影响。Human-in-the-Loop（HITL）是其中一种控制手段，但只有在状态、责任和提案都明确时才有效。

#1. 三层防御承担不同责任

层次典型机制主要作用不能替代
Model清晰指令、来源标签、分类器、结构化输出降低产生危险候选的概率授权、金额上限、网络隔离
Policy / RuntimeSchema 与语义校验、授权、审批、预算、状态机强制业务不变量并决定能否执行操作系统级隔离、第三方最终状态
Execution Environment最小凭证、Sandbox、Egress、文件/进程/资源限制限制策略突破后的影响范围（Blast Radius）正确的业务审批、用户意图

高影响控制通常至少需要两个独立的执行位置。例如，禁止读取云服务的 Metadata Endpoint，既要在 URL Policy 中检查，也要由网络层阻断；两者不应共用同一份未经校验的域名判断结果。

#2. Human-in-the-Loop 有四种不同语义

  • Clarification：目标、参数或证据不足，需要用户补充信息。
  • Approval：系统已经生成一份确定提案，需要有资格的人批准其执行。
  • Review：结果已经生成，需要人工检查质量、依据或合规性。
  • Escalation：系统无法在当前权限、预算或确定性下继续，由明确责任人接管。

把这四种状态都做成“是否允许”弹窗，会造成两个问题：用户无法判断自己在补信息还是承担执行责任；Runtime 也无法决定超时、拒绝和恢复后的合法转移。

一个明确的状态模型可以写成：

type HumanGate =
  | { kind: 'clarification'; questions: string[]; resumeToken: string }
  | { kind: 'approval'; proposalId: string; proposalHash: string; expiresAt: string }
  | { kind: 'review'; artifactRef: string; rubricVersion: string }
  | { kind: 'escalation'; queue: string; reasonCode: string; owner?: string };

这些对象是 Runtime 的持久状态，不是前端临时弹窗。

#3. 有效审批必须足够具体

审批界面至少展示：

  • 原始 actor、目标资源和动作；
  • 精确参数、变更 Diff 与资源版本；
  • 会发送到外部的字段和目标系统；
  • 金额、费用、可逆性与潜在影响；
  • 提案依据、风险提示、有效期和替代方案；
  • 审批后由哪个 Executor 使用什么范围的权限执行。

审批记录应绑定 Proposal Hash。对象、参数、资源版本、策略版本或期限发生关键变化时，旧审批失效。Runtime 可以生成新的低风险提案，但不能换一个工具反复绕过同一拒绝。

“批准 Agent 完成退款”不是有效提案，因为它没有限定订单、金额、资源版本和执行期限。

#4. 取消、Kill Switch 与补偿不是同一个动作

这些控制经常在产品界面中被混为一个“停止”按钮：

控制能保证什么不能保证什么
Cancellation停止创建新工作，并向在途步骤传播取消意图已提交的外部效果一定撤销
Kill Switch阻止某类新 Planning/Command，撤销短期凭证所有 Run 立即得到真实终态
Compensation以新的授权动作抵消已有业务效果回到从未发生过的历史状态
Rollback恢复代码、配置或数据的某个版本自动撤销第三方已经执行的动作

因此，安全响应不能在按下 Kill Switch 后把所有任务批量写成 FAILED。在途 Command 和未知效果仍需查询权威状态、去重、补偿或进入人工异常队列。

#5. 检测与响应是一条闭环

安全控制上线后仍会失效或漂移。生产系统至少需要：

  1. Detect：通过异常工具序列、权限拒绝率、未知目标、循环调用和费用突增发现问题；
  2. Contain：停止相关新动作，撤销短期凭证，限制网络出口和并发；
  3. Establish facts：查询已经发生的外部效果，确定受影响资源、Tenant 和版本；
  4. Recover：恢复兼容版本，执行必要补偿或转人工处理；
  5. Verify：将已经确认的攻击模式固化为 Regression Fixture 和发布准入条件。

处置链不保存原始 Chain-of-Thought，只保留核对当前事实所需的模型/Prompt 版本、Context 来源、Tool Call、Policy Decision、Approval、Executor、Receipt、状态转移和 Trace ID。

#6. 按能力逐级提高安全证据

新增能力最低证据
仅生成文本敏感数据边界、输出评测、版本记录
只读工具ACL、来源标签、Injection 测试、Timeout 与 Trace
写工具服务端授权、精确审批、幂等、Receipt、审计与补偿策略
代码/浏览器操作强 Sandbox、egress、Secret 隔离、资源上限与用户确认
长任务/多 Agent持久状态、委派链、恢复、撤销与级联故障测试

新增复杂度必须附带相应的控制与验证。一个原本只回答问题的应用加入邮件发送工具后，不能继续沿用同一套“输出文本安全”指标。

#实践：让退款写操作承受单层失效

#进入本章时已有能力

Resolution Desk 已能阻断 Injection、传递受限委派并在资源服务重新授权；退款 Executor 尚未向常规业务 Run 开放。

#本章增加的能力

为一份冻结的 Refund Proposal 建立原生 Approval Record，并区分 Clarification、Approval、Review 与 Escalation 四种持久状态。设计一个 Schema 合法但金额、资源版本或目标账户危险的 Tool Call，建立以下矩阵：

Model ControlPolicy ControlEnvironment Control预期真实结果
正常正常正常在最早控制点拒绝
失效正常正常Policy 拒绝
失效失效正常Environment 限制效果
正常失效失效Model 可能降低概率，但不能作为安全通过证据

Approval 必须由可信服务绑定 actor、订单、金额、资源版本、Proposal Hash 和有效期；不能由模型生成，也不能由通用“允许 Agent 行动”替代。随后补充检测信号、Kill Switch 范围、在途效果核对、补偿条件和人工责任人。

#验收证据

分别绕过 Model、Policy 或 Environment 中的一层，确认其余层仍能阻止越权、超额、过期或篡改后的退款。合法 Proposal 只有在原生 Approval Record、执行前重新授权和 Mock 支付额度限制同时成立时，才能进入 command_ready；本章仍以 Dry Run 验证，不开放真正的退款调用。若任一单层失效即可造成高影响结果，纵深防御尚未建立。

#本章小结

纵深防御不是堆叠多个同类分类器，而是让 Model、Policy 和 Execution Environment 分别承担概率控制、确定性约束和影响隔离。HITL 只有被建模为 Clarification、Approval、Review 或 Escalation，并绑定具体状态与责任时，才能成为可靠控制。Agentic UI 主线已经完成这些状态的产品化投影；下一章用 Agent 安全评测与 Red Team 主动组合攻击链，验证任一界面、模型或执行控制失效时，系统仍不会越权产生真实效果。
