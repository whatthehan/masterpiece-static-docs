#03 · 预训练、后训练与推理

当模型给出错误事实、没有按要求调用 Tool，或者生成了不合法的参数时，团队常会更换更大的模型、提高推理投入（Reasoning Effort），或者继续扩写 Prompt。这些手段有时有效，但前提是问题确实源于模型能力或当前 Context。

Agent 工程需要区分能力来源：哪些模式写在模型权重中，哪些信息由当前 Context 提供，哪些事实必须检索，哪些效果只能通过 Tool 产生。

#贯穿项目：Resolution Desk

Resolution Desk 在本章建立能力来源映射（Capability Source Map）：订单状态属于权威 Tool Observation，当前政策属于带版本的 Retrieval Evidence，输出格式由 Prompt 和 Schema 约束，授权与真实退款则由确定性应用负责。这个映射会约束后续实现，避免用更换模型或提高 Reasoning Effort 掩盖数据、权限和 Runtime 缺陷。

#1. 模型生命周期的基本结构

一个现代 LLM 的训练过程可以用以下阶段理解。具体 Provider 的数据和配方可能不同，未公开部分不应推测为事实。

#Pretraining

模型在大规模数据上学习预测 Token，获得语言、代码、知识模式和一般表征能力。Pretraining 使模型能够生成连贯内容，但目标本身不是“查询权威事实”或“遵守某个业务流程”。

#Continued Pretraining / Mid-training

模型可能继续在特定领域、长 Context、多模态或其他分布上训练，以调整能力覆盖。它仍属于权重层面的学习。

#Supervised Fine-Tuning（SFT）

模型从高质量示范中学习如何响应指令、使用目标格式或执行特定类型任务。

#Preference Optimization

通过人类或 AI 偏好信号调整回答倾向。Reinforcement Learning from Human Feedback（RLHF）、Direct Preference Optimization（DPO）等方法属于这一大类中的不同实现。

#Reasoning-oriented Training

使用可验证奖励或其他反馈强化多步推理、检查和规划行为。它可以改善复杂任务表现，但不会让模型自动获得当前业务事实或系统权限。

#Distillation

将更强模型的部分行为迁移到较小模型，以换取成本、延迟或部署优势。

这些阶段改变的是模型权重和行为分布，不改变应用的领域数据库、Tool 权限或 Workflow State。

#训练、适配与应用改进不是同一条路径

产品上线后，“系统从反馈中变好”可能指向三种完全不同的机制：

机制改变什么发生位置典型证据与门禁
Offline OptimizationPrompt、Example、Retriever、Model Route 或其他 Behavior Artifact与生产隔离的实验环境Development / Regression / Holdout Eval、审批、发布与回滚
Online Adaptation当前请求或 Run 的 Context、Route、State Projection 与 Memory ReadAgent Runtime预算、适用条件与当前 Run 的行为评测
Application Memory Update（应用记忆更新）可跨 Run 读取的 Long-term Memory Candidate 与 Record VersionMemory Writer 与受控 StoreScope、来源、同意、有效期、冲突、撤销和删除验证
Online / Continual Learning在线或持续更新模型权重Model Training System数据治理、训练稳定性、安全评测和独立发布流程

当前 Run 读取一条已发布 Memory 属于 Online Adaptation；把用户修订发布为可跨 Run 读取的新版本，属于 Application Memory Update；从脱敏 Trace 生成一版候选 Prompt，则属于 Offline Optimization。三者都不等于模型在生产请求中更新权重。应用层的“自进化”因此不能作为一个含混实现要求：后文会把它拆成可审计的 Controlled Agent Improvement（受控 Agent 改进），其中运行中的 Agent 只能提出候选，不能自行定义评分标准或发布新版本。

#2. 五种能力来源

来源是否改权重生命周期适合解决的问题
Pretraining / Post-training是Model Version通用能力、语言和行为倾向
In-Context Examples否当前请求临时说明任务、格式与少量示例
Retrieval / Knowledge否外部数据版本动态、可引用、有来源的事实
Tools否执行时查询或改变外部系统
Application Policy / Workflow否应用版本权限、业务不变量、步骤与恢复规则

退款案例可以清楚展示这五层：

  • 模型权重提供理解自然语言和比较条款的通用能力。
  • Context 中的示例说明期望的资格判断格式。
  • Retrieval 提供当前有效的退款政策。
  • getOrder Tool 查询实时订单状态。
  • Policy 判断当前客服是否有权创建退款；Workflow 管理审批和提交。

若把这五项都称为“模型能力”，故障归因就会失去意义。

#3. In-Context Learning 不会永久更新模型

向 Prompt 加入示例后，模型可以在当前请求中模仿任务结构，这通常称为 In-Context Learning。模型权重并没有因此改变；新 Session 若不再提供示例，行为也可能消失。

Few-shot Example 适合：

  • 展示输入输出格式。
  • 说明难以用一句话表达的分类边界。
  • 给出少量代表性 Tool 使用方式。

它不适合承载：

  • 持续变化的订单、价格与库存。
  • 大规模政策和知识库。
  • 服务器端 Authorization。
  • 需要强制执行的安全不变量。

示例越多也不必然越好；冲突、陈旧或过度相似的示例可能引入噪声和偏置。

#4. Base、Instruction 与 Reasoning Model

这些名称描述训练目标和行为差异，不是安全等级。

  • Base Model 更接近续写分布，通常不直接作为面向用户的指令接口。
  • Instruction Model 经过后训练，更倾向遵循角色、任务和格式要求。
  • Reasoning Model 可以在推理阶段投入更多计算，适合复杂规划、数学、代码和歧义分析。

Reasoning Model 仍可能引用错误事实、误用 Tool 或违反业务语义。更高 reasoning effort 通常意味着额外延迟和成本，应通过任务 Eval 决定，而不是全局设为最高。

一个简单事实查询如果已有权威 API，直接调用 Tool 往往比要求模型进行长推理更可靠。复杂政策冲突则可能从更多推理计算中获益，但最终执行仍需确定性校验。

#5. 为什么模型会生成不受支持的结论

模型的直接目标是生成符合当前条件分布的序列，而不是从事务数据库返回带来源的记录。当证据缺失、Context 冲突或前缀已经包含错误时，模型仍可能继续生成语言上合理的内容。

“Hallucination”是结果描述，不是单一根因。至少应区分：

表现可能根因主要修复方向
引用不存在的政策必要证据缺失或模型补全模式Retrieval、Citation Verification、Abstain
使用了过期政策Freshness / Version 未进入选择条件Knowledge Metadata 与 Context Builder
订单金额错误权威数据未查询或 Tool Result 解析错误Tool、Schema 与 Domain Validation
明知无权限仍提出退款模型行为或 Prompt 问题可以优化模型行为，但执行必须由 Policy 阻断
Tool 已成功却回答失败Runtime State 或 Event Mapping 错误Trace、State Machine 与 Outcome Query

一句更强的 Prompt 只能覆盖其中一部分。

#6. Structured Outputs 的位置

Constrained Decoding 或 Structured Outputs 可以把模型输出限制在 JSON Schema 允许的结构内。例如：

const RefundDecision = z.object({
  orderId: z.string(),
  eligible: z.boolean(),
  amount: z.number().nonnegative(),
  evidenceIds: z.array(z.string()),
});

即使输出通过 Schema，仍可能出现：

  • orderId 属于其他 tenant。
  • amount 超过可退余额。
  • evidenceIds 指向过期或无权文档。
  • eligible 与政策条件矛盾。

Schema 提供语法和结构保证；业务语义、权限和真实结果仍由应用验证。

#7. Reasoning Trace 与可审计性

模型可能返回 reasoning-related Item、Summary 或解释文本。它们有助于产品展示和调试，但不应默认视为完整、忠实的内部因果过程。

可审计系统优先记录：

  • Model、Prompt、Context Snapshot 和 Toolset Version。
  • 结构化计划或当前步骤。
  • Tool Proposal、参数、Policy Decision 和 Approval。
  • Tool Execution、Receipt 与 Outcome。
  • Evidence ID、Citation 和简洁 Decision Summary。

这些记录可重放、可校验，也能避免将潜在敏感的原始 Chain-of-Thought 当作日志长期保存。

#8. 模型选择必须回到 Task Eval

比较模型时至少固定：Dataset、Context Builder、Tool Contract、Policy、Runtime 和环境 Fixture。随后测量：

  • Task Success 与 Safety Violation。
  • Tool Selection、参数正确率和 Trajectory Efficiency。
  • Refusal、Abstain 与 Clarification 行为。
  • 延迟、Token 和单位成功任务成本。
  • 多 Trial 方差和尾部失败。

模型规模更大或 Reasoning Effort 更高，只有在目标 Task 上带来足够收益时，才值得承担额外成本。模型名称和公开榜单不能替代应用级 Eval。

#9. 最小实验

沿用 Resolution Desk Dataset 中的四个静态 Task，比对普通推理和较高 reasoning effort 两组 Recorded Output。可以使用模型控制台生成候选，也可以手工准备两组记录；本实验不要求 Tool 已经可调用，给模型的 Tool Observation 作为固定 Fixture 注入，候选 Tool Call 只评分、不执行。Task 至少包含：

  1. 只需调用权威 Tool 的简单事实查询。
  2. 需要比较多条政策条件的复杂判断。
  3. 信息不足、正确行为是 Clarification 的任务。
  4. JSON Schema 合法但业务参数非法的对抗任务。

记录每项配置的成功率、候选 Tool Call、参数语义、拒绝、延迟、Token 与成本。实验结论应按 Task Slice 分解；不能用复杂任务的提升掩盖简单任务不必要的成本增长。将结论写入 Resolution Desk 的 Model Evaluation Matrix，后续接入真实 Model Adapter 后使用相同 Task 重跑，而不是重新选择一批更有利的样本。

#常见误区

  • Pretraining 相当于对互联网建立了可精确查询的压缩数据库。
  • SFT 适合持续注入实时业务事实。
  • In-Context Learning 会永久更新模型权重。
  • Model 越大，事实性和安全性必然越高。
  • Reasoning effort 越高，所有任务都会更好。
  • Structured Outputs 保证业务参数正确。
  • 模型解释文本可以直接作为授权或审计依据。

#本章小结
