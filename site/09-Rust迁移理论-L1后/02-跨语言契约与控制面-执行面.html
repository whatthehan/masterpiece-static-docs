#02 · 跨语言契约与控制面/执行面

本章是 L1 之后 的迁移理论，不计入首个 TypeScript + Node Agent 的毕业门禁。

把一个组件改写成 Rust 并不难，难的是证明改写前后仍然是同一个系统：金额、时间、取消、授权、错误和审计只要有一个字段语义漂移，就可能出现“类型都通过，业务却变了”的迁移事故。

这一章把迁移视为契约演进问题。关注点从单侧实现转向两侧都能验证的 wire semantics、身份、故障语义与发布证据。

#学习目标

  • 用稳定进程协议连接 Node 控制面和 Rust 执行面。
  • 设计可版本化 Schema、wire type、错误、取消、Trace 和权限上下文。
  • 用 contract/golden/property tests、shadow 和 rollback 证明迁移等价。

#1. 推荐边界

进程边界使你能在两侧独立部署、回滚，并施加故障与资源限制；它本身不会自动提供 CPU/内存/网络/凭证隔离。这些仍需 cgroup/容器/沙箱、网络策略、身份和可观测性配置。不要默认使用 N-API 形成进程内强耦合，除非已实测 IPC 是瓶颈且仍能满足安全边界。

#2. Schema 与语义的单一来源

协议至少包含：

  • request/response/event discriminated union。
  • schema/protocol 版本和 capability。
  • trace/run/call/attempt/idempotency ID。
  • actor/tenant/purpose/scope、delegation chain 和 policy decision reference。
  • deadline/cancel correlation。
  • structured error code、retryable 和 unknown-effect 标记。
  • resource version、receipt、audit 字段与 ownership epoch。

可以从 JSON Schema/Protobuf 生成两侧类型，但生成类型只解决语法一致，不解决金额单位、时间边界、Unicode、授权或取消语义。

#3. Wire type 必须明确到字节

概念推荐 wire 语义禁止的默认假设
ID/idempotency key不透明 string，区分大小写与命名空间当 JS number、做数学运算
金额币种 + 最小货币单位整数，或定义精度的十进制 string；若整数可超 Number.MAX_SAFE_INTEGER 就必须用 stringIEEE-754 float，或无范围声明的 JSON number
时刻RFC 3339 UTC string；若用 epoch，必须指定单位，且超出 JS 安全整数时用十进制 string无时区本地字符串或无单位 number
持续时间明确单位的 integer；超出 JS 安全整数时用十进制 string；本地超时用 monotonic clock把时刻和时长混用
Deadline绝对 deadline + 接收方按当前时间重算并 clamp 剩余预算盲信跨主机 monotonic 数值
Missing/null区分未提供、显式空、默认和删除TS optional 和 Rust Option 自动等价
Bytes指定 base64 variant、大小上限与 content type将任意字节当 Unicode string
TextUTF-8；仅在业务需要时固定 Unicode normalization/casefold 规则视觉相同就字节相同
Canonical hash固定 canonical serialization、字段顺序、编码和 hash 版本对两边普通 JSON stringify 直接求 hash

#4. 执行服务的身份与完整性

Node 传入一个 JSON 中的 allowed: true 不是权限。执行面至少需：

  • 用 mTLS 或等价 workload identity 验证调用方服务身份。
  • 接受 audience-bound、短期、最小 scope 的委派 token，不透传广权用户 token。
  • 引用不透明且可核验的 policy decision ID/签名决策，绑定 actor、resource、action、arguments hash、expiry 和 nonce。
  • 用 call/idempotency ID、nonce、deadline 和服务端去重防 replay。
  • 最终资源服务在执行时重新授权，不把控制面判断当作永久许可。
  • 重要 event/receipt 要有完整性保护和可归属 audit。

#5. Contract、Golden 与 Property Tests

  • 同一 fixture 在 TS/Rust 解码、验证和编码结果一致。
  • Golden vectors 覆盖 ID 前导零、金额边界、时区/闰秒策略、null/missing、Unicode 等价字形、空 bytes 和 canonical hash。
  • Property tests 检查 round-trip、大小上限、无溢出、canonicalization 和去重不变量。
  • 未知字段、未知 enum、旧/新版兼容策略明确。
  • Error code 到 retry/state transition 的映射一致。
  • Cancellation、timeout、断流、迟到响应和 unknown effect 一致。
  • Trace parent/baggage 传播正确；权限上下文缺失默认拒绝。

#6. 迁移证据和版本演进

record production-like fixtures
→ replay against old/new
→ shadow new service without effects
→ compare outcome/trace/eval parity
→ limited canary
→ monitor SLO/cost/security
→ expand or rollback

不仅比较 HTTP 文本，还比较真实业务状态、错误分类、审计、延迟分布和副作用次数。

变更优先 additive；删除或改语义需新版本。Consumer 先兼容，producer 再发新字段。协议要区分“未知但可忽略”和“未知必须失败”；回滚后的旧 Node/Rust 组合仍要兼容。

#7. 迁移门禁

只有以下至少一项有测量证据时才继续迁移：

  • p95/p99 延迟或吞吐瓶颈位于本地组件。
  • Node 内存/CPU/GC 造成 SLO 问题。
  • 需要单二进制、静态部署或可验证的更严格资源控制。
  • 协议/执行面需要更严格的并发和生命周期保证。

“Rust 理论上更快”不是验收标准。

#L1 后迁移实验

为一个只读工具建立 JSON/golden/property fixtures，用 TS 和 Rust 双实现对拍；加入金额、时间、null/missing、Unicode、canonical hash、旧版请求、未知 enum、迟到响应和重复 call。然后 shadow 运行，比较 outcome、错误、span、timeout、取消和审计，不允许 shadow 产生副作用。

#常见误区

  • 共享 JSON 就等于共享语义。
  • 生成类型后不需要 contract tests。
  • 进程边界自动带来资源和安全隔离。
  • 控制面传来 authorized=true 就可直接执行。
  • 新服务输出相同文本就完成迁移。
  • 进程内 FFI 总比 RPC 高级。

#章末检查

  1. 为什么跨语言比较必须包含 Trace 和真实 outcome？
  2. 金额、deadline、missing/null 和 canonical hash 应如何规定？
  3. 为什么 Node 传入的授权结果不能取代资源服务重新授权？
  4. 为什么先 process boundary、后按需 N-API 更稳妥？

#本章小结

跨语言迁移的核心交付物不是一个 Rust 服务，而是一份两侧都能执行的语义契约和一条可回滚的证据链。完成本章后，用 Rust 迁移门禁 判断是否应继续；证据不足时，“保持 TypeScript + Node”就是正确结论。
