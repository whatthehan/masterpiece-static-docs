#04 · 从一次 Run 到 Trace、SLO 与成本

Resolution Desk 现在可以跨 Worker 恢复同一条退款 Run，并确保同一 Intent 最多产生一个可观察效果。可靠执行仍不等于可运营：某次退款最终只发生了一次，但用户等待了十二分钟才得到确认；零散日志无法说明哪份政策进入了 Context、谁批准了提案、ACK 在哪个环节丢失、新 Worker 何时接管，以及费用为什么比正常请求高四倍。

“记录了很多日志”不等于系统可观测。Agent 应用需要把一次 Run 的输入版本、模型决策、工具轨迹、公共状态、用户 Action、策略、外部 Outcome 和费用连成因果链，才能判断质量、交互、可靠性和成本是否同时达标。

#1. 五类证据各有职责

信号主要回答的问题
Metrics系统整体是否正在退化？哪些率、分位数或队列深度异常？
Logs某个组件在某一时刻记录了什么错误或诊断上下文？
Traces一次 Run 跨模型、工具、队列和服务的因果路径是什么？
Audit谁代表谁，在什么授权和审批下，对什么资源做了什么？
Evals某个版本在固定任务与真实结果上是否变好或退化？

Trace 不能替代 Audit：Span 能说明调用关系，却不一定满足不可抵赖、保留和访问控制要求。Eval 也不能替代生产指标：离线用例通过，不代表 Queue Time 和 Provider Error Rate 正常。

这条记录没有保存模型原始 Chain-of-Thought，却足以还原 Context 选择、提案、审批、调用、Cancel、接管和真实 Outcome。

#3. 建立稳定的 Canonical Telemetry Schema

Provider SDK 和 GenAI Observability 约定会演进，应用内部仍需要稳定字段：

thread_id / run_id / item_id
call_id / attempt_id / activity_id
ui_protocol_version / public_state_version / surface_id / action_id
state_before / event / state_after
model_route / prompt_version / context_builder_version
knowledge_snapshot_version / retrieval_index_version / retrieval_schema_version
parser_version / chunker_version / query_rewrite_version / reranker_version
tool_contract_version / policy_version / runtime_version
actor_ref / tenant_ref / resource_ref / action
authorized_scope_digest / retrieval_query_ref / citation_map_ref
approval_ref / idempotency_key_hash / ownership_epoch
execution_status / effect_status / outcome_ref
queue_ms / latency_ms / tokens / money / retry_count

跨 Node、Rust、MCP Server 和外部 Adapter 传播 W3C Trace Context 或等价协议。领域代码依赖内部 Schema，再由 Adapter 映射到具体 OpenTelemetry 版本，避免实验字段渗入业务模型。

这些字段不应不加区分地复制到每类信号。trace_id、run_id、intent_id、attempt_id 和 ownership_epoch 等高基数标识用于 Span、结构化 Log、Audit、Canonical Event 和 Eval Record，以支持单次 Run 的定位与审计。Metric 只使用低基数 attribute，例如 route_class、tool_class、outcome、error_category 和 risk_class；不将 Run、用户、Tenant 或资源 ID 作为 label。需要从异常指标回到单次请求时，使用 Exemplar 将少量测量点关联到 Trace，而不是把 trace_id 变成 Metric label。

#4. 用 OpenTelemetry 连接跨进程因果链

OpenTelemetry（OTel）提供 Trace、Metric 和 Log 的采集与传播标准，但不会自动理解 Run、Proposal 或 Outcome。应用应先定义 Canonical Telemetry，再由埋点层（Instrumentation Layer）映射到 OTel：

一条完整链路需要同时处理同步和异步边界：

  1. Application Server 从入站 HTTP 请求提取 W3C traceparent；没有可信上游时创建新 Trace。
  2. 发布 Queue Message 时，把传播字段写入受控 metadata，不混入业务 payload 或模型 Context。
  3. Worker 消费时提取 Context；重试和批消费根据消息语义建立 child span 或 span link，同时保留 attempt_id。
  4. 调用 MCP Server、内部 HTTP Tool 或模型 Provider 时继续注入传播字段；第三方不支持时，至少在本地 client span 记录边界。
  5. Collector 负责批处理、采样、脱敏和导出；业务代码不直接依赖某个观测厂商的数据模型。

Trace Sampling 是观测成本策略，不是业务证据保留策略。正常成功流量可按 Head Sampling 或 Tail Sampling 规则降采样；错误、IN_DOUBT、安全不变量违规、Canary 与 Migration 验收 Trace 应按明确策略优先保留。采样决策、保留期和数据分类必须版本化并接受回归验证。Audit 与 Canonical Event 由独立的持久化通道保留，不受 Trace Sampling 影响，也不能依赖 Collector 或观测后端作为唯一副本。

Span 名称保持低基数，例如 agent.run、model.call、tool.execute、queue.process 和 reconciliation.query；run_id、Tool 名称与状态放入 Span attribute，用户输入和完整 Prompt 不进入 span name。自动 instrumentation 可以覆盖 HTTP、数据库和消息客户端，Agent Loop、Proposal、Approval 与 Reconciliation 仍需要手工埋点。

OpenTelemetry 的 Generative AI Semantic Conventions 仍在演进。实现应固定 Semantic Convention 与 SDK 版本，在 OTel Adapter 中映射模型、Usage、Tool 和 Agent Span；内部 RunEvent 与领域状态不能直接采用实验字段。升级约定时运行 Telemetry Contract Test，确认 Dashboard、Alert、Trace Link 和脱敏规则没有静默失效。

时效注记（核验日期：2026-07-16）：OpenTelemetry 已宣布逐步弃用 Span Event API。新代码应优先通过 Logs API 发出带名称的事件，并利用当前 Context 将其与 Trace 关联，而不是继续新增对 Span.AddEvent 或 Span.RecordException 的依赖。现有 Span Event 数据与面向 Span 的事件视图仍受支持，迁移将保持渐进和兼容，因此无需立即重写已有埋点。Agent 应用的 Canonical Event 仍应独立于 OTel API；升级 SDK 时，通过 Telemetry Contract Test 验证 Log-based Event 能否继续关联到正确的 Run、Span 和后端事件视图。参见 OpenTelemetry 官方弃用说明。

#4.1 Retrieval Trace 要能定位数据管线层次

只记录 search_policy took 420ms 无法判断故障发生在 Query Rewrite、授权范围、Candidate Generation、Reranking、Packing 还是 Citation Validation。一次知识查询至少形成下面的 Span 层次：

retrieval.run
├─ retrieval.rewrite
├─ retrieval.authorize_scope
├─ retrieval.candidate.sparse
├─ retrieval.candidate.dense
├─ retrieval.fuse
├─ retrieval.rerank
├─ retrieval.pack
└─ citation.validate

Trace 记录版本和引用，而不是复制完整 Query、Embedding 或敏感 Chunk：

source_snapshot_version / index_build_id / index_schema_version
query_hash / rewrite_version / subquery_count
authorized_scope_digest / purpose / filter_digest
candidate_count_by_stage / selected_evidence_refs
cache_hit / cache_scope_digest
reranker_version / packing_version / citation_validator_version
acl_violation_count / stale_evidence_count / unsupported_claim_count
tokens / latency / money / stop_reason

离线 Index Build 使用独立 Trace 或 Build Manifest，记录 Source 数、Quarantine 数、Chunk 数、Tombstone 数、输入/输出 Digest、Parser/Chunker/Embedding 版本，以及 Validation、Promotion 或 Rejection 结果。在线 Run 只引用已经发布的 index_build_id；不能在同一 Run 中静默切换到未验证 Build。

selected_evidence_refs、Query Ref 和 Build ID 属于高基数调试字段，应放在 Span、结构化 Log 或 Eval Record，不成为 Metric Label。若 Query 或 Chunk 全文确实需要诊断，保存到受 ACL、短保留期和删除策略控制的 Artifact Store，并在 Trace 中只放引用。

#5. 把端到端延迟拆成可行动的阶段

“模型用了 4 秒”不能解释用户为什么等待 18 秒。至少分解：

admission / queue wait
+ context build / retrieval
+ provider connection
+ time to first token (TTFT)
+ decode / reasoning
+ tool fan-out and join
+ approval wait
+ external effect / reconciliation
+ AG-UI stream / A2UI render / client projection

同时记录 Provider input、output、cached input 与其他可用 usage 类别。Prompt Cache 命中可以降低部分输入成本和延迟，但不能假定所有 Provider、模型和 Prompt 结构都具有相同语义；Cache Key、前缀稳定性、数据边界和失效条件都要进入实验。

Multi-Agent 或并行 Tool 的总耗时不是所有 Child latency 的简单相加。Trace 既要显示 critical path，也要显示累计 compute/token；否则系统可能改善墙钟时间，却把成本、限流压力和尾部失败率放大数倍。

#6. SLI 从用户任务和安全不变量出发

Model API Uptime 只是依赖指标，不是 Agent 成功率。Agent 应用常见的 SLI 包括：

  • Task Outcome Success Rate，以及关键 Protected Slice 的成功率；
  • 未授权读取、重复副作用、跨租户访问等安全不变量违规率；
  • IN_DOUBT 发生率、自动核对成功率和人工升级率；
  • Time to Truth：从效果变为未知，到用户得到权威结论的时间；
  • Orphan / Stale Run 数与 Worker 接管成功率；
  • Source-to-Index Freshness Lag，以及紧急 Tombstone / ACL 收紧退出服务路径的传播时间；
  • Retrieval Empty / Partial Rate、Stale Evidence Rate、Citation Resolvability 与受保护 Slice 的 Recall；
  • 端到端 P50/P95/P99，以及 Model、Queue、Approval、Tool、Reconciliation 分解；
  • 每个成功任务的 Token、工具、计算和人工成本。

Time to Truth 特别适合衡量有副作用的 Agent。一次操作最终正确，但持续数小时处于未知状态，仍然是严重的产品与运营问题。

Index Freshness Lag 从权威 Source Version 可见时开始，到通过验证的新 Index Build 开始服务时结束。普通内容更新可以使用基于业务需求的 SLO；撤回、删除、ACL 收紧与跨 Tenant 泄漏属于安全关键路径，应优先阻断读取，并单独度量 Tombstone Propagation。Recall 需要 Qrels，通常属于 Offline / Shadow Eval，不能用没有 Ground Truth 的线上点击率冒充。

#7. SLO 与零容忍边界

SLO 是某个时间窗口内对服务水平的目标，例如：

退款任务成功率 ≥ 99.0%
P95 Time to Truth ≤ 5 min
自动核对率 ≥ 98.0%
人工升级率 ≤ 1.0%
P95 每成功任务成本 ≤ CNY 0.80

这些数字必须来自业务风险和基线，不应从示例直接复制。

Knowledge Plane 还应按数据类型声明不同目标，而不是设置一个统一 TTL：

P95 source-to-index freshness lag ≤ 该知识类型的业务阈值
critical tombstone / ACL tightening propagation ≤ 安全响应阈值
cross-tenant retrieval violation = 0
validated index promotion rate / rollback success 达到发布目标

错误预算适合管理普通可靠性退化，不适合稀释安全不变量。未经授权的写操作、跨租户泄漏、同一 Intent 产生重复退款等事件应直接阻断发布或触发 Kill Switch，而不是因为“仍在 0.1% 以内”被接受。

#8. 计算 Cost per Successful Task

按 Run 汇总：

Model input / output / cache / reasoning
+ Retrieval and third-party tools
+ Sandbox / compute / storage / network
+ Queue, retries and failed attempts
+ Reconciliation and manual review
+ Eval / Judge amortized cost

同时报告成功和失败任务的 P50/P95。只比较 Token 单价会漏掉重试放大、长 Queue、第三方工具、人工核对和 Multi-Agent Fan-out。更便宜的模型如果让成功率下降、工具调用增加，单位成功任务成本反而可能更高。

#9. 遥测本身也是数据边界

默认记录类型化 Event、版本、引用、Hash、错误码和用量。全文只在确有诊断需要、严格 ACL 和短保留期下保存。以下内容不应默认进入普通 Trace：

  • API Key、Cookie、OAuth Token 与完整 Authorization Header；
  • 完整个人资料、订单正文和附件；
  • 未脱敏 Tool Result；
  • 模型原始 Chain-of-Thought；
  • 可由低权限观察者推断敏感信息的高基数字段。

第三方观测平台属于新的数据出境方，需要独立的权限、保留、删除和供应链评估。

#实践：用 Trace 还原一笔退款的确认过程

#进入本章时已有能力

Resolution Desk 已有 Event History、Checkpoint、Ownership Epoch 和权威 Receipt，能够恢复 Run；现有日志仍不能把 Context、Approval、Attempt、接管、Outcome 和成本连成同一条因果链。

#本章增加的能力

为同一条包含政策冲突、A2UI 澄清、AG-UI Stop、Tool Timeout、Worker 重启和 Reconciliation 的 Run 建立 Canonical Telemetry：Application Server 把 Trace Context 注入 Queue Message，Worker 提取后继续传播到模型 Adapter、MCP Server 和 Tool。Span、Log、Audit、Event 与 Eval 通过 trace_id、run_id、intent_id、action_id、attempt_id 和 ownership_epoch 关联；Metric 只使用低基数维度，并以 Exemplar 连回有代表性的 Trace。

将政策检索展开为 Rewrite、Authorized Scope、Sparse/Dense Candidate、Fusion、Rerank、Packing 与 Citation Validation Span，并关联 Source Snapshot、Index Build 与 Evidence Ref。再模拟一次政策更新、一次紧急撤回和一次 ACL 收紧，记录 Source-to-Index Lag、Tombstone/ACL Propagation、Stale Evidence、Citation Resolvability 和跨 Tenant Violation。实现 Task Success、IN_DOUBT Rate、自动核对率、Time to Truth、P95/P99 延迟和 Cost per Successful Task 等 SLI，并验证异常与发布验收 Trace 的采样策略。

#验收证据

从最终 Receipt 反向定位选中的政策版本、Index Build、Query Rewrite、授权范围摘要、Rerank、Citation Map、产生澄清输入的 A2UI Action、对应的 AG-UI Public State、原生 Approval、首次 Command、ACK 丢失、Cancel、Worker 接管和核对完成。同时证明 Trace Context 能跨浏览器、HTTP、Queue 与 MCP 边界传播，且没有保存原始 Query、Embedding、敏感 Chunk、密钥、完整个人资料、未脱敏 Tool Result 或原始 Chain-of-Thought。

仪表盘必须能区分 Queue、Context/Retrieval 各阶段、TTFT、Decode、Tool Fan-out、Reconciliation 与人工耗时，并显示 Index Freshness 与 Tombstone Propagation。安全不变量一旦违规，应立即阻止发布或触发响应，不能把它计入一般错误预算。

#本章小结

可观测性把 Run 从零散日志还原为可验证的因果链。Trace 解释路径，Audit 解释责任，Outcome 解释真实结果，Eval 解释版本差异，SLI/SLO 与成本则决定系统是否具备进一步放量的条件。下一章将讨论这套证据如何进入 发布、模型依赖与生产运营。
