# 02 · 并发、Backpressure 与任务预算

Resolution Desk 已能让单个退款 Run 在 Timeout 后进入 Reconciliation，并用原幂等键确认真实效果。并发扩大后，新的风险是大量政策检索、模型规划和重试挤满连接池，反而让已经承诺给用户的退款核对得不到执行容量。

在前端代码中，`Promise.all()` 是常见的并行工具；放进 Agent Loop 后，它也可能成为故障放大器。一个 Run 规划十个步骤，每步并发五个工具，每个工具最多三次 Attempt，再委派两个 Worker，理论调用量很快增长到数百次。

模型并不知道数据库连接池、Model Provider Quota 或第三方 API 还剩多少容量。Runtime 必须在创建工作之前检查并发限制和预算，而不是把限制写进 Prompt 后期待模型自觉遵守。

## 1. 先计算 Fan-out 上界

一个简化的放大模型是：

```text
总 Attempt 上界
≈ Run 数 × Step 数 × 每步 Tool Call 数 × 每调用 Attempt 数 × Worker 数
```

这个公式不是容量规划的精确结果，却能快速暴露无界设计。还要继续追问：

- 模型是否能在每轮重新生成更多分支；
- Subagent 是否又能创建 Subagent；
- Gateway、Runtime 与 Tool Client 是否分别重试；
- Stream Event 和队列消息是否可能重复投递；
- Cancel 后是否仍有后台 Promise 未被跟踪。

若无法给出单个输入的最大在途调用、最大排队数和最大费用，Runtime 仍然是无界的。

## 2. Backpressure 要在生产者一侧生效

Backpressure 表示消费者跟不上时，生产者必须等待、拒绝或降级。无限队列只是把过载从调用处推迟到内存、连接池或用户等待时间。

常用机制包括：

- 有界队列（bounded queue）；
- Semaphore、Worker Pool 与连接池；
- Per-tenant、per-tool、per-provider 和全局 Quota；
- Admission Control 与 Load Shedding；
- 相同只读查询的 Coalescing / Cache；
- Circuit Breaker，停止继续调用已知故障依赖。

队列满时必须定义产品语义：等待到 Deadline、持久排队、返回部分结果、降级为只读、拒绝新任务或转人工。让请求继续堆积不是一种语义。

## 3. Node.js 中的有界并发

下面的 Semaphore 展示取消感知的核心结构；生产实现还需补充公平性、Timeout、指标和服务关闭语义：

```ts
type Waiter = {
  signal: AbortSignal;
  onAbort: () => void;
  resolve: (release: () => void) => void;
  reject: (reason?: unknown) => void;
};

class Semaphore {
  private active = 0;
  private readonly waiters: Waiter[] = [];

  constructor(private readonly limit: number) {}

  acquire(signal: AbortSignal): Promise<() => void> {
    signal.throwIfAborted();

    if (this.active < this.limit) {
      this.active += 1;
      return Promise.resolve(this.releaseOnce());
    }

    return new Promise((resolve, reject) => {
      const waiter: Waiter = {
        signal,
        resolve,
        reject,
        onAbort: () => {
          const index = this.waiters.indexOf(waiter);
          if (index >= 0) this.waiters.splice(index, 1);
          reject(signal.reason);
        },
      };
      signal.addEventListener('abort', waiter.onAbort, { once: true });
      this.waiters.push(waiter);
    });
  }

  private releaseOnce(): () => void {
    let released = false;
    return () => {
      if (released) return;
      released = true;

      const next = this.waiters.shift();
      if (next) {
        next.signal.removeEventListener('abort', next.onAbort);
        next.resolve(this.releaseOnce());
      } else {
        this.active -= 1;
      }
    };
  }
}
```

实现时还要特别检查进程关闭时等待者如何失败、队列长度是否有硬上限，以及监控能否区分 Active、Queued、Rejected 与 Cancelled。

## 4. 用 Bulkhead 隔离不同责任的工作

所有任务共用一个并发池时，低价值的新规划可能挤占高风险动作的收尾容量。可以按责任建立舱壁（Bulkhead）：

| 通道                 | 典型工作            | 过载时语义                   |
| ------------------ | --------------- | ----------------------- |
| Planning           | Model Call、计划更新 | 拒绝或延后新 Step，不创建隐藏后台任务   |
| Read Tools         | 检索、查询、RAG       | 缓存、返回 Partial、等待或拒绝     |
| Commands           | 退款、发送、删除等写动作    | 小并发池；同一资源或 Intent 必要时串行 |
| Reconciliation     | Receipt 与权威状态查询 | 预留有限容量，优先核对未知效果         |
| Evaluation / Batch | 回归、Judge、Shadow | 独立低优先级池，可暂停             |

预留 Reconciliation Pool 不是让核对无限优先。它仍有并发、Deadline 和人工升级上限，只是不会被大量新 Planning 饿死。

## 5. 任务预算是一组确定性约束

一个 Run 通常需要多维预算：

- **Step Budget**：最多允许多少次模型决策或状态转移；
- **Token Budget**：输入、输出、Reasoning 与 Judge 用量；
- **Time Budget**：Run Deadline 和每个 Step Timeout；
- **Money Budget**：模型、工具、Sandbox、存储与人工费用；
- **Fan-out Budget**：并发 Tool、Worker、分支和队列深度；
- **In-flight Effect Budget**：同时允许多少个未确认 Command；
- **Reconciliation Budget**：查询未知效果的次数和期限。

Runtime 在创建下一项工作之前检查预算，持久化消费记录，再发起调用。模型可以看到剩余预算以便做计划，但不能修改或绕过它。

预算耗尽后的状态也要区分：没有在途效果时可以进入 `BUDGET_EXHAUSTED`；存在未知 Command 时则停止新目标，继续有限核对，不能用预算终态覆盖风险。

## 6. 并行执行必须先定义 Join 语义

并行适合相互独立、结果可分别验证且副作用安全的任务。启动前应选择一种汇合规则：

- **All**：全部成功才继续；
- **Partial**：允许缺失，但结果必须标明不完整；
- **First valid**：首个满足条件的结果获胜，其余协作式 Cancel；
- **Quorum**：达到法定数量后继续，同时保留分歧。

对同一订单、同一文件或同一审批的写操作，通常不应并行发出多个候选再“选一个成功的”。共享资源应由版本检查、锁或权威服务原子裁决。

## 7. 过载降级必须保持任务语义

假设 Model Provider 延迟升高，同时支付系统不稳定：

```text
Circuit / Quota 停止新的低优先级 Planning
→ 新任务被有界排队或明确拒绝
→ 已提交 Command 不被伪装成已取消
→ Reconciliation Pool 查询已有幂等记录
→ 以 Receipt 确认效果，或转入人工异常队列
→ 恢复后逐步放量，而不是瞬间清空积压
```

系统降级的目标不是维持“所有请求都接收”的表面可用性，而是优先保护正确性、已发生的用户承诺和可恢复性。

## 实践：在过载中保住退款核对通道

### 进入本章时已有能力

Resolution Desk 可以正确处理单个 `IN_DOUBT` Run，但 Planning、Read Tools、Commands 与 Reconciliation 仍通过无差别的 Promise 并发，共用同一个容量池。

### 本章增加的能力

为四类工作建立有界队列、独立 Semaphore、per-tenant Quota、任务预算和明确的队列满语义。Reconciliation 使用独立但仍有限的容量，Command 对同一订单或 Intent 串行。随后构造 100 个 Run，每个 Run 尝试创建 10 个慢工具调用，每个工具 `max_attempts = 3`，依次验证：

1. 无限制 `Promise.all()` 下的连接、内存、延迟和费用放大；
2. 加入 Semaphore 和有界队列后的最大在途量；
3. Per-tenant Quota 能否阻止单个租户占满系统；
4. Provider 限流时 Circuit Breaker 是否停止新工作；
5. 用户 Cancel 后，排队项是否被移除、在途信号是否传播；
6. Planning 过载时，Reconciliation 是否仍能在自身预算内完成。

### 验收证据

用指标断言最大在途调用、最大队列深度、拒绝数、各 Tenant 占用、P95/P99 Queue Time、Token/费用上界与 Reconciliation Time to Truth。Provider 限流和 Planning 洪峰期间，新任务只能有界等待、降级或拒绝；已提交 Command 不被误报为取消，Reconciliation 不被饿死，恢复后也不会瞬间清空积压形成第二次洪峰。

## 本章小结

Backpressure 把“模型还想继续”转换成“系统是否有能力创建下一项工作”。有界队列、Bulkhead、Quota、Admission Control 和多维预算共同限制 Fan-out，也为未知效果保留有限的收尾容量。下一章将继续处理进程退出后的恢复：[持久执行、Checkpoint 与 Exactly-Once](/masterpiece-static-docs/09-可靠性与可观测/03-持久执行-Checkpoint与Exactly-Once.md)。
