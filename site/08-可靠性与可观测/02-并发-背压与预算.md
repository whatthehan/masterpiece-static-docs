# 02 · 并发、背压与预算

一次重试有界，不代表整个 Agent Run 有界。规划步骤、工具调用、工作进程（Worker）和 Subagent 会层层扇出（fan-out）；每一层都看似合理地并发或重试时，最底层依赖承受的负载、Token 与费用可能已经成倍增长。

本章把背压（Backpressure）与预算放到 Runtime 创建新工作的前一刻执行。它们一面保护系统容量，一面给取消、部分失败和超载定义可预测状态，使“任务很多”不再自动演化成无界队列和级联故障。

## 学习目标

- 区分并发、并行、队列和 fan-out。
- 使用有界队列、并发上限和背压控制 Agent 放大。
- 把预算当作运行时强制不变量。

## 1. Agent 的放大结构

```text
1 user request
→ N plan steps
→ each step M tool calls
→ each tool retries R times
→ workers/subagents fan out K
```

没有上界时，调用、Token、网络和费用呈乘法增长。模型不会天然理解系统容量。

## 2. Concurrency 与 Parallelism

- Concurrency：多个任务在时间上重叠推进。
- Parallelism：多个任务同时在不同执行资源上运行。

并行只适合无依赖、结果可独立验证且副作用安全的工作。共享可变状态会引入 race、顺序和部分成功语义。

## 3. Backpressure

当消费者比生产者慢，系统必须让生产者减速、拒绝或降级，而不是无限缓冲。

常见机制：

- Bounded queue。
- Semaphore/worker pool。
- Rate limit 与 per-tenant quota。
- Admission control/load shedding。
- Batch/coalesce。
- Tool 和模型的独立并发池。

队列满时需要明确策略：等待到 deadline、拒绝、降级或持久排队。

## 4. 五类预算

- Step：最多模型决策轮数。
- Token：输入、输出、reasoning 和 Judge。
- Time：Run deadline 与各 span timeout。
- Money：模型、工具、沙箱、存储和重试。
- Fan-out：并发工具、Worker、分支和队列深度。

预算检查应在创建下一步工作前发生。达到上限时，Runtime 持久化 budget-exhausted fact 并停止新工作，不让模型决定是否继续。这个 fact 不覆盖已存在的 cancel intent 或 unknown-effect 状态；收敛时 cancel 语义优先。只有在无 in-flight command、无 unknown effect 时才直接进入 `BUDGET_EXHAUSTED`；否则先进入/保持 `IN_DOUBT → RECONCILING`，根据真实效果收敛终态。

## 5. 部分并行失败

定义：全部成功才继续、允许部分结果、失败即取消其余、还是收集到 quorum。对副作用调用，一般不应把“并行全部发出后再决定”作为默认。

## 纸面微实验（30–45 分钟）

对“100 个任务，每个 fan-out 10 个慢工具”计算无界并发的在途调用上界。然后设计 semaphore、bounded queue、per-tenant quota、deadline 和 overload 策略，写出队列满、用户取消、部分工具失败时的状态转移和数值上界。

## L1 后系统实验

启动 100 个任务，每个 fan-out 10 个慢工具。先使用无限 Promise 并发观察内存/连接；再加入 semaphore、有界队列、deadline 和 overload rejection，证明调用数与内存有上限。

## 常见误区

- `Promise.all` 就是正确并发控制。
- 队列越大越不丢任务。
- 多 Agent 并行只增加速度，不增加费用和错误面。
- Rate limit 只应由下游服务负责。
- 预算可以写在 Prompt 里让模型自觉遵守。

## 本章小结

并发提升吞吐的前提是依赖关系、副作用与部分失败语义都已明确；有界队列、并发池、准入控制和五类预算负责把 fan-out 锁在可证明的上限内。当任务还要跨进程和部署继续时，这些运行时事实必须被保存，下一章因此进入[持久执行、Checkpoint 与 Exactly-Once](/masterpiece-static-docs/08-可靠性与可观测/03-持久执行-Checkpoint与Exactly-Once.md)。
