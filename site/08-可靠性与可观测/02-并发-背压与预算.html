#02 · 并发、背压与预算

order_123 进入核对后，支付服务的延迟告警同时影响了更多订单。新的退款请求仍在进入，普通查询开始重试，后台 Worker 还要查询未知效果。如果所有工作共享一个无限队列，最需要尽快确认的退款反而可能排在几千次新规划之后。

上一章已经定义单个 Run 如何收敛未知效果；本章只解决系统规模问题：谁可以创建多少工作，队列满时谁等待、谁降级，以及哪些安全收尾必须拥有独立容量。

#学习目标

  • 识别 Agent 中计划、工具、Retry、Worker 和 Subagent 的乘法放大。
  • 用有界队列、隔离并发池、配额和准入控制表达容量边界。
  • 把预算变成 Runtime 创建新工作前执行的确定性约束。

#1. 一个请求怎样被放大

1 个用户请求
→ N 个计划步骤
→ 每步 M 个工具调用
→ 每个工具最多 A 个 attempts
→ 每项又可能产生 K 个 Worker/Subagent

理论上界接近 N × M × A × K。模型不会天然知道连接池、Provider Quota 或下游支付服务的剩余容量，也不能靠 Prompt 承诺“不调用太多”。

对于 order_123，核对只需要一次权威退款查询。若系统又让模型生成三个补救计划、每个计划并行查询五个来源，就把一个确定性恢复问题错误放大成新的 Agent 探索。

#2. 按责任隔离容量

不要让所有工作争夺同一个并发池：

工作通道典型工作容量策略过载时语义
Planning模型调用、计划更新Provider/tenant 配额，低优先级可拒绝或延后不创建新 Run/Step
Read tools检索、查询、RAG有界并发与缓存，允许降级返回明确的 overloaded/partial
Commands真实写动作更小并发池，按资源/租户串行化关键意图不在队列外偷偷提交
ReconciliationReceipt/权威状态查询预留容量、有限核对预算优先收敛未知效果，超期转人工
Evaluation/Batch回归、Judge、Shadow独立低优先级池暂停，不挤占在线安全路径

这类隔离称为舱壁（bulkhead）。它不是让核对无限优先，而是为已有风险提供一条仍然有上限的收尾通道。

#3. Backpressure 发生在创建工作之前

消费者慢于生产者时，系统必须让生产者等待、拒绝或降级，而不是无限缓冲。常用机制包括：

  • 有界队列（bounded queue）和 Semaphore/Worker Pool。
  • Per-provider、per-tool、per-tenant 与全局 Quota。
  • Admission Control 与 Load Shedding。
  • Batch/Coalesce，对相同只读查询合并请求。
  • Circuit Breaker，阻止已知故障依赖继续吸收流量。

队列满时必须提前定义一种产品语义：等待到 Deadline、持久排队、使用只读降级、拒绝新任务，或转人工。不能先无限接收，再让内存耗尽替系统做决定。

#4. 五类预算与一个收尾预算

  • Step：最多模型决策轮数。
  • Token：输入、输出、推理与 Judge 用量。
  • Time：Run Deadline 与各 Span Timeout。
  • Money：模型、工具、沙箱、存储与重试费用。
  • Fan-out：并发工具、Worker、分支和队列深度。
  • Reconciliation：已有在途/未知效果允许使用的查询次数和期限。

普通预算在创建下一步工作前检查。达到上限时，Runtime 持久化 budget_exhausted 并停止新规划；它不覆盖已有 Cancel intent 或 unknown effect。order_123 即使已经耗尽模型 Step Budget，也仍可在独立且有界的 Reconciliation Budget 内查询 refund:order_123:v42，因为这是收敛既有风险，不是继续追求新业务目标。

#5. 并行不是默认加速键

并行只适合依赖独立、结果可分别验证且副作用安全的工作。开始前要明确汇合语义：

  • 全部成功才继续。
  • 允许部分结果并显式标注缺失。
  • 首个满足条件的结果获胜，其余协作式取消。
  • 收集到 Quorum，但仍记录分歧。

对写动作，通常不应“先全部并行发出，再根据结果选择”。共享订单、同一审批或同一幂等意图应在资源边界上串行化或由权威服务原子裁决。

#6. 过载时 order_123 应怎样继续

Provider 延迟升高
→ Circuit/Quota 停止创建新的低优先级 Planning
→ 新退款请求被拒绝或持久排队
→ 已提交 Commands 不被假装取消
→ Reconciliation Pool 查询既有幂等记录
→ order_123 得到 refund_id=rf_789
→ 释放 in-flight slot，记录真实终态

安全收尾优先于新工作，但同样受 Deadline、并发和人工升级上限约束。否则“永不放弃核对”也会成为新的资源泄漏。

#纸面微实验（30–45 分钟）

假设 100 个 Run 各规划 10 个慢工具、每个工具 max_attempts=3：

  1. 计算无约束的最底层 attempt 上界。
  2. 设计 Planning、Read、Command、Reconciliation 四个并发池及具体数值。
  3. 写出队列满、用户 Cancel、Provider 限流和部分工具失败时的状态与用户语义。
  4. 证明任一输入都有在途调用、排队数、Token 和费用上界。

#L1 后系统实验

先用无限 Promise 并发运行上述负载，观察连接、内存、延迟与取消传播；再加入 Semaphore、有界队列、per-tenant quota 和预留 Reconciliation Pool。验证 order_123 的核对不会被新 Planning 饿死，过载也不会产生无界后台任务。

#常见误区

  • Promise.all 等于并发控制。
  • 队列越大越不丢任务。
  • 所有工作共用一个池更容易提高利用率。
  • Budget 写进 Prompt 就能成为硬约束。
  • Reconciliation 不受预算，因为它是安全工作。

#本章小结

容量控制把“模型想继续”转换成“系统是否允许创建下一项工作”。有界队列、Bulkhead、Quota、Admission Control 与预算为放大结构建立上限，也为未知效果保留有限收尾能力。order_123 已经排入核对通道，但此时 Worker 突然重启；下一章将进入持久执行、Checkpoint 与 Exactly-Once，回答新进程怎样接住同一事实而不重复退款。
