# 01 · 失败分类、超时、重试与取消

进入长链路后，安全执行还需要面对一个更日常的事实：模型、检索、队列和外部工具不会同时、清晰地成功或失败。用户按下取消时，某个命令（command）可能已提交但回执尚未返回；如果 Runtime 只保存一个“失败”布尔值，就无法决定能否重试，也无法向界面给出诚实状态。

本章用执行状态与效果状态两条轴拆解部分失败，并区分截止时间（Deadline）、超时（Timeout）、重试（Retry）与取消（Cancellation）。这些不是 SDK 选项的集合，而是贯穿 Run 状态机、下游调用和用户恢复入口的失败语义。

## 学习目标

- 接受分布式调用会发生部分失败和未知完成状态。
- 区分 deadline、timeout、retry 和 cancellation。
- 用明确尝试数、全局预算和 reconciliation 避免级联放大与重复副作用。

## 1. 失败是正常状态

Agent 一次 Run 会跨模型、检索、数据库、工具和第三方 API。任一层都可能慢、失败或只完成一部分。最难的不是“收到了错误”，而是调用者有时无法知道下游是否已产生效果。

需同时记录两个维度：

- Execution status：本地 attempt 是运行、超时、断连还是收到响应。
- Effect status：权威系统中副作用是未发生、已发生、已补偿还是不明。

## 2. Deadline 与 Timeout

- Deadline：整个用户目标最晚何时还值得继续。
- Timeout：某一步最多等待多久。

下游 timeout 应从剩余 deadline 派生，并为传输、清理和回应留出余量；不能每层重新获得完整时间预算。否则五个串行步骤各等 30 秒会破坏 30 秒总目标。

## 3. Retry 的适用条件与算术

仅对明确瞬时、可重试且操作语义安全的错误重试：

- 有限 `max_attempts`，数字包含首次调用。
- 指数退避与 jitter。
- 全局 retry budget 和单一责任层。
- 每次 attempt 共享业务幂等键，但有独立 attempt ID。

若第 `i` 层最多重试 `r_i` 次，则它最多尝试 `r_i + 1` 次，层层独立重试的下游上界是：

```text
Π (r_i + 1)
```

因此：

- 三层各 `max_attempts = 4` → `4 × 4 × 4 = 64` 个最底层 attempt。
- 三层各“最多重试 4 次” → 每层共 5 次尝试，即 `5³ = 125`。

设计和试题中必须写清 `max_attempts` 还是 `max_retries`。

## 4. Cancellation 是协作式协议

取消表示“上游不再需要继续新工作”，不是 rollback。最小状态链是：

```text
RUNNING
→ CANCEL_REQUESTED     持久化意图，停止新工作
→ CANCELLING          向模型流、工具、队列和子任务传播
→ IN_DOUBT            若在途 command 效果未知
→ RECONCILING         查 receipt/权威业务状态，必要时补偿/转人工
→ terminal state
```

终态不能只有 `CANCELLED`：

- `CANCELLED`：确认未产生需处理的效果。
- `COMPLETED_WITH_EFFECT_AFTER_CANCEL`：取消竞态中效果已提交，已向用户说明。
- `PARTIAL` / `MANUAL_INTERVENTION`：已有部分效果或仍无法自动收敛。

将 Promise 标记为 rejected、关闭 UI 连接或取消本地 Future，都不能证明第三方已停止。

## 5. 错误矩阵

| 错误                               | 建议                            |
| -------------------------------- | ----------------------------- |
| invalid/denied                   | 不重试，修输入或权限                    |
| conflict/version mismatch        | 重新读取状态，再决定                    |
| rate limit/transient unavailable | 在 deadline/retry budget 内有界退避 |
| timeout on query                 | 若仍有预算，可重试                     |
| timeout/disconnect on command    | 进入 `IN_DOUBT`，先查询幂等记录/业务状态    |
| cancel requested                 | 停止新工作、传播、核对在途效果               |
| budget exhausted                 | 明确终止或等待新授权预算                  |

## 纸面微实验（30 分钟）

1. 计算三层各 `max_attempts=4` 和各 `max_retries=4` 的不同上界。
2. 画出写工具提交成功后丢 ACK，随后用户取消的时序。
3. 对每一步写出 execution/effect status、允许转移、幂等键和用户文案。

将 timeout 解释为“未执行”，或将 cancel 直接写成“已回滚”，即不通过。

## 常见误区

- Retry 总会提高可用性。
- timeout 后可以安全重复 command。
- 每一层都重试最稳妥。
- cancel Promise 等于下游停止。
- 模型可以自行判断基础设施错误是否瞬时。

## 本章小结

可靠性首先要求承认“本地调用结束”与“外部效果确定”是两回事；deadline 约束整条链路，retry 只处理明确且语义安全的失败，cancel 则停止新工作并推动未知效果进入 reconciliation。下一章将这些规则放进[并发、背压与预算](/masterpiece-static-docs/08-可靠性与可观测/02-并发-背压与预算.md)，控制并行与重试造成的乘法放大。
