# 05 · 结构化输出与工具调用

学会写 Schema 后，最危险的误解是把“结构正确”直接等同于“可以执行”。模型生成的退款调用即使字段全部合法，也只是一个候选动作：Runtime 仍要确认 Item 完整、订单状态未变化、当前用户有权操作，并在必要时取得与精确参数绑定的审批。

本章把结构化输出（Structured Outputs）与函数调用（Function Calling）放回完整执行链，展示 Schema 与语义、授权、并发和目标系统安全如何层层衔接。它们共同构成受控行动的入口，而不是七个互相替代的检查项。

## 学习目标

- 理解 JSON mode、Structured Outputs 和 Function Calling 的差异。
- 建立七层校验模型。
- 知道 tool call 是候选动作，不是执行授权。

## 1. 三类机制

- JSON mode：主要保证输出是可解析 JSON。
- Structured Outputs：按受支持的 JSON Schema 子集约束生成结构。
- Function Calling：模型产生与工具定义关联的结构化调用，由应用决定是否执行。

约束解码会在生成时限制下一 Token 的合法集合。这能减少语法错误，但不能验证现实世界和业务语义。

## 2. 完整工具调用协议

```text
应用提供工具定义
→ 模型提出 tool call
→ 协议完整性检查
→ Schema 校验
→ 业务语义和当前状态校验
→ 身份、授权与策略校验
→ 必要时具体审批
→ 幂等执行并记录回执
→ 结果作为不可信观察返回模型
```

模型永远不应直接持有数据库连接、支付密钥或执行权限。

## 3. 七层校验

1. **Protocol**：流是否完整、call\_id 是否存在、是否处于允许状态。
2. **Schema**：类型、必填字段、枚举和附加属性。
3. **Semantic**：实体存在、金额范围、参数组合和业务规则。
4. **State/Concurrency**：资源版本、前置状态、是否已经执行。
5. **Authorization/Policy**：actor 是否能对该 resource 执行 action。
6. **Sink-specific safety**：按目标解释器防 shell/SQL/template injection、路径穿越/symlink、SSRF/重定向/DNS rebinding、XSS 和危险反序列化。
7. **Execution reliability**：幂等、超时、重试、审计、补偿和隔离。

`strict: true` 主要加强第 2 层，不能跳过其他六层。

Sink-specific controls 必须贴近 sink：SQL 使用参数化查询；命令使用固定 executable 和分离 argv，对 option/值/路径做 allowlist 与目标程序语义校验，程序支持时用 `--` 终止 option 解析，而非拼 shell；文件在预先打开的根目录句柄下解析并拒绝 symlink 逃逸；URL 每次重定向都重做 scheme/host/IP/egress 校验；HTML 按输出上下文编码；禁用不可信对象反序列化。

## 4. 工具设计反过来影响模型准确率

- 名称和描述应说明何时用、何时不用。
- 让非法状态无法表示，例如不用两个可能冲突的布尔字段表示一个枚举状态。
- 已知参数由代码注入，不要求模型重复填写。
- 同时暴露的工具保持精简；相似工具要有明确边界。
- 查询、草案和提交拆开，高风险动作提供 preview/diff。

## 5. 拒绝与不完整响应

安全拒绝、内容截断、超时或不相关输入可能不走正常 Schema 成功路径。应用必须显式处理这些状态，不能假设“只要请求了 Schema 就必有合法对象”。

## 前置微实验（45 分钟）

定义严格退款 Schema，构造：

- 金额超过订单余额。
- 订单属于另一租户。
- 订单已退款。
- 相同调用并行出现两次。
- 工具返回包含恶意指令。
- `reason` 字段结构合法但包含 shell/SQL/HTML payload，应用试图送入危险 sink。

逐层标出应在哪个门禁被拒绝，并证明只有 Schema 的实现仍然危险。通过证据是每个 payload 都对应明确 sink 与具体编码/参数化/allowlist 控制，而不是泛化“sanitize”。

## 常见误区

- Structured Outputs 保证事实正确。
- Function Calling 表示模型已经调用本地函数。
- 枚举可以替代授权策略。
- 工具结果来自“内部系统”所以可信。
- 并行工具调用只是性能优化，没有状态语义。

## 本章小结

Tool call 是模型提交给应用审查的提案，只有协议、结构、语义、状态、授权、目标系统安全与执行可靠性全部通过，Runtime 才能产生真实副作用。下一章将在 [Agent Loop 与状态机](/masterpiece-static-docs/04-模型接口与Agent内核/06-Agent-Loop与状态机.md)中把这些门禁、等待与回执组织成可验证的运行过程。
